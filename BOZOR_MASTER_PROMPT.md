## 1. Видение и контекст проекта

**Bozor** — это backend торговой площадки объявлений (C2C/B2C classifieds) для рынка Узбекистана, функциональный аналог olx.uz. На данном этапе разрабатывается **только backend** (фронтенда нет). Backend строится как система **микросервисов на Go** с хранением данных в **PostgreSQL** (паттерн database-per-service).

**Что делают пользователи площадки:**
- регистрируются и входят через Telegram (номер телефона);
- публикуют объявления с фотографиями, ценой, категорией и атрибутами;
- ищут и фильтруют объявления по категориям, регионам, цене и характеристикам;
- добавляют объявления в избранное, сохраняют поисковые запросы с уведомлениями о новых совпадениях;
- общаются с продавцами/покупателями во встроенном чате;
- продвигают объявления платными услугами (VIP, Топ, поднятие);
- оставляют отзывы и рейтинги;
- получают уведомления через Telegram-бота.

**Рыночный контекст (обязательно учесть в модели данных и логике):**
- Двуязычие: **узбекский (uz) и русский (ru)**. Категории, атрибуты, шаблоны уведомлений, сообщения об ошибках — с локализацией.
- Валюта — **UZS (сум)**.
- Платёжные системы Узбекистана — **Payme** и **Click**.
- География — регионы и города/районы Узбекистана (Ташкент и районы, Самарканд, Бухара, Наманган, Андижан, Фергана, Коканд, Джизак, Нукус и т.д.).

**Требования по нагрузке.** Площадкой будут пользоваться **десятки тысяч человек**. Backend и база **не должны деградировать/падать** при высокой конкуренции. Проектная цель: выдерживать **десятки тысяч DAU** и **тысячи одновременных пользователей**, с горизонтально масштабируемыми stateless-сервисами и чётко описанным путём выхода за пределы одного хоста.

| Слой | Выбор по умолчанию | Альтернатива / примечание |
|---|---|---|

**Обоснование выбора Compose-совместимого лёгкого стека.** Заказчик выбрал Docker Compose ради простоты старта. ## 4. Архитектура системы (высокоуровнево)

**Топология (словесная диаграмма):**

```
                         ┌────────────────────────┐
   Клиенты (будущие  ───▶│  nginx / Traefik (edge) │  TLS, LB, статик
   моб./веб)             │  + API Gateway (Go)     │  authN(JWT), routing, rate-limit
                         └───────────┬────────────┘
                                     │ gRPC / HTTP
        ┌──────────────┬─────────────┼───────────────┬───────────────┐
        ▼              ▼             ▼               ▼               ▼
     Auth          User/Profile   Catalog         Listing         Search
     (Telegram)                   (+Location)     (Ads)           (Typesense)
        │              │             │               │               ▲
        ▼              ▼             ▼               ▼               │ индексация
     Media         Favorites/     Moderation     Notification     (по событиям)
     (MinIO)       SavedSearch                   (Telegram-бот)
        │                                            ▲
        ▼                                            │
      Chat (WS + backplane)   Payments/Promotions ───┘   Reviews/Ratings

   Инфраструктура: PostgreSQL(×N БД) · PgBouncer · Redis · NATS JetStream ·
                   Typesense · MinIO+CDN · OTel/Prometheus/Grafana/Loki/Tempo
```

**Ключевые архитектурные принципы:**
- **Database-per-service.** На старте — один инстанс PostgreSQL с **отдельной базой на каждый сервис** (логическая изоляция, никаких кросс-сервисных JOIN и прямого доступа к чужой БД). Путь разделения на отдельные инстансы задокументирован.
- **Синхронно — gRPC**, для запрос-ответ между сервисами. **Асинхронно — события через NATS JetStream** (интеграция, реакция на изменения, устранение временной связанности).
- **Transactional Outbox.** Любое доменное событие пишется в таблицу `outbox` в той же транзакции, что и изменение данных; отдельный relay-воркер публикует его в брокер (at-least-once). Потребители идемпотентны, есть DLQ.
- **CQRS-lite.** Поисковый сервис (Typesense) — это read-модель для листингов; он строится из событий сервиса объявлений, а не запрашивает его БД.
- **API Gateway** отвечает за: маршрутизацию, проверку JWT (и проброс `user_id`/claims во внутренние вызовы), rate limiting (Redis token-bucket), CORS, единый формат ошибок, request-id/correlation-id, агрегирование при необходимости. Gateway **не** содержит бизнес-логики.

**Каталог доменных событий (subjects `bozor.<domain>.<event>`, envelope в стиле CloudEvents):**
- `bozor.ad.created`, `bozor.ad.updated`, `bozor.ad.deleted`, `bozor.ad.approved`, `bozor.ad.rejected`, `bozor.ad.sold`, `bozor.ad.expired`, `bozor.ad.bumped`
- `bozor.user.created`, `bozor.user.updated`, `bozor.user.banned`
- `bozor.chat.message_sent`
- `bozor.promotion.activated`, `bozor.payment.succeeded`, `bozor.payment.failed`, `bozor.wallet.refunded`
- `bozor.review.created`
- `bozor.moderation.report_created`

## 5. Детальная спецификация сервисов

Для каждого сервиса указаны: зона ответственности, модель данных (ключевые таблицы), основные API, публикуемые/потребляемые события, особая логика. Полные схемы БД и OpenAPI/proto ты создаёшь в ходе реализации и держишь в `ARCHITECTURE.md` и `api/`.

### 5.1. Auth / Identity — авторизация через Telegram-бота

**Зона ответственности:** регистрация/вход по номеру телефона через Telegram-бота, выпуск и ротация токенов, сессии.

**Flow авторизации (без фронтенда):**
1. Клиент инициирует вход: `POST /api/v1/auth/session/init` → сервис создаёт одноразовый `nonce` (TTL 2–5 мин, single-use, хранится в Redis) и возвращает `{ nonce, bot_deeplink: "https://t.me/<bot>?start=<nonce>", expires_at }`.
2. Пользователь открывает бота по deep-link, жмёт **Start**. Бот отправляет сообщение с reply-кнопкой `KeyboardButton{ text: "📱 Отправить номер", request_contact: true }`.
3. Пользователь тапает кнопку → Telegram присылает боту сообщение с полем `contact`.
4. Webhook-обработчик Auth-сервиса (`POST /api/v1/auth/telegram/webhook`, внутренний):
   - проверяет подлинность запроса по заголовку `X-Telegram-Bot-Api-Secret-Token` (secret из `setWebhook`);
   - **КРИТИЧНО:** проверяет `message.contact.user_id == message.from.id` (иначе пользователь мог переслать чужой контакт — отклонить);
   - нормализует телефон в E.164 (+998…), валидирует;
   - upsert пользователя по `telegram_user_id` (сохраняет phone, first_name, last_name, username, language_code);
   - если сообщение пришло по `/start <nonce>` — связывает аутентифицированного пользователя с этим `nonce` (помечает сессию как confirmed и генерирует пару токенов).
5. Клиент опрашивает `GET /api/v1/auth/session/{nonce}` → `pending` | `confirmed` (+ `access_token`, `refresh_token`) | `expired`.
6. **JWT access** (TTL ~15 мин, claims: `sub`, `roles`, `exp`, `iat`, `jti`) + **refresh** (TTL ~30 дней, хранится хешированным, **ротируется** при каждом использовании, привязан к `device_id`, поддерживает **reuse-detection**: повторное использование отозванного refresh → отзыв всего семейства токенов).

**Эндпоинты:** `POST /auth/session/init`, `GET /auth/session/{nonce}`, `POST /auth/telegram/webhook` (внутр.), `POST /auth/refresh`, `POST /auth/logout`.

**Таблицы:** `users` (id UUIDv7, telegram_user_id UNIQUE, phone, username, first_name, last_name, language_code, status, created_at), `refresh_tokens` (id, user_id, token_hash, device_id, family_id, revoked_at, expires_at), `auth_audit_log`.

**Безопасность:** rate-limit на `init`/webhook; nonce single-use + TTL; проверка владения контактом; secret_token; ротация refresh + reuse-detection; аудит.

> Примечание: сам сервис хранит идентичность; расширенный профиль — в User/Profile. При создании пользователя Auth публикует `bozor.user.created`.

### 5.2. User / Profile

**Ответственность:** расширенный профиль, настройки, контактные предпочтения, тип аккаунта, настройки уведомлений.

**Таблицы:** `profiles` (user_id, display_name, avatar_media_id, about, user_type[individual|business], business_name, city_id, contact_phone_visible, created_at), `notification_prefs` (user_id, channel, event_type, enabled), `user_ratings_cache` (user_id, avg_rating, reviews_count — обновляется по событию из Reviews).

**API:** `GET /api/v1/me`, `PATCH /api/v1/me`, `GET /api/v1/users/{id}` (публичный профиль продавца), `GET/PUT /api/v1/me/notification-prefs`.

**События:** потребляет `bozor.user.created`; публикует `bozor.user.updated`.

### 5.3. Catalog — категории и атрибуты

**Ответственность:** дерево категорий и определения динамических атрибутов/фильтров под каждую категорию. Данные справочные, редко меняются, **агрессивно кешируются** (Redis, инвалидация по событию).

**Категории верхнего уровня (сид, 12 разделов OLX.uz):** Детский мир, Недвижимость, Транспорт, Работа, Животные, Дом и сад, Электроника, Бизнес и услуги, Мода и стиль, Хобби/отдых/спорт, Отдам даром, Обмен. Засеять эти 12 + репрезентативные подкатегории. Для минимум трёх категорий (Недвижимость, Транспорт, Электроника) завести примерные атрибуты.

**Таблицы:** `categories` (id, parent_id, slug, name_uz, name_ru, icon, sort_order, level, path), `attributes` (id, key, name_uz, name_ru, type[enum|int|decimal|bool|string], unit, is_required, is_filterable, options_json для enum), `category_attributes` (category_id, attribute_id — с наследованием вниз по дереву).

**API:** `GET /api/v1/categories` (дерево), `GET /api/v1/categories/{id}`, `GET /api/v1/categories/{id}/attributes`. Админские CRUD — под ролью moderator/admin.

### 5.4. Listing / Ads — объявления

**Ответственность:** CRUD объявлений, значения атрибутов (валидируются против Catalog), привязка изображений, полный жизненный цикл, счётчик просмотров. Источник истины по объявлениям.

**Статусы и переходы:** `draft` → `pending`(на модерации) → `active` | `rejected`(→ редактирование → снова `pending`) ; `active` → `sold` | `expired`(по `expires_at`) | `archived` ; любой → `blocked`(модерацией). Правка активного объявления по правилам может отправлять его на повторную модерацию.

**Таблицы:** `ads` (id UUIDv7, user_id, category_id, title, description, price, currency, region_id, city_id, lat, lng, status, phone_display, contact_prefs, published_at, expires_at, bumped_at, views_count, created_at, updated_at), `ad_attribute_values` (ad_id, attribute_key, value — типизированные колонки или JSONB с валидацией по Catalog), `ad_images` (ad_id, media_id, sort_order, is_cover), `outbox`.

**Особая логика:**
- **Истечение срока:** воркер помечает `active`→`expired` по `expires_at`; продление (`renew`) сдвигает срок.
- **«Продано»**, **удаление**, **редактирование** (правка кросс-полей → повторная модерация).
- **Счётчик просмотров** инкрементируется **асинхронно** (буфер в Redis, периодический флеш в БД), чтобы не создавать write-hotspot.
- Публикует `bozor.ad.*` через outbox.

**API:** `POST /api/v1/ads` (add ad), `GET /api/v1/ads/{id}`, `PATCH /api/v1/ads/{id}`, `DELETE /api/v1/ads/{id}`, `POST /api/v1/ads/{id}/sold`, `POST /api/v1/ads/{id}/renew`, `GET /api/v1/ads` (лента всех активных, с пагинацией и сортировкой), `GET /api/v1/me/ads` (мои объявления по статусам).

### 5.5. Media — изображения

**Ответственность:** загрузка, валидация, обработка и хранение изображений; выдача через CDN.

**Flow:** `POST /api/v1/media` (multipart) → валидация (тип, размер, размеры, лимит N изображений на объявление) → сохранение оригинала в MinIO → постановка задачи в брокер → воркер генерирует размеры (например, 120/480/1080 px), удаляет EXIF, (опц.) водяной знак; NSFW/вирус-проверка заложена на будущее. Выдача — presigned/CDN URL.

**Таблицы:** `media` (id, owner_user_id, bucket, object_key, status[uploaded|processed], sizes_json, created_at). Очистка «сирот» (незалинкованных к объявлениям) по TTL.

**API:** `POST /api/v1/media`, `GET /api/v1/media/{id}` (метаданные/URL).

### 5.6. Location — регионы и города

**Ответственность:** справочник регионов/городов/районов Узбекистана. Почти статика, кешируется.

**Таблицы:** `regions` (id, name_uz, name_ru, sort_order), `cities` (id, region_id, name_uz, name_ru, lat, lng). Засеять реальными данными по Узбекистану.

**API:** `GET /api/v1/regions`, `GET /api/v1/regions/{id}/cities`.

> Можно объединить Location и Catalog в один «reference»-сервис на старте (зафиксировать решение в DECISIONS.md).

### 5.7. Search — поиск объявлений

**Ответственность:** индексация объявлений и полнотекстовый + фасетный поиск.

**Схема коллекции Typesense `ads`:** id, title, description, category_id, category_path, region_id, city_id, price, currency, `attrs.*` (плоские типизированные поля-фасеты), created_at, bumped_at (для сортировки), is_top, promotion_rank, status, geo(lat,lng), поля для uz/ru.

**Индексация:** потребитель событий `bozor.ad.created|updated|deleted|approved|bumped|sold|expired` (через outbox→consumer→Typesense). Инструмент полной переиндексации (reindex) обязателен.

**Запросы:** полнотекст (uz/ru, typo-tolerance), фильтры (категория, регион/город, диапазон цены, атрибуты категории как фасеты), сортировки (релевантность, цена ↑/↓, дата, `bumped_at`; **Топ-объявления featured сверху**), пагинация (cursor/offset), гео-поиск/сортировка по расстоянию.

**API:** `GET /api/v1/ads/search` (search ads) — с фильтрами, фасетами, сортировкой, пагинацией; `GET /api/v1/ads/search/facets` (доступные фасеты для категории).

### 5.8. Favorites / Saved-Search — избранное и сохранённые поиски

**Ответственность:** избранное и сохранённые поиски с уведомлениями о новых совпадениях.

**Таблицы:** `favorites` (user_id, ad_id, created_at), `saved_searches` (id, user_id, name, query_json, notify_enabled, last_notified_at, created_at).

**Матчинг (важно для масштаба):** на событие `bozor.ad.approved` (новое активное объявление) сервис проверяет его против сохранённых поисков подписчиков и при совпадении публикует событие для Notification. У Typesense нет percolate — реализуй матчинг через отдельный matcher (оценка фильтров сохранённого поиска против нового объявления), с батчингом/дедупом и защитой от лавины. Зафиксируй стратегию в ARCHITECTURE.md.

**API:** `POST/DELETE /api/v1/favorites/{adId}`, `GET /api/v1/me/favorites`; `POST /api/v1/saved-searches`, `GET /api/v1/me/saved-searches`, `DELETE /api/v1/saved-searches/{id}`.

### 5.9. Chat — чат покупатель↔продавец

**Ответственность:** переписка по объявлению между покупателем и продавцом (WebSocket + история).

**Модель:** `conversations` (id, ad_id, buyer_id, seller_id, created_at, last_message_at), `messages` (id, conversation_id, sender_id, body, created_at, read_at). Диалог уникален по (ad_id, buyer_id, seller_id).

**Транспорт:** WebSocket-эндпоинт `/ws` (аутентификация по JWT — в query-параметре или первым сообщением). REST для истории: `GET /api/v1/conversations`, `GET /api/v1/conversations/{id}/messages`.

**Доставка и масштаб:**
- Онлайн-получатель — доставка через WS; оффлайн — публикуется событие → Notification шлёт сообщение в Telegram.
- **Backplane обязателен:** при нескольких репликах Chat-сервиса сообщение от пользователя на реплике A должно дойти до получателя на реплике B — используй Redis Pub/Sub или NATS как шину. Это требование к горизонтальному масштабированию даже под Compose.
- Непрочитанные счётчики, отметки о прочтении; (опц.) индикатор набора.
- Блокировка пользователя; жалоба на сообщение → Moderation; rate limiting на отправку.

### 5.10. Notification — уведомления

**Ответственность:** событийные уведомления пользователям. В v1 канал — **Telegram-бот** (абстракция каналов заложена для push/email позже).

**Шаблоны (uz/ru) под типы событий:** новое сообщение в чате, совпадение по сохранённому поиску, объявление одобрено/отклонено, объявление скоро истекает, платная услуга активирована, успешная оплата, новый отзыв.

**Логика:** потребляет доменные события → рендерит локализованный шаблон (по `language_code` пользователя) → отправляет через Telegram Bot API на `telegram_user_id`. Учитывает `notification_prefs`, делает батчинг/rate limiting (не спамить), хранит статус доставки, ретраит с backoff, имеет DLQ.

**Таблицы:** `notifications` (id, user_id, event_type, channel, payload_json, status, sent_at, attempts).

### 5.11. Moderation — модерация и жалобы

**Ответственность:** проверка объявлений, авто- и ручная модерация, жалобы, баны.

**Логика:**
- Каждое новое/отредактированное объявление создаёт `moderation_task` (status pending). **Авто-проверки:** стоп-слова (uz/ru; минимум «копия», «реплика», «подделка», «под оригинал» + расширяемый словарь), запрещённые категории, простая детекция дублей (проверки изображений — на будущее). Прошло авто → `active`; иначе → ручная очередь.
- **Ручные действия модератора:** approve / reject(с причиной) / request-edit(с правом правки → повторная модерация).
- **Жалобы:** пользователи жалуются на объявления/пользователей → очередь жалоб → действие (предупреждение/снятие/бан).
- **Баны:** временный/постоянный, опционально shadow-ban.
- Публикует `bozor.ad.approved|rejected`, `bozor.user.banned` → Notification.

**Таблицы:** `moderation_tasks`, `reports`, `bans`, `stopwords`.

**API (роль moderator/admin):** очередь задач, действия, список жалоб, управление банами; публичный `POST /api/v1/reports` (пожаловаться).

### 5.12. Payments / Promotions — платные услуги и монетизация

**Ответственность:** платные услуги продвижения, кошелёк, транзакции, интеграции оплаты, возвраты. Это ключевая доходная подсистема — моделируется по OLX.

**Каталог услуг (сид):**
- `TOP` — Топ-объявление на **7 или 30 дней** (в топе рубрики одновременно ротируются **не более 5** оплаченных объявлений в **случайном** порядке; на карточке — значок «ТОП»).
- `VIP` — размещение на главной, **7 дней**.
- `BUMP` — поднятие в верх списка (ручное в любой момент **и** авто по расписанию из наборов).
- `LOGO` — лого в списке, **30 дней**.
- **Наборы:** `EASY_START` (Топ 3 дня); `FAST_SALE` (Топ 7 дней + 3 авто-поднятия через 2/4/6 дней); `TURBO_SALE` (Топ 30 дней + VIP 7 дней + 9 авто-поднятий через 3/6/9/12/15/18/21/24/27 дней).

**Ценообразование:** цена услуги **зависит от региона и категории** объявления и длительности. Таблица `pricing` (service_code, region_id, category_id, duration_days, amount_uzs).

**Логика:**
- Применение услуги к объявлению создаёт `ad_promotion` (ad_id, service_code, starts_at, ends_at, status, schedule_json для авто-поднятий).
- **Ротация Топа:** макс. 5 активных Топ-объявлений на категорию, случайная выборка при каждом запросе списка — реализуется на стороне Search/Listing (Promotions помечает объявления как TOP с указанием категории и публикует события; лента/поиск формирует «топ-блок», выбирая до 5 случайных активных TOP из категории).
- **Планировщик авто-поднятий:** воркер в запланированное время «поднимает» объявление (обновляет `bumped_at` / публикует `bozor.ad.bumped` → переиндексация в поиске).
- **Кошелёк:** баланс пользователя; пополнение через Payme/Click; списание при покупке услуги; журнал транзакций (леджер, стиль двойной записи).
- **Провайдеры оплаты:** интерфейс `PaymentProvider { CreateInvoice; HandleCallback }` + адаптеры **Payme** (JSON-RPC merchant API: CheckPerformTransaction/CreateTransaction/PerformTransaction/CancelTransaction/CheckTransaction) и **Click** (Prepare/Complete) + **Mock** для dev. Колбэки **идемпотентны** (провайдеры ретраят).
- **Возвраты:** при удалении объявления модерацией с активной платной услугой — возврат неиспользованных дней в кошелёк (**пропорционально**), транзакция `refund`. Правило взаимодействия со сроком: если объявление истекает, а услуга ещё активна — услуга приостанавливается до реактивации объявления (как у OLX).
- Услуги обрабатываются асинхронно (ориентир — минуты), не блокируя пользователя.

**Таблицы:** `promotion_services`, `pricing`, `ad_promotions`, `wallets`, `wallet_transactions`, `payments` (провайдер, invoice, статус, external_id), `outbox`.

**API:** `GET /api/v1/promotions/catalog`, `POST /api/v1/ads/{id}/promote` (выбор услуги/набора → счёт/списание), `GET /api/v1/me/wallet`, `POST /api/v1/wallet/topup`, вебхуки провайдеров (внутренние).

### 5.13. Reviews / Ratings — отзывы и рейтинги

**Ответственность:** отзывы о продавцах и агрегированный рейтинг.

**Таблицы:** `reviews` (id, from_user_id, to_user_id, ad_id, rating[1..5], text, created_at). Ограничения: один отзыв на взаимодействие; анти-абьюз (нельзя себе, нельзя спамить). Текст отзыва проходит модерацию.

**Логика:** при создании отзыва пересчитывается агрегат (avg_rating, count) и публикуется событие → обновление кеша рейтинга в Profile + уведомление.

**API:** `POST /api/v1/reviews`, `GET /api/v1/users/{id}/reviews`.

## 6. Модель данных (сводно, общие правила)

- **ID:** UUIDv7 (сортируемые по времени) для всех сущностей.
- **Временные метки:** `created_at`, `updated_at` (UTC) во всех таблицах; где нужно — `deleted_at` (soft-delete).
- **Изоляция:** каждая таблица принадлежит ровно одному сервису; кросс-сервисные связи — только по ID, без внешних ключей между БД сервисов; консистентность между сервисами — через события (eventual consistency).
- **Деньги:** храним в минимальных единицах (тийины? — уточни; по умолчанию целочисленно в тийинах, отображаем в сумах), поле валюты.
- **Локализация:** пары полей `*_uz` / `*_ru` для справочных данных; пользовательский контент (заголовок/описание объявления) — как ввёл автор, с указанием языка при необходимости.
- **Аудит:** чувствительные действия (авторизация, платежи, модерация, баны) логируются в аудит-таблицы.
- **Оптимистичная конкуренция:** `version`-колонка на сущностях с конкурентной правкой.

Полные ERD по каждому сервису держи в `ARCHITECTURE.md`.

## 7. API-контракты и конвенции

- **Внешний API:** REST/JSON, версионирование префиксом `/api/v1`. Спецификации OpenAPI 3.1 — в `api/openapi/<service>.yaml`.
- **Внутренний API:** gRPC, proto — в `api/proto/<service>/`, генерация через buf.
- **Пагинация:** cursor-based по умолчанию (opaque cursor), лимит и `next_cursor` в ответе.
- **Фильтрация/сортировка:** через query-параметры; описать в OpenAPI.
- **Ошибки:** единый формат RFC 7807 (`type`, `title`, `status`, `detail`, `instance`, + `errors[]` для валидации).
- **Идемпотентность:** для небезопасных операций, где это применимо (оплаты, публикация), поддержать заголовок `Idempotency-Key` (хранить результат в Redis).
- **Rate limiting:** заголовки `X-RateLimit-*`; лимиты на уровне gateway.
- **Auth:** `Authorization: Bearer <access>`; во внутренние вызовы пробрасывается `user_id` и роли; роли: `user`, `moderator`, `admin`.
- **Correlation:** сквозной `X-Request-Id` / trace-id во всех логах и вызовах.

Конкретизируй сигнатуры всех эндпоинтов из раздела 5 в OpenAPI по мере реализации соответствующих Stage.

## 8. Нефункциональные требования (детально)

**Масштабируемость:**
- Все прикладные сервисы **stateless**; горизонтальное масштабирование репликами за nginx/gateway (в Compose — через несколько инстансов/`--scale`, апстрим-балансировка в nginx).
- **Read-heavy паттерн:** кеш (Redis, cache-aside) + вынос листинговых запросов в поисковый движок + путь к read-репликам PostgreSQL + CDN на изображения.
- **Сглаживание записи** через брокер: асинхронная индексация, обработка медиа, уведомления, флеш счётчиков просмотров.
- **PostgreSQL:** корректные индексы; PgBouncer (transaction pooling) обязателен при мультирепликах; путь к read-репликам; **партиционирование `ads`** по `created_at` (помесячно) при росте; заложить путь к шардированию/Citus.
- **Лимиты и квоты:** ограничения на публикацию объявлений на пользователя (антиспам), особенно для новых аккаунтов.

**Надёжность / устойчивость:**
- Health/readiness-пробы на каждом сервисе; graceful shutdown (дренаж соединений, докоммит).
- Ретраи с экспоненциальным backoff + jitter, таймауты, circuit breaker на межсервисных вызовах, bulkhead-изоляция.
- **Transactional outbox** для событий; **идемпотентные** потребители; **DLQ** для «ядовитых» сообщений.
- **Saga/компенсации** для мультисервисных сценариев (пример — покупка услуги: списать с кошелька → применить услугу → при сбое компенсировать возвратом). Хореография через события + компенсирующие транзакции.

**Производительность (целевые ориентиры, уточняются нагрузочным тестом):**
- p95 latency публичного чтения (лента/поиск/карточка) — целево < 200–300 мс на прогретом кеше.
- p95 записи (публикация объявления без обработки медиа) — целево < 500 мс.
- Индексация нового объявления в поиске — целево < несколько секунд.

**Безопасность:**
- JWT: короткий access, ротация refresh, отзыв, reuse-detection; secret webhook Telegram; проверка владения контактом.
- Валидация всего ввода; только параметризованные запросы (защита от SQL-инъекций); защита от IDOR (проверка владения ресурсом на каждом write/чтении приватного).
- Rate limiting и анти-спам квоты; секреты — только через env/`.env` (в репозиторий не коммитятся), `.env.example` присутствует.
- TLS на edge; CORS; сканирование зависимостей (govulncheck); принципы OWASP API Security.

**Наблюдаемость:**
- OpenTelemetry SDK во всех сервисах; трейсы → Tempo/Jaeger; метрики → Prometheus (RED на сервис, USE на ресурсы); логи → Loki (структурированный JSON с trace/correlation id).
- Дашборды Grafana на ключевые потоки (авторизация, публикация, поиск, чат, платежи, лаг очередей, соединения БД); правила алертинга (error rate, latency, лаг консьюмеров, исчерпание пула БД).

**Локализация:** uz/ru во всех пользовательских строках и справочных данных.

**Резервное копирование:** PostgreSQL PITR (WAL-архивация); резервирование/репликация MinIO; ранбуки восстановления.

## 9. Развёртывание (Docker Compose)

- **`docker-compose.yml`** поднимает: nginx (edge, TLS, LB) + gateway; все прикладные сервисы; PostgreSQL (+ отдельная БД на сервис) + PgBouncer; Redis; NATS JetStream; Typesense; MinIO; стек наблюдаемости (Prometheus, Grafana, Loki, Tempo). Плюс `docker-compose.override.yml` для локальной разработки и `docker-compose.prod.yml` для прод-профиля.
- **`.env` / `.env.example`** — вся конфигурация через переменные окружения (12-factor); секретов в репозитории нет.
- **Балансировка и реплики:** stateless-сервисы масштабируются несколькими инстансами; nginx настроен на апстрим-пул. Для WS-сервиса — sticky/консистентная балансировка + backplane.
- **Миграции:** порядок и автозапуск миграций при старте (goose), сиды (категории, регионы, каталог услуг, ценообразование).
- **Путь масштабирования (задокументировать в ARCHITECTURE.md):** одиночный хост Compose → Docker Swarm (мультихост, `deploy.replicas`) → Kubernetes (Deployments/HPA/Ingress, вынос БД/поиска/брокера в managed/StatefulSet). Тяжёлые компоненты (PostgreSQL, Typesense/OpenSearch, NATS/Kafka) при росте выносятся на отдельные хосты/managed-сервисы.
- **CI/CD (заложить):** пайплайн `lint → test → build → образы`; на PR обязателен зелёный прогон.

Будь честен в документации: одиночный Compose-хост подходит для запуска и раннего роста при stateless-сервисах и мощном хосте; при выходе на тысячи одновременных — переход на мультихост по описанному пути.

## 10. Инженерные правила (содержимое `RULES.md`)

**Структура репозитория (монорепо):**
```
/services/<name>/{cmd,internal,api}   # каждый сервис
/pkg/shared/                          # общие библиотеки: config, logging, errors, otel, auth-middleware, http/grpc helpers, outbox
/api/{openapi,proto}/                 # контракты
/deploy/{compose,nginx,migrations}/   # инфраструктура
/scripts/                             # утилиты (seed, reindex, load-tests)
/docs/                                # SPMS-файлы
```

**Go:** последняя стабильная версия; модули; `golangci-lint` (конфиг в репо); `gofmt`/`goimports`; ошибки оборачиваются `%w`; контекст пробрасывается везде; без `panic` в библиотечном коде; структурированное логирование (slog/zerolog).

**HTTP/gRPC:** chi + цепочка middleware (recover, logging, otel, auth, ratelimit); ошибки RFC 7807; валидация запросов; proto линтуется/генерится через buf.

**БД:** pgx-пул; sqlc или чистые репозитории; goose forward-only миграции (ревью, без деструктива без плана); одна БД на сервис; **запрещён** доступ к чужой БД.

**События:** NATS JetStream; именование subjects `bozor.<domain>.<event>`; envelope (id, type, source, time, data); **outbox** на публикацию; **идемпотентные** консьюмеры; DLQ.

**Тестирование:** unit (table-driven); интеграционные на testcontainers-go (postgres/nats/redis/typesense); контрактные тесты API; нагрузочные (k6) на ключевые потоки; покрытие доменной логики — установить целевой порог.

**Конфиг/секреты:** только env; `.env.example`; никаких секретов в git.

**Git:** conventional commits; PR-шаблон; CI (lint+test+build) должен проходить; фичеветки.

**Версионирование API:** `/api/v1`; изменения аддитивны; политика депрекации.

**Definition of Done (шаблон для каждого Stage):** код + тесты (unit/интеграц.) написаны и зелёные; линт чистый; миграции применяются; сервис поднимается в Compose; эндпоинты соответствуют OpenAPI; трейсы/логи/метрики отдаются; `STATUS.yaml` обновлён; решения записаны в `DECISIONS.md`.

### `SYNC-PROTOCOL.md` (адаптированный — создать с этим содержанием)

```markdown
# Bozor — Sync Protocol

## Назначение
Формальный протокол синхронизации контекста между чатами для непрерывности разработки.

## Основные принципы
1. STATUS.yaml — единственный источник истины о состоянии проекта.
2. Каждый чат начинается с загрузки контекста из файлов SPMS.
3. Каждый чат заканчивается обновлением STATUS.yaml с новой версией.
4. Никакие данные не теряются между чатами.

## Начало нового чата
Разработчик предоставляет: DESCRIPTION.md, ROADMAP.md, RULES.md, ARCHITECTURE.md, STATUS.yaml (обязательно), DECISIONS.md (желательно).
AI валидирует STATUS.yaml (наличие status_id, current.chapter/stage/status; соответствие ROADMAP; консистентность completed; отсутствие открытых блокеров) и подтверждает позицию:

    Контекст загружен:
    - Проект: Bozor
    - Глава: CHAPTER X: <название>
    - Этап: STAGE X.Y: <название>
    - Статус: <in_progress|completed|blocked>
    - Завершено этапов: N из M
    - Открытых вопросов: K
    Готов продолжить с текущего этапа?

## Работа в течение чата
При завершении Stage AI: проверяет DoD, собирает доказательства, создаёт новую версию STATUS.yaml (новый status_id, обновлённые completed/current/next_actions/metrics), пишет решения в DECISIONS.md.
При архитектурном решении: обсуждает варианты → фиксирует ADR в DECISIONS.md → при необходимости обновляет ARCHITECTURE.md.
При блокере: добавляет вопрос в open_questions, ставит current.status=blocked, НЕ продолжает до решения.

## Завершение чата
AI создаёт финальную версию STATUS.yaml и предоставляет summary: что сделано, что в процессе (%), следующие действия, новый status_id, число открытых блокеров.

## Аварийное восстановление
Если файлы недоступны — разработчик даёт EMERGENCY_SYNC.yaml; AI запрашивает недостающее (DESCRIPTION/ROADMAP), восстанавливает STATUS.yaml, просит подтверждение.

## Валидация STATUS.yaml
Обязательные поля: status_id, project, current.chapter, current.stage, current.status.
Логика: current.stage существует в ROADMAP; completed предшествуют current; metrics.completed_stages == длине completed; если status=blocked — open_questions не пуст.

Текущая версия протокола: 1.0
```

## 13. Первые шаги (bootstrap для ИИ)

При старте самой первой сессии выполни строго по порядку:

1. **Не пиши код сразу.** Сначала создай все SPMS-файлы из раздела 12 (`DESCRIPTION.md`, `ROADMAP.md`, `RULES.md`, `ARCHITECTURE.md`, `STATUS.yaml`, `SYNC-PROTOCOL.md`, `EMERGENCY_SYNC.yaml`, `DECISIONS.md`, `README.md`, каркас `api/`).
2. Заполни `STATUS.yaml` стартовым содержимым (раздел 12), проставь `total_stages` из `ROADMAP.md`.
3. Покажи заказчику получившийся `STATUS.yaml` и краткое подтверждение позиции по `SYNC-PROTOCOL.md`. **Дождись подтверждения.**
4. После подтверждения приступай к **CHAPTER 0, STAGE 0.1**: создай структуру монорепо, `Makefile`, конфиг линтера, CI-скелет. Декомпозируй Stage на задачи, обнови `tasks_total`.
5. По завершении Stage 0.1 — проверь DoD, обнови `STATUS.yaml` (новый `status_id`, `completed`, `current`, `metrics`), зафиксируй решения в `DECISIONS.md`, предложи перейти к 0.2.
6. Если на любом шаге возникает неоднозначность, влияющая на архитектуру/данные — **останавливайся, задавай вопрос, ставь `blocked`**.

## Приложение A. Глоссарий (создать `GLOSSARY.md` при желании)

- **Объявление (ad)** — единица контента: товар/услуга с ценой, категорией, атрибутами, фото.
- **Атрибут категории** — динамическое поле фильтрации, зависящее от категории (например, «пробег» у транспорта).
- **Топ-объявление** — платное размещение над обычными в рубрике; макс. 5 в ротации на категорию.
- **VIP** — платное размещение на главной.
- **Поднятие (bump)** — обновление позиции объявления в списке (ручное/авто).
- **Сохранённый (наблюдаемый) поиск** — сохранённый набор фильтров с уведомлением о новых совпадениях.
- **Outbox** — таблица исходящих событий для надёжной публикации в брокер.
- **Backplane** — шина между репликами WS-сервиса для доставки сообщений между инстансами.
- **DoD** — Definition of Done, критерии завершённости этапа.

## Приложение B. Инварианты, которые нельзя нарушать

- Никакого прямого доступа сервиса к чужой БД; только API/события.
- Каждое доменное событие — через outbox; каждый консьюмер идемпотентен.
- Каждое приватное чтение/запись проверяет владение ресурсом (нет IDOR).
- Секреты — только в env; никогда в git.
- `STATUS.yaml` обновляется в конце каждого Stage; решения — в `DECISIONS.md`.
- Пользовательские строки локализованы (uz/ru).
- Деньги — целочисленно в минимальных единицах; валюта хранится явно.
