# Bozor — Инженерные правила (RULES.md)

> Обязательные конвенции разработки backend-платформы Bozor.
> Документ входит в состав SPMS. Изменения правил фиксируются через ADR в `DECISIONS.md`.
>
> Дата актуализации: 2026-07-06.

---

## 1. Структура монорепозитория

Проект — единый монорепозиторий. Каждый сервис изолирован в своей директории, общий код — только в `/pkg/shared`.

```
bozor/
├── services/                        # прикладные микросервисы
│   └── <name>/                      # auth, user-profile, catalog, location,
│       │                            # listing-ads, media, search, favorites-savedsearch,
│       │                            # chat, notification, moderation, payments-promotions,
│       │                            # reviews, gateway
│       ├── cmd/                     # main-пакеты (entrypoint'ы: server, worker, cli)
│       │   └── server/main.go
│       ├── internal/                # приватный код сервиса (недоступен извне)
│       │   ├── domain/              # доменные сущности и бизнес-правила
│       │   ├── app/                 # use-cases / сервисный слой
│       │   ├── repo/                # репозитории (PostgreSQL, Redis)
│       │   ├── transport/           # HTTP (chi) и gRPC хендлеры
│       │   └── events/              # публикация (outbox) и консьюмеры событий
│       ├── api/                     # локальные артефакты контрактов сервиса
│       ├── migrations/              # goose-миграции БД сервиса
│       └── Dockerfile
├── pkg/
│   └── shared/                      # общие библиотеки (без бизнес-логики!)
│       ├── config/                  # загрузка конфигурации из env
│       ├── logging/                 # slog: JSON, trace/correlation id
│       ├── errors/                  # типы ошибок, маппинг в RFC 7807
│       ├── otel/                    # инициализация OpenTelemetry
│       ├── authmw/                  # auth-middleware (JWT, роли)
│       ├── httpx/                   # HTTP-хелперы, middleware-цепочка
│       ├── grpcx/                   # gRPC-хелперы, интерсепторы
│       └── outbox/                  # transactional outbox + relay
├── api/
│   ├── openapi/                     # OpenAPI 3.1: <service>.yaml (внешние REST-контракты)
│   └── proto/                       # protobuf: <service>/ (внутренние gRPC-контракты)
├── deploy/
│   ├── compose/                     # docker-compose.yml (+ override, prod)
│   ├── nginx/                       # конфиги edge-балансировщика
│   └── migrations/                  # инфраструктурные скрипты (создание БД и т.п.)
├── scripts/                         # утилиты: seed, reindex, load-tests
├── docs/                            # SPMS-файлы (STATUS.yaml, RULES.md, ...)
├── Makefile
├── .golangci.yml
├── .env.example
└── README.md
```

**Правила структуры:**
- Код сервиса живёт только в `services/<name>/internal/...` — Go-компилятор гарантирует невозможность импорта чужого `internal`.
- В `/pkg/shared` — только инфраструктурный код (конфиг, логи, ошибки, otel, middleware, outbox-хелперы). **Бизнес-логика в shared запрещена.**
- Сервисы не импортируют пакеты друг друга. Взаимодействие — только через gRPC/REST-контракты и события.
- Контракты (`api/openapi`, `api/proto`) — единственный источник истины по API; код генерируется из них, а не наоборот.

## 2. Правила Go

- **Версия:** последняя стабильная Go 1.2x, единая для всех сервисов; фиксируется в `go.mod` (`go`-директива) и в базовом Docker-образе. Обновление версии — отдельный PR на весь монорепозиторий.
- **Модули:** `go.work` в корне + отдельный `go.mod` на каждый сервис + отдельный модуль `pkg/shared` (см. ADR-011). Зависимости пинятся, `go mod tidy` обязателен перед коммитом.
- **Линт:** `golangci-lint` v2 с конфигом `.golangci.yml` в корне репозитория (единственный источник истины по набору линтеров: default `standard` + `bodyclose`, `goconst`, `gocritic`, `gosec`, `misspell`, `noctx`, `prealloc`, `revive`, `rowserrcheck`, `sqlclosecheck`, `unconvert`, `unparam`, `whitespace`). CI падает при любом замечании линтера.
- **Форматирование:** `gofmt` + `goimports` — обязательны; неотформатированный код не проходит CI. Правило импортов: stdlib → внешние → внутренние (группировка goimports).
- **Ошибки:**
  - Ошибки всегда оборачиваются с контекстом: `fmt.Errorf("создание объявления: %w", err)`.
  - Проверка — только через `errors.Is` / `errors.As`, никогда сравнением строк.
  - Сентинел-ошибки домена (`ErrNotFound`, `ErrForbidden`, `ErrConflict`, ...) объявляются в доменном слое и маппятся в RFC 7807 на транспортном слое.
  - Ошибка обрабатывается ровно один раз: либо оборачивается и возвращается наверх, либо логируется и обрабатывается. Двойное логирование запрещено.
- **Контексты:** `context.Context` — первый аргумент каждой функции, выполняющей I/O, и пробрасывается по всей цепочке вызовов (handler → use-case → repo → драйвер). Таймауты и отмена уважаются; `context.Background()` допустим только в `main` и воркерах верхнего уровня.
- **panic:** в библиотечном и прикладном коде **запрещён**. `panic` допустим только в `main` при невозможности старта (невалидная конфигурация). Каждый HTTP/gRPC-обработчик и каждая горутина воркера защищены recover-middleware/обёрткой с логированием и метрикой.
- **Логирование:** структурированное, через `log/slog` (обёртка в `pkg/shared/logging`). Формат — JSON; каждый лог содержит `trace_id`, `request_id`, `service`, `level`, `msg` и типизированные атрибуты (`slog.String`, `slog.Int`, ...). Логи через `fmt.Println`/`log.Printf` запрещены. Уровни: `DEBUG` (dev), `INFO` (бизнес-события), `WARN` (деградация), `ERROR` (сбой операции). Секреты и персональные данные (телефон, токены) в логи не пишутся или маскируются.
- **Горутины:** каждая запущенная горутина имеет владельца и путь завершения (context/WaitGroup/errgroup); «висячие» горутины запрещены. Graceful shutdown обязателен для каждого сервиса.

## 3. HTTP и gRPC

### HTTP (внешний REST)

- Роутер — **chi**, единообразно во всех сервисах.
- Обязательная middleware-цепочка (порядок фиксирован):
  1. `recover` — перехват паник, ответ 500 в формате RFC 7807, лог с уровнем ERROR;
  2. `request-id` — генерация/проброс `X-Request-Id`;
  3. `logging` — структурированный access-лог (метод, путь, статус, длительность, trace_id);
  4. `otel` — трейсинг (span на запрос), метрики RED;
  5. `auth` — проверка JWT, извлечение `user_id`/ролей (где требуется);
  6. `ratelimit` — лимиты (Redis token-bucket; основной rate-limit — на gateway).
- **Ошибки — только RFC 7807** (`application/problem+json`): поля `type`, `title`, `status`, `detail`, `instance`, плюс `errors[]` для ошибок валидации. Внутренние детали (stack trace, SQL) наружу не утекают.
- **Валидация:** каждый входящий запрос валидируется до вызова бизнес-логики (типы, диапазоны, обязательность, длины, формат). Невалидный запрос → 400/422 с перечнем `errors[]`. Валидация значений атрибутов объявлений — против определений Catalog.
- Ответы — JSON; время — RFC 3339 UTC; деньги — целые числа в минимальных единицах.

### gRPC (внутренний)

- Proto-файлы — в `api/proto/<service>/`; линт и генерация — только через **buf** (`buf lint`, `buf generate`, `buf breaking` против main в CI).
- Стиль proto — Google API Design / buf default style; breaking changes в proto без мажорной версии пакета запрещены.
- Обязательные интерсепторы (аналог middleware-цепочки): recover, logging, otel, auth (проброс `user_id` и ролей в metadata), таймауты.
- Межсервисные вызовы: таймауты обязательны, ретраи с экспоненциальным backoff + jitter только для идемпотентных методов, circuit breaker на клиентах.

## 4. Базы данных (PostgreSQL)

- **Доступ:** только через **pgx** (пул `pgxpool`); параметры пула — из конфигурации. При мультирепликах — через PgBouncer (transaction pooling; учитывать ограничения: без session-level фич).
- **Слой доступа:** чистые репозитории (интерфейс в домене, реализация в `internal/repo`) — **решение зафиксировано в ADR-005** (`DECISIONS.md`). sqlc может вводиться точечно только отдельным ADR с указанием границ применения; контракт слоя — репозиторий.
- **Только параметризованные запросы.** Конкатенация SQL со строками пользователя запрещена (SQL-инъекции).
- **Миграции:** **goose**, строго **forward-only**. Правила:
  - каждая миграция проходит ревью в PR;
  - деструктивные операции (DROP, изменение типа с потерей данных) — только с отдельным планом миграции и отдельным ADR;
  - down-миграции не используются в проде; откат — новой forward-миграцией;
  - миграции сервиса лежат в `services/<name>/migrations/` и применяются автоматически при старте сервиса (или отдельным шагом деплоя).
- **Database-per-service:** у каждого сервиса — своя БД (на старте — в одном инстансе PostgreSQL) и свой пользователь БД с правами только на неё.
- **ЗАПРЕЩЕНО:** любой доступ сервиса к чужой БД — ни чтение, ни JOIN, ни общие view, ни «временно для отчёта». Данные другого сервиса получаются только через его API или события (eventual consistency).
- Общие соглашения модели: UUIDv7 для ID; `created_at`/`updated_at` (UTC) везде; `deleted_at` для soft-delete; `version` для оптимистичной конкуренции; аудит-таблицы для чувствительных действий.

## 5. События (NATS JetStream)

- Брокер — **NATS JetStream** (ADR: выбран вместо Kafka ради лёгкости на одном хосте; Kafka — готовая замена при росте throughput).
- **Именование subjects:** строго `bozor.<domain>.<event>`, всё в нижнем регистре, событие — глагол в прошедшем времени: `bozor.ad.created`, `bozor.payment.succeeded`, `bozor.user.banned`. Новые subjects регистрируются в каталоге событий в `ARCHITECTURE.md`.
- **Envelope** (стиль CloudEvents), обязательные поля:

  ```json
  {
    "id": "uuid v7 события",
    "type": "bozor.ad.created",
    "source": "listing-ads",
    "time": "2026-07-06T12:00:00Z",
    "data": { }
  }
  ```

  Схемы `data` версионируются аддитивно; удаление/переименование полей — breaking change и требует ADR.
- **Публикация — только через transactional outbox:** доменное событие записывается в таблицу `outbox` в той же транзакции, что и изменение данных; relay-воркер публикует в JetStream (at-least-once) и помечает запись отправленной. Прямая публикация в брокер из бизнес-транзакции запрещена.
- **Консьюмеры идемпотентны:** повторная доставка того же события (по `id`) не меняет состояние повторно. Техники: таблица/Redis-набор обработанных `id`, естественная идемпотентность (upsert), уникальные ограничения.
- **DLQ обязателен:** «ядовитое» сообщение после N ретраев (backoff + jitter) уходит в dead-letter-стрим с алертом; консьюмер не блокируется на нём вечно. Обработка DLQ — ручная или по ранбуку.
- Консьюмеры используют durable-подписки с явным ack; лаг консьюмеров — метрика с алертом.

## 6. Тестирование

- **Unit-тесты:** table-driven стиль, `t.Run` для подкейсов, `t.Parallel()` где возможно. Тестируется в первую очередь доменная логика и use-cases; зависимости — за интерфейсами (моки/фейки).
- **Интеграционные тесты:** **testcontainers-go** — реальные PostgreSQL, NATS, Redis, Typesense в контейнерах. Покрывают репозитории, outbox/консьюмеры, поисковую индексацию. Помечаются build-тегом или `testing.Short()` для разделения с unit.
- **Контрактные тесты:** соответствие HTTP-ответов OpenAPI-спецификации (`api/openapi/<service>.yaml`); для gRPC — `buf breaking` в CI против main.
- **Нагрузочные тесты:** **k6** на ключевые потоки (поиск/лента, карточка объявления, публикация, чат, оплата); сценарии — в `scripts/load-tests/`; целевые метрики — из раздела НФТ (`ARCHITECTURE.md`).
- **Порог покрытия:** доменная логика (пакеты `domain` и `app`) — **не менее 70%** по строкам; порог проверяется в CI. Транспорт/генерированный код в порог не входят, но покрываются контрактными и интеграционными тестами.
- Тесты детерминированы: без sleep-синхронизации (использовать ожидание условия), без зависимости от порядка, время — через инжектируемые часы (`clock`-интерфейс).
- Красный тест блокирует merge. Флаки-тест — баг: чинится или карантинится с issue, но не игнорируется молча.

## 7. Конфигурация и секреты

- Конфигурация — **только через переменные окружения** (12-factor). Загрузка и валидация — через `pkg/shared/config` при старте; при невалидной конфигурации сервис не стартует (fail-fast).
- В корне репозитория поддерживается актуальный **`.env.example`** со всеми переменными, безопасными значениями по умолчанию и комментариями. Новая переменная без записи в `.env.example` не проходит ревью.
- **Секреты в git запрещены** в любом виде: `.env` — в `.gitignore`; токены/пароли/ключи не попадают в код, конфиги, тесты, фикстуры, логи и историю коммитов. Утёкший секрет немедленно ротируется.
- Разделение окружений (dev/prod) — только значениями env, не ветвлением кода.

## 8. Git и CI

- **Коммиты — Conventional Commits:** `feat|fix|refactor|test|docs|chore|perf|ci(scope): описание`. Scope — имя сервиса или пакета (`feat(listing-ads): ...`, `fix(shared/outbox): ...`). Breaking change помечается `!` и описывается в теле.
- **Ветки:** фичеветки от `main` (`feat/<stage>-<кратко>`, `fix/<кратко>`); прямые пуши в `main` запрещены. Ветка живёт коротко, PR — небольшой и сфокусированный.
- **PR:** по шаблону (`.github/PULL_REQUEST_TEMPLATE.md`): что сделано, как проверить, связанный Stage из `ROADMAP.md`, чек-лист (тесты, линт, миграции, обновлённые контракты/доки). Минимум одно ревью.
- **CI обязателен и блокирует merge:** пайплайн `lint → test → build` (golangci-lint; unit + интеграционные тесты; сборка бинарей и Docker-образов; `buf lint`/`buf breaking`; govulncheck). Красный CI — merge невозможен, `--no-verify` и обход проверок запрещены.

## 9. Версионирование API

- Внешний REST — с префиксом версии: **`/api/v1/...`**. Версия меняется только при несовместимых изменениях; появление `/api/v2` — отдельный ADR.
- **Изменения в рамках v1 — только аддитивные:** новые эндпоинты, новые необязательные поля запроса, новые поля ответа. Запрещено: удалять/переименовывать поля, менять типы и семантику, ужесточать валидацию существующих полей, менять коды ответов.
- Клиенты обязаны игнорировать неизвестные поля ответа (tolerant reader) — фиксируется в контрактах.
- **Политика депрекации:** устаревший эндпоинт/поле помечается `deprecated` в OpenAPI и отдаёт заголовки `Deprecation`/`Sunset`; период поддержки после объявления — минимум один релизный цикл (не менее 90 дней после появления клиентов); удаление — только в следующей мажорной версии. Все депрекации ведутся списком в `ARCHITECTURE.md`.
- gRPC: аддитивность полей protobuf (номера полей не переиспользуются, `reserved` для удалённых), `buf breaking` — страж совместимости.

## 10. Definition of Done (шаблон для каждого Stage)

Stage считается завершённым, только когда выполнены **все** пункты:

- [ ] Код написан и соответствует настоящим правилам.
- [ ] Unit- и интеграционные тесты написаны и зелёные; покрытие доменной логики ≥ 70%.
- [ ] `golangci-lint` — без замечаний; `gofmt`/`goimports` чисто.
- [ ] Миграции goose применяются с нуля и на существующей БД без ошибок.
- [ ] Сервис поднимается в `docker compose up`, health/readiness-пробы зелёные.
- [ ] Эндпоинты соответствуют OpenAPI/proto-контрактам (контрактные тесты зелёные).
- [ ] Трейсы, метрики и структурированные логи отдаются (проверено в Grafana/Tempo/Loki).
- [ ] Секреты не попали в git; `.env.example` актуален.
- [ ] `STATUS.yaml` обновлён (новый `status_id`, `completed`, `current`, `next_actions`, `metrics`).
- [ ] Архитектурные решения этапа записаны в `DECISIONS.md` (ADR); при необходимости обновлён `ARCHITECTURE.md`.
- [ ] CI на PR зелёный; PR отревьюен и смержен.

## 11. Инварианты (нарушать нельзя)

Эти правила не имеют исключений; любое отступление возможно только через явный ADR и подтверждение заказчика:

1. **Никакого прямого доступа сервиса к чужой БД.** Только API (gRPC/REST) и события.
2. **Каждое доменное событие публикуется через outbox; каждый консьюмер идемпотентен.**
3. **Нет IDOR:** каждое приватное чтение и каждая запись проверяют владение ресурсом (или роль moderator/admin).
4. **Секреты — только в env; никогда в git** (ни в коде, ни в конфигах, ни в истории).
5. **`STATUS.yaml` обновляется в конце каждого Stage; решения фиксируются в `DECISIONS.md`.**
6. **Все пользовательские строки локализованы** (uz/ru): уведомления, ошибки для клиента, справочные данные.
7. **Деньги — целочисленно в минимальных единицах (тийины); валюта хранится явно.**
