## 1. Технологический стек

### Обоснование Compose-совместимого лёгкого стека

Заказчик выбрал Docker Compose ради простоты старта на одном хосте. Поэтому по умолчанию взяты инфраструктурные компоненты с **минимальным ресурсным следом**:

- **NATS JetStream вместо Kafka** — один лёгкий бинарник вместо кластера брокеров с ZooKeeper/KRaft; при этом даёт персистентные стримы, at-least-once доставку, consumer groups и DLQ-паттерны. Kafka остаётся готовой заменой при выходе на высокий throughput (сотни тысяч событий/с).
- **Typesense вместо OpenSearch** — потребляет на порядок меньше памяти, из коробки даёт typo-tolerance, фасеты и гео-поиск, чего достаточно для classifieds-поиска. OpenSearch — замена при потребности в сложных агрегатах и аналитике.
- **MinIO** — S3-совместимое хранилище, которое при росте прозрачно заменяется на managed S3-совместимый сервис без изменения кода.

Ключевой принцип: **лёгкий стек не жертвует масштабируемостью**. Все прикладные сервисы stateless и масштабируются репликами; состояние живёт только в БД/кеше/брокере/хранилище. Тяжёлые компоненты (PostgreSQL, Typesense, NATS) при росте выносятся на отдельные хосты или managed-сервисы при переходе на Swarm/K8s. Все такие развилки зафиксированы в `DECISIONS.md` и остаются пересматриваемыми.

## 2. Топология системы и ключевые принципы

### 2.1. Топология

```
                         ┌────────────────────────┐
   Клиенты (будущие  ───▶│  nginx / Traefik (edge) │  TLS, LB, статик
   моб./веб)             │  + API Gateway (Go)     │  authN(JWT), routing, rate-limit
                         └───────────┬────────────┘
                                     │ gRPC / HTTP
        ┌──────────────┬─────────────┼───────────────┬───────────────┐
        ▼              ▼             ▼               ▼               ▼
     Auth          User/Profile   Catalog         Listing         Search
     (Telegram)                   (+Location)     (Ads)           (Typesense)
        │              │             │               │               ▲
        ▼              ▼             ▼               ▼               │ индексация
     Media         Favorites/     Moderation     Notification     (по событиям)
     (MinIO)       SavedSearch                   (Telegram-бот)
        │                                            ▲
        ▼                                            │
      Chat (WS + backplane)   Payments/Promotions ───┘   Reviews/Ratings

   Инфраструктура: PostgreSQL(×N БД) · PgBouncer · Redis · NATS JetStream ·
                   Typesense · MinIO+CDN · OTel/Prometheus/Grafana/Loki/Tempo
```

Всего **13 прикладных сервисов**: `auth`, `user-profile`, `catalog`, `location`, `listing-ads`, `media`, `search`, `favorites-savedsearch`, `chat`, `notification`, `moderation`, `payments-promotions`, `reviews` — плюс API Gateway и edge-балансировщик.

### 2.2. Database-per-service

- На старте — **один инстанс PostgreSQL с отдельной базой на каждый сервис**. Это логическая изоляция: никаких кросс-сервисных JOIN, никаких внешних ключей между базами разных сервисов и **никакого прямого доступа к чужой БД** (инвариант проекта).
- Кросс-сервисные связи — только по ID; консистентность между сервисами — через события (eventual consistency).
- Путь разделения на отдельные инстансы задокументирован: поскольку базы уже изолированы, любую из них можно перенести на отдельный хост/managed-инстанс изменением строки подключения, без рефакторинга кода.

### 2.3. Синхронное и асинхронное взаимодействие

- **Синхронно — gRPC** (protobuf, buf): для запрос-ответных сценариев между сервисами, где нужен немедленный результат (например, Listing валидирует атрибуты против Catalog).
- **Асинхронно — события через NATS JetStream**: для интеграции, реакции на изменения и устранения временной связанности (индексация поиска, уведомления, матчинг сохранённых поисков, обновление кешей рейтингов).

Правило выбора: если вызывающий сервис не может продолжить без ответа — gRPC; если это «уведомление о факте», которое можно обработать позже, — событие.

### 2.4. Transactional Outbox

Проблема: нельзя атомарно записать в БД и опубликовать в брокер — при сбое между двумя операциями данные и события расходятся.

Решение — паттерн **transactional outbox**:

1. Любое доменное событие пишется в таблицу `outbox` (id события, type, payload, created_at, published_at NULL) **в той же транзакции**, что и изменение бизнес-данных. Либо коммитится всё вместе, либо ничего — событие не может «потеряться» и не может быть опубликовано без изменения данных.
2. Отдельный **relay-воркер** (в составе каждого сервиса-издателя) периодически/по нотификации читает неопубликованные записи outbox, публикует их в NATS JetStream и помечает как отправленные (`published_at`).
3. Семантика доставки — **at-least-once**: при сбое relay между публикацией и пометкой событие уйдёт повторно.
4. Поэтому **все потребители обязаны быть идемпотентными**: дедупликация по `id` события (уникальный UUIDv7 в envelope), таблица/кеш обработанных событий или естественная идемпотентность операции (upsert).
5. **DLQ (dead-letter queue)**: «ядовитые» сообщения, которые не удаётся обработать после N ретраев с backoff, перекладываются в DLQ-стрим для ручного разбора; консьюмер не блокируется навсегда.

### 2.5. CQRS-lite

Поисковый сервис (Typesense) — это **read-модель** для листингов:

- источник истины по объявлениям — БД Listing-сервиса (write-модель);
- read-модель строится **из событий** `bozor.ad.*`, а не запросами к чужой БД;
- листинговые/поисковые чтения (самый горячий трафик) уходят в Typesense, разгружая PostgreSQL;
- read-модель eventually consistent (целевой лаг индексации — секунды); при рассинхронизации есть инструмент полной переиндексации (reindex).

### 2.6. API Gateway

Единая точка входа за nginx/Traefik. Отвечает за:

- **маршрутизацию** внешних REST-запросов `/api/v1/*` к сервисам;
- **проверку JWT** (подпись, exp, отзыв) и проброс `user_id`/claims во внутренние вызовы — сервисы доверяют уже проверенной идентичности;
- **rate limiting** — Redis token-bucket по пользователю/IP/маршруту, с заголовками `X-RateLimit-*`;
- **CORS**;
- **единый формат ошибок** RFC 7807;
- **request-id/correlation-id** — генерация `X-Request-Id`, если не пришёл, и сквозной проброс во все вызовы, логи и трейсы;
- агрегирование ответов нескольких сервисов при необходимости.

Gateway **не содержит бизнес-логики** — это принципиальное ограничение: вся доменная логика живёт в сервисах, gateway остаётся тонким и легко заменяемым.

### 3.1. Subjects

| `bozor.ad.deleted` | Listing | Search, Favorites, Payments (возвраты) |

| `bozor.moderation.report_created` | Moderation | внутренняя очередь жалоб |

### 3.2. Envelope (в стиле CloudEvents)

Каждое событие оборачивается в единый конверт:

```json
{
  "id": "018f6f2a-7c3e-7b21-9d4a-0242ac120002",
  "type": "bozor.ad.approved",
  "source": "moderation-service",
  "time": "2026-07-06T12:34:56.789Z",
  "specversion": "1.0",
  "datacontenttype": "application/json",
  "data": {
    "ad_id": "018f6f2a-...",
    "user_id": "018f6e11-...",
    "category_id": "...",
    "region_id": "..."
  }
}
```

- `id` — UUIDv7 события, ключ идемпотентности/дедупликации у потребителей;
- `type` — subject-совместимый тип события;
- `source` — сервис-издатель;
- `time` — UTC-время возникновения (момент коммита бизнес-транзакции);
- `data` — доменная нагрузка; схемы полей фиксируются в контрактах (`api/`).

Правила: события **аддитивны** (новые поля добавляются, старые не ломаются); каждый издатель пишет события только через outbox; потребители идемпотентны; необрабатываемые сообщения — в DLQ.

## 4. Спецификация сервисов

Для каждого сервиса: зона ответственности, ключевые таблицы БД, основные API, публикуемые/потребляемые события, особая логика. Полные схемы БД (ERD) и OpenAPI/proto создаются по мере реализации и живут здесь и в `api/`.

### 4.1. Auth / Identity — авторизация через Telegram-бота

**Зона ответственности:** регистрация и вход по номеру телефона через Telegram-бота, выпуск и ротация токенов, сессии, аудит авторизаций. Хранит идентичность; расширенный профиль — в User/Profile.

**Flow авторизации (без фронтенда, через request_contact):**

1. Клиент инициирует вход: `POST /api/v1/auth/session/init` → сервис создаёт одноразовый **nonce** (TTL 2–5 минут, single-use, хранится в Redis) и возвращает:
   ```json
   { "nonce": "...", "bot_deeplink": "https://t.me/<bot>?start=<nonce>", "expires_at": "..." }
   ```
2. Пользователь открывает бота по deep-link и жмёт **Start**. Бот отправляет сообщение с reply-кнопкой `KeyboardButton{ text: "📱 Отправить номер", request_contact: true }`.
3. Пользователь нажимает кнопку → Telegram присылает боту сообщение с полем `contact`.
4. Webhook-обработчик Auth-сервиса (`POST /api/v1/auth/telegram/webhook`, внутренний):
   - проверяет подлинность запроса по заголовку `X-Telegram-Bot-Api-Secret-Token` (secret задаётся при `setWebhook`);
   - **КРИТИЧНО:** проверяет `message.contact.user_id == message.from.id` — иначе пользователь мог переслать **чужой** контакт; такие запросы отклоняются;
   - нормализует телефон в E.164 (`+998…`) и валидирует;
   - делает upsert пользователя по `telegram_user_id` (сохраняет phone, first_name, last_name, username, language_code);
   - если сообщение пришло по `/start <nonce>` — связывает аутентифицированного пользователя с этим nonce: помечает сессию как `confirmed` и генерирует пару токенов.
5. Клиент опрашивает `GET /api/v1/auth/session/{nonce}` → `pending` | `confirmed` (+ `access_token`, `refresh_token`) | `expired`.
6. Токены:
   - **JWT access** — TTL ~15 минут; claims: `sub`, `roles`, `exp`, `iat`, `jti`;
   - **refresh** — TTL ~30 дней; хранится **хешированным**; **ротируется при каждом использовании**; привязан к `device_id`; объединён в семейство (`family_id`);
   - **reuse-detection:** повторное использование уже отозванного/использованного refresh-токена трактуется как компрометация → **отзывается всё семейство токенов** этого устройства, пользователь вынужден войти заново.

**Таблицы:**
- `users` (id UUIDv7, telegram_user_id UNIQUE, phone, username, first_name, last_name, language_code, status, created_at)
- `refresh_tokens` (id, user_id, token_hash, device_id, family_id, revoked_at, expires_at)
- `auth_audit_log` (события входа/выхода/ротации/reuse-detection)
- `outbox`

**API:** `POST /api/v1/auth/session/init`, `GET /api/v1/auth/session/{nonce}`, `POST /api/v1/auth/telegram/webhook` (внутренний), `POST /api/v1/auth/refresh`, `POST /api/v1/auth/logout`.

**События:** публикует `bozor.user.created` (при первом upsert пользователя).

**Безопасность:** rate-limit на `init` и webhook; nonce single-use + TTL; проверка владения контактом; secret token вебхука; ротация refresh + reuse-detection; полный аудит.

### 4.2. User / Profile

**Зона ответственности:** расширенный профиль пользователя, настройки, контактные предпочтения, тип аккаунта (физлицо/бизнес), настройки уведомлений, кеш агрегированного рейтинга.

**Таблицы:**
- `profiles` (user_id, display_name, avatar_media_id, about, user_type[individual|business], business_name, city_id, contact_phone_visible, created_at)
- `notification_prefs` (user_id, channel, event_type, enabled)
- `user_ratings_cache` (user_id, avg_rating, reviews_count) — обновляется по событию из Reviews

**API:** `GET /api/v1/me`, `PATCH /api/v1/me`, `GET /api/v1/users/{id}` (публичный профиль продавца), `GET/PUT /api/v1/me/notification-prefs`.

**События:** потребляет `bozor.user.created` (создание профиля), `bozor.review.created` (пересчёт кеша рейтинга); публикует `bozor.user.updated`.

**Примечание на вырост:** `user_type=business` заложен в модель, но storefront-функционал (бизнес-витрины) — вне scope v1.

### 4.3. Catalog — категории и атрибуты

**Зона ответственности:** дерево категорий и определения динамических атрибутов/фильтров под каждую категорию. Данные справочные, редко меняются, **агрессивно кешируются** (Redis, инвалидация по событию изменения).

**Сид — 12 разделов верхнего уровня (по OLX.uz):** Детский мир, Недвижимость, Транспорт, Работа, Животные, Дом и сад, Электроника, Бизнес и услуги, Мода и стиль, Хобби/отдых/спорт, Отдам даром, Обмен — плюс репрезентативные подкатегории. Минимум для трёх категорий (Недвижимость, Транспорт, Электроника) заводятся примерные атрибуты.

**Таблицы:**
- `categories` (id, parent_id, slug, name_uz, name_ru, icon, sort_order, level, path)
- `attributes` (id, key, name_uz, name_ru, type[enum|int|decimal|bool|string], unit, is_required, is_filterable, options_json — для enum)
- `category_attributes` (category_id, attribute_id) — привязка с **наследованием вниз по дереву** категорий

**API:** `GET /api/v1/categories` (дерево), `GET /api/v1/categories/{id}`, `GET /api/v1/categories/{id}/attributes` (с учётом наследования). Админские CRUD — под ролями moderator/admin.

> Возможное решение на старте: объединить Catalog и Location в один «reference»-сервис (фиксируется в `DECISIONS.md`).

### 4.4. Location — регионы и города

**Зона ответственности:** справочник регионов/городов/районов Узбекистана (Ташкент и районы, Самарканд, Бухара, Наманган, Андижан, Фергана, Коканд, Джизак, Нукус и т.д.). Почти статика, агрессивно кешируется. Засевается реальными данными.

**Таблицы:**
- `regions` (id, name_uz, name_ru, sort_order)
- `cities` (id, region_id, name_uz, name_ru, lat, lng)

**API:** `GET /api/v1/regions`, `GET /api/v1/regions/{id}/cities`.

### 4.5. Listing / Ads — объявления

**Зона ответственности:** CRUD объявлений, значения атрибутов (валидируются против Catalog), привязка изображений, полный жизненный цикл, счётчик просмотров. **Источник истины по объявлениям.**

**Статусная машина:**

```
draft ──▶ pending ──▶ active ──▶ sold
             │  ▲        │  ├──▶ expired (по expires_at, воркером)
             ▼  │        │  └──▶ archived
          rejected ──────┘
          (редактирование → снова pending)

          любой статус ──▶ blocked (действием модерации)
```

- `draft` → `pending` — отправка на модерацию;
- `pending` → `active` — одобрено (авто- или ручной модерацией);
- `pending` → `rejected` — отклонено с причиной; после правки объявление снова уходит в `pending`;
- `active` → `sold` | `expired` | `archived`;
- любой → `blocked` — блокировка модерацией;
- правка **активного** объявления по правилам (изменение ключевых полей: заголовок, описание, цена, категория, фото) отправляет его на **повторную модерацию**.

**Таблицы:**
- `ads` (id UUIDv7, user_id, category_id, title, description, price, currency, region_id, city_id, lat, lng, status, phone_display, contact_prefs, published_at, expires_at, bumped_at, views_count, created_at, updated_at)
- `ad_attribute_values` (ad_id, attribute_key, value — типизированные колонки или JSONB с валидацией по Catalog)
- `ad_images` (ad_id, media_id, sort_order, is_cover)
- `outbox`

**Особая логика:**
- **Истечение срока:** воркер переводит `active` → `expired` по `expires_at`; продление (`renew`) сдвигает срок.
- **Асинхронный счётчик просмотров:** инкременты буферизуются в Redis (`INCR` по ad_id), периодический воркер флешит агрегаты в PostgreSQL батчем. Это устраняет write-hotspot: карточка популярного объявления не порождает поток одиночных UPDATE в БД.
- Все переходы статусов и изменения публикуются событиями `bozor.ad.*` через transactional outbox.

**API:** `POST /api/v1/ads` (создание), `GET /api/v1/ads/{id}`, `PATCH /api/v1/ads/{id}`, `DELETE /api/v1/ads/{id}`, `POST /api/v1/ads/{id}/sold`, `POST /api/v1/ads/{id}/renew`, `GET /api/v1/ads` (лента активных, пагинация/сортировка), `GET /api/v1/me/ads` (мои объявления по статусам).

**События:** публикует `bozor.ad.created|updated|deleted|sold|expired|bumped`; потребляет решения модерации (`bozor.ad.approved|rejected`) для смены статуса.

### 4.6. Media — изображения

**Зона ответственности:** загрузка, валидация, обработка и хранение изображений; выдача через CDN.

**Flow:** `POST /api/v1/media` (multipart) → валидация (MIME-тип, размер файла, размеры изображения, лимит N изображений на объявление) → сохранение оригинала в MinIO → постановка задачи обработки в брокер → воркер генерирует размеры (например, 120/480/1080 px), удаляет EXIF, опционально накладывает водяной знак. NSFW/вирус-проверка заложена интерфейсно на будущее. Выдача — presigned URL / CDN URL.

**Таблицы:**
- `media` (id, owner_user_id, bucket, object_key, status[uploaded|processed], sizes_json, created_at)

**Особая логика:** очистка «сирот» — медиа, не привязанных ни к одному объявлению, удаляются воркером по TTL.

**API:** `POST /api/v1/media`, `GET /api/v1/media/{id}` (метаданные/URL).

### 4.7. Search — поиск объявлений

**Зона ответственности:** индексация объявлений в Typesense и полнотекстовый + фасетный поиск. Read-модель (CQRS-lite), не источник истины.

**Схема коллекции Typesense `ads`:**

| Поле | Назначение |
|---|---|
| `id` | id объявления |
| `title`, `description` | полнотекст, поля под uz/ru |
| `category_id`, `category_path` | фильтр/фасет, поиск по поддереву категорий |
| `region_id`, `city_id` | гео-фильтры |
| `price`, `currency` | диапазонный фильтр, сортировка |
| `attrs.*` | плоские типизированные поля-фасеты динамических атрибутов |
| `created_at`, `bumped_at` | сортировки (свежесть, поднятие) |
| `is_top`, `promotion_rank` | платное продвижение, топ-блок |
| `status` | индексируются только активные |
| `geo (lat,lng)` | гео-поиск/сортировка по расстоянию |

**Индексация:** консьюмер событий `bozor.ad.created|updated|deleted|approved|bumped|sold|expired` (цепочка outbox → NATS → consumer → Typesense). Upsert/delete документов идемпотентны. **Инструмент полной переиндексации (reindex)** обязателен — на случай изменения схемы или рассинхронизации read-модели: скрипт вычитывает объявления из Listing (API/экспорт) и перезаливает коллекцию (blue/green: новая коллекция + переключение alias).

**Возможности запросов:** полнотекст uz/ru с typo-tolerance; фильтры (категория с поддеревом, регион/город, диапазон цены, атрибуты категории как фасеты); сортировки (релевантность, цена ↑/↓, дата, `bumped_at`); **Топ-объявления (featured) — сверху выдачи**; пагинация (cursor/offset); гео-поиск и сортировка по расстоянию.

**API:** `GET /api/v1/ads/search` (фильтры, фасеты, сортировка, пагинация), `GET /api/v1/ads/search/facets` (доступные фасеты для категории).

### 4.8. Favorites / Saved-Search — избранное и сохранённые поиски

**Зона ответственности:** избранное и сохранённые поиски с уведомлениями о новых совпадениях.

**Таблицы:**
- `favorites` (user_id, ad_id, created_at)
- `saved_searches` (id, user_id, name, query_json, notify_enabled, last_notified_at, created_at)

**Стратегия matcher (важно для масштаба).** У Typesense нет percolate-запросов (как в Elasticsearch), поэтому «обратный поиск» реализуется собственным matcher-компонентом поверх событий:

1. **Триггер:** событие `bozor.ad.approved` (новое активное объявление) приходит в matcher.
2. **Оценка фильтров:** matcher прогоняет атрибуты объявления (категория, регион/город, цена, значения атрибутов, при необходимости — текстовые термы) против условий сохранённых поисков. Для эффективности сохранённые поиски держатся в памяти/кеше, **индексированные по грубым ключам** (category_id, region_id) — так на одно объявление проверяется не вся база поисков, а только потенциально совпадающее подмножество.
3. **Батчинг:** совпадения не отправляются по одному — накапливаются в окно (например, 30–60 с) и публикуются пакетом событий для Notification.
4. **Дедупликация:** пара (saved_search_id, ad_id) уведомляется не более одного раза; ключ дедупа — в Redis/таблице; `last_notified_at` ограничивает частоту уведомлений на один сохранённый поиск.
5. **Защита от лавины:** при массовом одобрении объявлений (после reindex, разбора очереди модерации, ретраев) — rate limiting исходящих уведомлений на пользователя, ограничение размера батча, back-pressure через pull-consumer JetStream; при переполнении — деградация до дайджеста «N новых объявлений по вашему поиску» вместо поштучных уведомлений.
6. Matcher — идемпотентный консьюмер (at-least-once), с DLQ.

**API:** `POST/DELETE /api/v1/favorites/{adId}`, `GET /api/v1/me/favorites`; `POST /api/v1/saved-searches`, `GET /api/v1/me/saved-searches`, `DELETE /api/v1/saved-searches/{id}`.

**События:** потребляет `bozor.ad.approved`, `bozor.ad.deleted`; публикует события совпадений для Notification.

### 4.9. Chat — чат покупатель↔продавец

**Зона ответственности:** переписка по объявлению между покупателем и продавцом: WebSocket в реальном времени + REST-история.

**Таблицы:**
- `conversations` (id, ad_id, buyer_id, seller_id, created_at, last_message_at) — диалог уникален по (ad_id, buyer_id, seller_id)
- `messages` (id, conversation_id, sender_id, body, created_at, read_at)

**Транспорт:** WebSocket-эндпоинт `/ws` (аутентификация по JWT — в query-параметре или первым сообщением после подключения). REST для истории: `GET /api/v1/conversations`, `GET /api/v1/conversations/{id}/messages`.

**Доставка и масштабирование (backplane обязателен):**
- Онлайн-получатель — доставка по WS; оффлайн — публикуется `bozor.chat.message_sent` → Notification отправляет сообщение в Telegram.
- **Backplane между репликами:** при нескольких репликах Chat-сервиса отправитель может быть подключён к реплике A, а получатель — к реплике B. Сообщение публикуется в шину (**Redis Pub/Sub или NATS**), все реплики подписаны и доставляют его локально подключённым получателям. Это требование горизонтального масштабирования действует даже под Compose — Chat никогда не полагается на «получатель подключён к этому же процессу».
- На edge (nginx) для WS — sticky/консистентная балансировка + поддержка upgrade-заголовков.
- Непрочитанные счётчики и отметки о прочтении (`read_at`); опционально — индикатор набора текста.
- Блокировка пользователя; жалоба на сообщение → Moderation; rate limiting на отправку сообщений.

**События:** публикует `bozor.chat.message_sent` (для оффлайн-уведомлений).

### 4.10. Notification — уведомления

**Зона ответственности:** событийные уведомления пользователям. В v1 единственный канал — **Telegram-бот**; абстракция каналов заложена для push/email в будущем.

**Шаблоны (uz/ru) по типам событий:** новое сообщение в чате; совпадение по сохранённому поиску; объявление одобрено/отклонено; объявление скоро истекает; платная услуга активирована; успешная оплата; новый отзыв.

**Логика:** потребляет доменные события → проверяет `notification_prefs` пользователя → рендерит локализованный шаблон по `language_code` → отправляет через Telegram Bot API на `telegram_user_id`. Батчинг и rate limiting (не спамить пользователя и не упираться в лимиты Bot API), хранение статуса доставки, ретраи с экспоненциальным backoff, DLQ для недоставляемых.

**Таблицы:**
- `notifications` (id, user_id, event_type, channel, payload_json, status, sent_at, attempts)

**События:** потребляет `bozor.chat.message_sent`, `bozor.ad.approved|rejected|expired`, события совпадений сохранённых поисков, `bozor.promotion.activated`, `bozor.payment.succeeded|failed`, `bozor.wallet.refunded`, `bozor.review.created`, `bozor.user.banned`.

### 4.11. Moderation — модерация и жалобы

**Зона ответственности:** проверка объявлений (авто + ручная), жалобы, баны, повторная модерация после правки.

**Логика:**
- Каждое новое/отредактированное объявление (события `bozor.ad.created|updated`) создаёт `moderation_task` (status `pending`).
- **Авто-проверки:**
  - **стоп-слова uz/ru** — минимальный словарь: «копия», «реплика», «подделка», «под оригинал» + расширяемый словарь в таблице `stopwords` (управляется админом);
  - запрещённые категории товаров;
  - простая детекция дублей (одинаковые объявления одного пользователя);
  - проверки изображений (NSFW и т.п.) — заложены на будущее.
  - Прошло все авто-проверки → объявление одобряется автоматически (`bozor.ad.approved` → `active`); не прошло → задача уходит в **ручную очередь**.
- **Ручные действия модератора:** `approve` / `reject` (с обязательной причиной) / `request-edit` (с правом правки → после правки повторная модерация).
- **Жалобы:** пользователи жалуются на объявления/пользователей/сообщения → очередь жалоб → действие модератора (предупреждение / снятие объявления / бан).
- **Баны:** временный/постоянный, опционально shadow-ban.

**Таблицы:** `moderation_tasks`, `reports`, `bans`, `stopwords`, `outbox`.

**API (роль moderator/admin):** очередь задач, действия по задаче, список жалоб, управление банами и стоп-словами. Публичный: `POST /api/v1/reports` (пожаловаться).

**События:** потребляет `bozor.ad.created|updated`; публикует `bozor.ad.approved|rejected`, `bozor.user.banned`, `bozor.moderation.report_created` → Notification и заинтересованные сервисы.

### 4.12. Payments / Promotions — платные услуги и монетизация

**Зона ответственности:** платные услуги продвижения, кошелёк, транзакции, интеграции оплаты (Payme/Click), возвраты. Ключевая доходная подсистема, моделируется по OLX.

**Каталог услуг (сид):**

| Код | Услуга | Условия |
|---|---|---|
| `TOP` | Топ-объявление | 7 или 30 дней; в топе рубрики одновременно ротируются **не более 5** оплаченных объявлений в **случайном** порядке; значок «ТОП» на карточке |
| `VIP` | Размещение на главной | 7 дней |
| `BUMP` | Поднятие в верх списка | ручное в любой момент **и** авто по расписанию из наборов |
| `LOGO` | Лого в списке | 30 дней |

**Наборы:**

| Код | Состав |
|---|---|
| `EASY_START` | Топ 3 дня |
| `FAST_SALE` | Топ 7 дней + 3 авто-поднятия через 2/4/6 дней |
| `TURBO_SALE` | Топ 30 дней + VIP 7 дней + 9 авто-поднятий через 3/6/9/12/15/18/21/24/27 дней |

**Ценообразование:** цена зависит от **региона и категории** объявления и длительности. Таблица `pricing` (service_code, region_id, category_id, duration_days, amount_uzs).

**Логика:**
- Применение услуги к объявлению создаёт `ad_promotion` (ad_id, service_code, starts_at, ends_at, status, schedule_json — расписание авто-поднятий).
- **Ротация Топа:** максимум 5 активных Топ-объявлений на категорию, случайная выборка при каждом запросе списка. Реализуется на стороне Search/Listing: Promotions помечает объявления флагом TOP с указанием категории и публикует события (`bozor.promotion.activated`); лента/поиск формирует «топ-блок», выбирая до **5 случайных активных TOP** из категории на каждый запрос.
- **Планировщик авто-поднятий:** воркер в запланированное время (из `schedule_json`) «поднимает» объявление — обновляет `bumped_at` и публикует `bozor.ad.bumped` → переиндексация в поиске.
- **Кошелёк:** баланс пользователя; пополнение через Payme/Click; списание при покупке услуги; **леджер в стиле двойной записи** — каждая операция отражается парными проводками (дебет/кредит) в `wallet_transactions`, баланс всегда сверяем с суммой проводок, история неизменяемая (append-only).
- **Провайдеры оплаты:** интерфейс
  ```go
  type PaymentProvider interface {
      CreateInvoice(...)  // выставление счёта на пополнение
      HandleCallback(...) // обработка колбэка провайдера
  }
  ```
  Адаптеры:
  - **Payme** — JSON-RPC merchant API: `CheckPerformTransaction`, `CreateTransaction`, `PerformTransaction`, `CancelTransaction`, `CheckTransaction`;
  - **Click** — двухфазный протокол `Prepare` / `Complete`;
  - **Mock** — для dev/тестов.

  **Колбэки идемпотентны** — провайдеры ретраят запросы; повторный колбэк с тем же external_id/состоянием даёт тот же результат без двойного зачисления.
- **Возвраты:** при удалении объявления модерацией с активной платной услугой — возврат **пропорционально неиспользованным дням** в кошелёк, транзакция типа `refund` (`bozor.wallet.refunded`).
- **Истечение объявления ↔ активная услуга:** если объявление истекает, а услуга ещё активна — услуга **приостанавливается** и возобновляется при реактивации объявления (как у OLX).
- Услуги применяются **асинхронно** (ориентир — минуты), не блокируя пользователя; сценарий покупки — сага с компенсациями (см. раздел 7).

**Таблицы:** `promotion_services`, `pricing`, `ad_promotions`, `wallets`, `wallet_transactions`, `payments` (провайдер, invoice, статус, external_id), `outbox`.

**API:** `GET /api/v1/promotions/catalog`, `POST /api/v1/ads/{id}/promote` (выбор услуги/набора → счёт/списание), `GET /api/v1/me/wallet`, `POST /api/v1/wallet/topup`, вебхуки провайдеров (внутренние).

**События:** публикует `bozor.promotion.activated`, `bozor.payment.succeeded|failed`, `bozor.wallet.refunded`, `bozor.ad.bumped`; потребляет `bozor.ad.deleted`, `bozor.ad.expired`.

### 4.13. Reviews / Ratings — отзывы и рейтинги

**Зона ответственности:** отзывы о продавцах и агрегированный рейтинг.

**Таблицы:**
- `reviews` (id, from_user_id, to_user_id, ad_id, rating[1..5], text, created_at)

**Ограничения и анти-абьюз:** один отзыв на взаимодействие (уникальность по from/to/ad); нельзя оставить отзыв самому себе; анти-спам квоты. Текст отзыва проходит модерацию.

**Логика:** при создании отзыва пересчитывается агрегат (avg_rating, count) и публикуется `bozor.review.created` → User/Profile обновляет `user_ratings_cache`, Notification уведомляет продавца.

**API:** `POST /api/v1/reviews`, `GET /api/v1/users/{id}/reviews`.

## 5. Общие правила модели данных

- **ID:** **UUIDv7** (сортируемые по времени) для всех сущностей — дружелюбны к B-tree-индексам и не раскрывают счётчики.
- **Временные метки:** `created_at`, `updated_at` (всегда **UTC**) во всех таблицах; где нужно — `deleted_at` (**soft-delete**: запись помечается, а не удаляется физически).
- **Изоляция:** каждая таблица принадлежит ровно одному сервису; кросс-сервисные связи — только по ID, **без внешних ключей между БД разных сервисов**; консистентность между сервисами — через события (eventual consistency).
- **Деньги:** храним **целочисленно (int64) в минимальных единицах — тийинах**, отображаем в сумах; поле валюты хранится явно (в v1 — `UZS`). Никаких float для денег.
- **Локализация:** пары полей `*_uz` / `*_ru` для справочных данных (категории, атрибуты, регионы, шаблоны); пользовательский контент (заголовок/описание объявления) — как ввёл автор, с указанием языка при необходимости.
- **Аудит:** чувствительные действия (авторизация, платежи, модерация, баны) логируются в аудит-таблицы соответствующих сервисов.
- **Оптимистичная конкуренция:** колонка `version` на сущностях с конкурентной правкой (объявления, кошельки); обновление с проверкой версии, конфликт → 409/повтор.

Полные ERD по каждому сервису дополняются в этом документе по мере реализации соответствующих Stage.

## 6. API-конвенции

- **Внешний API:** REST/JSON, версионирование префиксом **`/api/v1`**. Спецификации **OpenAPI 3.1** — в `api/openapi/<service>.yaml`. Изменения аддитивны; политика депрекации описана в `RULES.md`.
- **Внутренний API:** **gRPC**; proto — в `api/proto/<service>/`; линт и генерация через **buf**.
- **Пагинация:** **cursor-based по умолчанию** (opaque cursor); в ответе — `limit` и `next_cursor`.
- **Фильтрация/сортировка:** query-параметры, описанные в OpenAPI.
- **Ошибки:** единый формат **RFC 7807** (`application/problem+json`): `type`, `title`, `status`, `detail`, `instance`, плюс `errors[]` для ошибок валидации по полям.
- **Идемпотентность:** для небезопасных операций, где применимо (оплаты, публикация объявления), поддерживается заголовок **`Idempotency-Key`**; результат первой обработки хранится в Redis и возвращается на повторы.
- **Rate limiting:** лимиты применяются на уровне gateway (Redis token-bucket); клиент получает заголовки **`X-RateLimit-*`** (limit/remaining/reset).
- **Аутентификация:** `Authorization: Bearer <access>`; во внутренние вызовы пробрасываются `user_id` и роли. Роли: **`user`**, **`moderator`**, **`admin`**.
- **Корреляция:** сквозной **`X-Request-Id`** / trace-id во всех логах, вызовах и ответах.

Сигнатуры всех эндпоинтов из раздела 4 конкретизируются в OpenAPI по мере реализации соответствующих Stage.

## 7. Нефункциональные требования

Целевая нагрузка: **десятки тысяч DAU**, **тысячи одновременных пользователей**; система не деградирует при высокой конкуренции.

### 7.1. Масштабируемость

- Все прикладные сервисы **stateless**; горизонтальное масштабирование репликами за nginx/gateway (в Compose — несколько инстансов / `--scale`, апстрим-балансировка в nginx). Состояние — только в БД/кеше/брокере/хранилище.
- **Read-heavy паттерн:** кеш Redis (cache-aside) на горячих данных (каталог, локации, карточки), вынос листинговых/поисковых запросов в Typesense, путь к read-репликам PostgreSQL, CDN на изображения.
- **Сглаживание записи** через брокер: асинхронная индексация поиска, обработка медиа, уведомления, флеш счётчиков просмотров.
- **PostgreSQL:** корректные индексы; **PgBouncer (transaction pooling) обязателен** при мультирепликах; путь к read-репликам; **партиционирование `ads` по `created_at` (помесячно)** при росте; заложен путь к шардированию/Citus.
- **Лимиты и квоты:** ограничения на число публикаций объявлений на пользователя (антиспам), ужесточённые для новых аккаунтов.

### 7.2. Надёжность и устойчивость

- Health/readiness-пробы на каждом сервисе; **graceful shutdown** (дренаж соединений, докоммит начатых операций).
- **Ретраи с экспоненциальным backoff + jitter** и таймауты на все межсервисные вызовы; **circuit breaker** — при устойчивых сбоях зависимости запросы быстро отклоняются, не выжигая ресурсы; **bulkhead-изоляция** — отдельные пулы/лимиты на разные зависимости, чтобы деградация одной не топила остальные.
- **Transactional outbox** для всех событий; **идемпотентные** потребители; **DLQ** для «ядовитых» сообщений (раздел 2.4).
- **Saga/компенсации** для мультисервисных сценариев — хореография через события + компенсирующие транзакции. **Пример — покупка платной услуги:**
  1. Payments списывает средства с кошелька (локальная транзакция + outbox-событие);
  2. услуга применяется к объявлению (`ad_promotion`, событие `bozor.promotion.activated` → Search помечает is_top);
  3. **при сбое** на шаге применения (объявление удалено/заблокировано, ошибка) выполняется **компенсация**: возврат средств в кошелёк транзакцией `refund` (`bozor.wallet.refunded`) — вместо распределённой транзакции.

### 7.3. Производительность (целевые ориентиры, уточняются нагрузочным тестом k6)

| Поток | Цель |
|---|---|
| p95 публичного чтения (лента/поиск/карточка), прогретый кеш | < 200–300 мс |
| p95 записи (публикация объявления без обработки медиа) | < 500 мс |
| Лаг индексации нового объявления в поиске | < нескольких секунд |

### 7.4. Безопасность

- JWT: короткий access, ротация refresh, отзыв, reuse-detection; secret token Telegram-вебхука; проверка владения контактом (`contact.user_id == from.id`).
- Валидация всего ввода; только параметризованные запросы (защита от SQL-инъекций); защита от **IDOR** — проверка владения ресурсом на каждом write и чтении приватных данных.
- Rate limiting и анти-спам квоты; секреты — только через env/`.env` (в репозиторий не коммитятся), `.env.example` присутствует.
- TLS на edge; CORS; сканирование зависимостей (**govulncheck**); принципы OWASP API Security.

### 7.5. Наблюдаемость

- OpenTelemetry SDK во всех сервисах; трейсы → Tempo/Jaeger; метрики → Prometheus (**RED** на сервис: rate/errors/duration; **USE** на ресурсы); логи → Loki (структурированный JSON с trace/correlation id).
- Дашборды Grafana на ключевые потоки: авторизация, публикация, поиск, чат, платежи, лаг очередей (consumer lag), соединения БД.
- Правила алертинга: error rate, latency, лаг консьюмеров, исчерпание пула БД.

### 7.6. Бэкапы и восстановление

- **PostgreSQL PITR** (point-in-time recovery, WAL-архивация).
- Резервирование/репликация **MinIO**.
- Ранбуки восстановления (документированные процедуры).

### 7.7. Локализация

uz/ru во всех пользовательских строках, шаблонах уведомлений, ошибках для клиента и справочных данных.

## 8. Развёртывание и путь масштабирования

### 8.1. Docker Compose (старт)

- **`docker-compose.yml`** поднимает: nginx (edge, TLS, LB) + API Gateway; все 13 прикладных сервисов; PostgreSQL (+ отдельная БД на сервис) + PgBouncer; Redis; NATS JetStream; Typesense; MinIO; стек наблюдаемости (Prometheus, Grafana, Loki, Tempo). Плюс `docker-compose.override.yml` для локальной разработки и `docker-compose.prod.yml` для прод-профиля.
- **`.env` / `.env.example`** — вся конфигурация через переменные окружения (12-factor); секретов в репозитории нет.
- **Балансировка и реплики:** stateless-сервисы масштабируются несколькими инстансами; nginx настроен на апстрим-пулы. Для WS-сервиса (Chat) — sticky/консистентная балансировка + backplane (раздел 4.9).
- **Миграции и сиды:** автозапуск миграций при старте (goose, forward-only); сиды: категории, регионы/города, каталог услуг, ценообразование.
- **CI/CD (заложить):** пайплайн `lint → test → build → образы`; на PR обязателен зелёный прогон.

### 8.2. Путь масштабирования

```
Docker Compose (1 хост) ──▶ Docker Swarm (мультихост) ──▶ Kubernetes
```

1. **Compose, одиночный хост.** Реплики stateless-сервисов через `--scale` + nginx-балансировка. Вертикальный рост хоста.
2. **Docker Swarm.** Тот же Compose-формат с `deploy.replicas`; сервисы распределяются по нескольким хостам; overlay-сеть; минимальные изменения конфигурации.
3. **Kubernetes.** Deployments + HPA (автомасштабирование по метрикам) + Ingress; stateful-компоненты — в managed-сервисы или StatefulSet.

**Вынос тяжёлых компонентов** (на любом шаге, по мере роста):
- PostgreSQL → отдельный хост / managed-инстанс; затем read-реплики; затем партиционирование `ads` и путь к Citus/шардированию. Благодаря database-per-service любую базу можно вынести независимо.
- Typesense → отдельный хост/кластер; при потребности в сложных агрегатах — миграция на OpenSearch.
- NATS JetStream → кластер из 3 нод; при экстремальном throughput — миграция на Kafka (издатели изолированы за outbox/relay, поэтому смена брокера локализована).
- Redis → отдельный хост / Sentinel / Cluster.
- MinIO → распределённый режим / managed S3-совместимое хранилище; изображения — за CDN.

### 8.3. Честная оценка одиночного хоста

Одиночный Compose-хост **подходит для запуска и раннего роста**: при stateless-сервисах, лёгком инфраструктурном стеке (NATS/Typesense/MinIO), агрессивном кешировании и мощном хосте он способен обслуживать десятки тысяч DAU. Однако это единая точка отказа (нет HA), и вертикальный рост упирается в железо. При устойчивом выходе на **тысячи одновременных пользователей** и/или при требованиях к отказоустойчивости необходим переход на мультихост по описанному пути (Swarm → K8s) с выносом БД, поиска и брокера на отдельные хосты. Архитектура спроектирована так, что этот переход не требует переписывания сервисов — только изменения развёртывания и строк подключения.
