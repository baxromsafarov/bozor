# Bozor — Описание проекта

> Документ SPMS. Основан на разделах 1–2 мастер-промта (`BOZOR_MASTER_PROMPT.md`).
> Дата создания: 2026-07-06.

---

## 1. Видение

**Bozor** — backend торговой площадки объявлений (C2C/B2C classifieds) для рынка Узбекистана, функциональный аналог **olx.uz**. На текущем этапе разрабатывается **только backend** — фронтенда (веб/мобайл) нет; потребители API появятся позже.

Система строится как набор **микросервисов на Go** с хранением данных в **PostgreSQL** (паттерн database-per-service), асинхронной интеграцией через события (NATS JetStream, transactional outbox) и развёртыванием в Docker Compose с задокументированным путём масштабирования до Swarm/Kubernetes.

## 2. Цели проекта

1. Реализовать полнофункциональный backend classifieds-платформы: от авторизации и каталога до чата, модерации и монетизации.
2. Спроектировать систему под масштаб с первого дня: stateless-сервисы, горизонтальное масштабирование репликами, состояние только в БД/кеше/брокере.
3. Обеспечить надёжность и безопасность как базовые свойства: идемпотентность write-операций, transactional outbox, валидация всего ввода, защита от IDOR.
4. Построить доходную подсистему платных услуг продвижения (по модели OLX) с интеграцией локальных платёжных систем.
5. Обеспечить полную наблюдаемость (метрики, трейсы, логи) и управляемость эксплуатации.

## 3. Целевая аудитория

- **Рынок:** Узбекистан.
- **Частные продавцы и покупатели (C2C)** — основная аудитория: размещают и ищут объявления о товарах и услугах.
- **Малый бизнес (B2C)** — в модели данных заложен `user_type = business`; полноценные бизнес-витрины появятся позже.
- **Модераторы и администраторы площадки** — внутренние пользователи (очереди модерации, жалобы, баны, управление справочниками).
- **Будущие клиентские приложения** (веб, мобильные) — потребители публичного REST API.

## 4. Что делают пользователи площадки

- Регистрируются и входят через Telegram (по номеру телефона).
- Публикуют объявления с фотографиями, ценой, категорией и атрибутами.
- Ищут и фильтруют объявления по категориям, регионам, цене и характеристикам.
- Добавляют объявления в избранное, сохраняют поисковые запросы с уведомлениями о новых совпадениях.
- Общаются с продавцами/покупателями во встроенном чате.
- Продвигают объявления платными услугами (VIP, Топ, поднятие).
- Оставляют отзывы и рейтинги.
- Получают уведомления через Telegram-бота.

## 5. Рыночный контекст

Обязательно учитывается в модели данных и логике:

- **Двуязычие:** узбекский (uz) и русский (ru). Категории, атрибуты, шаблоны уведомлений, сообщения об ошибках — с локализацией.
- **Валюта:** UZS (сум); деньги хранятся целочисленно в минимальных единицах, валюта — явным полем.
- **Платёжные системы Узбекистана:** Payme и Click (+ Mock-провайдер для разработки).
- **География:** регионы и города/районы Узбекистана (Ташкент и районы, Самарканд, Бухара, Наманган, Андижан, Фергана, Коканд, Джизак, Нукус и т.д.) — справочник с реальными данными.
- **Канал коммуникации:** Telegram — и для авторизации, и для уведомлений (единственный канал уведомлений в v1).

## 6. Требования по нагрузке

Площадкой будут пользоваться **десятки тысяч человек**. Backend и база данных не должны деградировать или падать при высокой конкуренции.

Проектная цель:

- выдерживать **десятки тысяч DAU** и **тысячи одновременных пользователей**;
- горизонтально масштабируемые stateless-сервисы;
- чётко описанный путь выхода за пределы одного хоста (Compose → Swarm → Kubernetes, вынос тяжёлых компонентов на отдельные хосты/managed-сервисы).

Целевые ориентиры производительности (уточняются нагрузочным тестом):

- p95 latency публичного чтения (лента/поиск/карточка) — < 200–300 мс на прогретом кеше;
- p95 записи (публикация объявления без обработки медиа) — < 500 мс;
- индексация нового объявления в поиске — < нескольких секунд.

## 7. Метрики успеха проекта

**Продуктовые/эксплуатационные:**
- Система стабильно обслуживает десятки тысяч DAU и тысячи одновременных пользователей без деградации.
- Сквозные пользовательские сценарии работают end-to-end: вход через Telegram → публикация объявления с фото → модерация → появление в поиске → чат → покупка услуги продвижения → уведомления.
- Платёжный цикл (Payme/Click) корректен и идемпотентен, включая пропорциональные возвраты.

**Технические:**
- Целевые показатели latency (см. раздел 6) подтверждены нагрузочными тестами (k6).
- Все Stage дорожной карты завершены по Definition of Done: тесты зелёные, линт чистый, миграции применяются, сервисы поднимаются в Compose, эндпоинты соответствуют OpenAPI.
- Наблюдаемость: метрики/трейсы/логи отдаются со всех сервисов, дашборды и алерты настроены на ключевые потоки.
- Чек-лист безопасности пройден (JWT-ротация, rate limiting, анти-спам квоты, отсутствие секретов в git, скан зависимостей).
- Поисковый индекс консистентен с источником истины (Listing) через события; путь масштабирования задокументирован.

**Процессные (SPMS):**
- `STATUS.yaml` актуален после каждого Stage; все архитектурные решения зафиксированы в `DECISIONS.md`.

## 8. Границы версии v1 (scope)

Входит в v1 (обязательно реализовать):

- **Авторизация** через Telegram-бота (по номеру телефона).
- **Профили пользователей** (физлицо/бизнес — базово).
- **Каталог:** дерево категорий + динамические атрибуты категорий.
- **Справочник регионов/городов** Узбекистана.
- **Объявления:** полный жизненный цикл (черновик → модерация → активно → продано/истекло/архив/заблокировано), атрибуты, фото.
- **Медиа:** загрузка, валидация, генерация превью, хранение в объектном хранилище, отдача через CDN.
- **Поиск:** полнотекстовый (uz/ru) + фасетные фильтры + сортировка + гео.
- **Избранное** + сохранённые поиски с алертами.
- **Чат** покупатель↔продавец (WebSocket + история).
- **Модерация:** очередь проверки, авто-проверки (стоп-слова), ручные действия, жалобы, баны, повторная модерация после правки.
- **Уведомления** через Telegram-бота (событийные).
- **Платные услуги / монетизация:** VIP, Топ (с ротацией макс. 5 случайных на категорию), поднятие (ручное и авто по расписанию), лого, наборы (Лёгкий старт / Быстрая продажа / Турбо продажа), кошелёк, транзакции, интеграции Payme/Click, возвраты при удалении на модерации.
- **Рейтинги и отзывы** продавцов.
- **Наблюдаемость** (метрики, трейсы, логи) и базовые дашборды/алерты.

## 9. Non-goals: что НЕ входит в v1

Спроектировать «на вырост», но не реализовывать:

- **Фронтенд** (веб/мобайл) — потребители API появятся позже.
- **Полноценные бизнес-витрины/магазины** — в модели заложен `user_type = business`, но storefront-функционал реализуется позже.
- **Доставка/логистика** (аналог OLX Доставки) — позже.
- **ML-рекомендации и антифрод на ML** — позже.
- **Push/email как каналы уведомлений** — абстракция каналов закладывается, но в v1 реализуется только Telegram.
