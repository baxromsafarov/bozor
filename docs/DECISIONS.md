# Bozor — Журнал архитектурных решений (ADR)

Этот документ — единственный журнал архитектурных решений проекта **Bozor** (backend маркетплейса объявлений на Go, микросервисы). Каждое значимое техническое решение фиксируется здесь в формате ADR (Architecture Decision Record) и, если оно критично для системы, отражается в `ARCHITECTURE.md`.

## Формат записи

Каждый ADR оформляется по следующей схеме:

- **ADR-NNN: Заголовок** — краткая формулировка решения.
- **Статус:** `proposed` | `accepted` | `deprecated` | `superseded by ADR-NNN`.
- **Дата:** дата принятия решения (YYYY-MM-DD).
- **Автор:** кто принял решение и на каком основании.
- **Контекст:** какая проблема решается, какие ограничения и требования влияют на выбор.
- **Рассмотренные варианты:** перечень альтернатив с плюсами и минусами.
- **Решение:** что выбрано и почему.
- **Последствия:** что это означает для проекта — позитивные и негативные эффекты, а также **условия пересмотра** (при каких обстоятельствах решение нужно вернуть на обсуждение).

Нумерация сквозная (`ADR-001`, `ADR-002`, …). Принятые ADR не редактируются задним числом: если решение меняется, создаётся новый ADR со ссылкой на устаревший (`superseded by`).

## ADR-001: Брокер событий — NATS JetStream, а не Kafka

- **Статус:** accepted
- **Дата:** 2026-07-06
- **Автор:** AI-инженер, по согласованию с ТЗ (мастер-промт, раздел 3)

### Контекст

Архитектура Bozor событийно-ориентированная: все межсервисные интеграции (индексация поиска, уведомления, модерация, матчинг сохранённых поисков, обработка платежей) строятся на доменных событиях `bozor.<domain>.<event>`, публикуемых через transactional outbox. Нужен брокер с персистентностью (at-least-once), поддержкой durable-консьюмеров, ack/redelivery и DLQ.

Ключевое ограничение — развёртывание на **одном хосте под Docker Compose**. Ресурсный след инфраструктурных компонентов должен быть минимальным, чтобы на одном хосте помещались все прикладные сервисы, PostgreSQL, Redis, Typesense, MinIO и стек наблюдаемости. Целевая нагрузка (десятки тысяч DAU) не создаёт экстремального throughput по событиям.

### Рассмотренные варианты

1. **NATS JetStream**
   - Плюсы: один лёгкий бинарник, минимальное потребление CPU/RAM; персистентные stream'ы, durable/pull-консьюмеры, ack, redelivery, max-deliver + DLQ-паттерн; иерархические subjects идеально ложатся на схему `bozor.<domain>.<event>`; простая эксплуатация в Compose; отличный Go-клиент.
   - Минусы: экосистема (коннекторы, инструменты) беднее, чем у Kafka; меньше опыта индустрии на экстремальных объёмах; нет такого богатого тулинга для реплея/аналитики потоков.
2. **Apache Kafka**
   - Плюсы: индустриальный стандарт для высоких throughput; огромная экосистема (Kafka Connect, Schema Registry, ksqlDB); проверенные гарантии упорядоченности по партициям; долгое хранение и реплей.
   - Минусы: тяжёлый ресурсный след (JVM, брокеры + контроллер), заметно усложняет один Compose-хост; операционная сложность (партиции, ребалансировки, тюнинг) избыточна для текущей нагрузки.
3. **Redis Streams** (не как основной кандидат)
   - Плюсы: Redis уже есть в стеке; минимальная дополнительная инфраструктура.
   - Минусы: слабее гарантии долговременной персистентности; consumer groups менее удобны для схемы многих независимых доменных потоков; смешивание ролей кеша и брокера — операционный риск.

### Решение

Выбран **NATS JetStream** как единый брокер доменных событий. Публикация — только через transactional outbox (relay-воркер читает `outbox` и публикует в JetStream), потребители идемпотентны, «ядовитые» сообщения после исчерпания max-deliver уходят в DLQ-stream.

### Последствия

- Инфраструктура остаётся лёгкой: весь стек уверенно помещается на один Compose-хост.
- Код прикладных сервисов не должен зависеть от специфики брокера: публикация/подписка инкапсулируются в `pkg/shared` (тонкий слой поверх клиента), чтобы замена брокера не расползалась по кодовой базе.
- Semantика доставки — at-least-once; дедупликация обязана обеспечиваться на стороне консьюмеров (идемпотентность по `event_id`).
- **Условия пересмотра:** устойчивый рост throughput событий за пределы комфортной работы JetStream на выделенном хосте (сотни тысяч событий/сек), потребность в длительном хранении и реплее потоков для аналитики, потребность в экосистеме Kafka Connect. В этом случае **Kafka — задокументированная замена**; путь миграции: перевод relay-воркера и консьюмеров на Kafka-клиент при сохранении envelope и именования событий.

## ADR-002: Поисковый движок — Typesense, а не OpenSearch / PostgreSQL FTS

- **Статус:** accepted
- **Дата:** 2026-07-06
- **Автор:** AI-инженер, по согласованию с ТЗ (мастер-промт, раздел 3)

### Контекст

Поиск — центральный read-путь площадки: полнотекстовый поиск на узбекском и русском с typo-tolerance, фасетные фильтры по динамическим атрибутам категорий, сортировки (релевантность, цена, дата, `bumped_at`, топ-блок), гео-поиск по координатам и пагинация. Поисковый индекс — это read-модель (CQRS-lite), которая строится из событий Listing-сервиса.

Ограничения те же, что и в ADR-001: один Compose-хост, минимальный ресурсный след, при этом p95 поиска целево < 200–300 мс.

### Рассмотренные варианты

1. **Typesense**
   - Плюсы: очень лёгкий (C++, один бинарник, in-memory индекс) — комфортно живёт на общем хосте; фасеты, typo-tolerance и гео-поиск **из коробки** без тюнинга; простой API и предсказуемая низкая латентность; простая эксплуатация.
   - Минусы: индекс в RAM — объём данных ограничен памятью; нет сложных агрегаций/аналитики (как у Elasticsearch aggregations); меньше гибкости анализаторов для морфологии; кластерные возможности скромнее.
2. **OpenSearch / Elasticsearch**
   - Плюсы: богатейшие возможности — сложные агрегации, кастомные анализаторы (морфология русского), percolate-запросы (полезно для сохранённых поисков), зрелое кластерирование.
   - Минусы: JVM, большой ресурсный след (гигабайты RAM только на движок), операционная сложность (шарды, реплики, тюнинг heap) — тяжело для одного Compose-хоста.
3. **PostgreSQL FTS (tsvector/pg_trgm)**
   - Плюсы: нулевая дополнительная инфраструктура; транзакционная консистентность с данными.
   - Минусы: слабая typo-tolerance; фасетирование по динамическим атрибутам — тяжёлые GROUP BY на нагруженной БД; смешивает поисковую нагрузку с OLTP-нагрузкой основной БД, что противоречит цели разгрузки PostgreSQL; нет удобного гео-ранжирования «из коробки».

### Решение

Выбран **Typesense** как поисковый движок для коллекции `ads`. Индексация — асинхронная, по событиям `bozor.ad.*` (outbox → consumer → Typesense); обязателен инструмент полной переиндексации (reindex) из БД Listing-сервиса.

### Последствия

- Быстрый старт: фасеты, typo и гео работают без длительного тюнинга; поисковая нагрузка снята с PostgreSQL.
- Так как у Typesense нет percolate, матчинг сохранённых поисков реализуется собственным matcher-компонентом (см. спецификацию Favorites/Saved-Search); это осознанная плата за лёгкость движка.
- Объём индекса нужно контролировать (только активные объявления + необходимые поля), т.к. индекс держится в памяти.
- Слой поиска изолируется за интерфейсом в Search-сервисе, чтобы замена движка не затронула публичный API `GET /api/v1/ads/search`.
- **Условия пересмотра:** потребность в сложных агрегациях/аналитике по объявлениям, продвинутой морфологии uz/ru, либо рост индекса за пределы доступной RAM. В этом случае **OpenSearch — задокументированная замена**; путь миграции: реализация второго индексатора и переключение read-пути после полной переиндексации.

## ADR-003: Авторизация — Telegram-бот с request_contact + nonce/deep-link

- **Статус:** accepted
- **Дата:** 2026-07-06
- **Автор:** AI-инженер, по согласованию с ТЗ (мастер-промт, раздел 5.1)

### Контекст

В v1 у Bozor **нет фронтенда** — есть только API. При этом требуется регистрация и вход по номеру телефона, а канал взаимодействия с пользователем — Telegram (он же используется для уведомлений). Нужен flow, который: (а) работает без веб-страницы, (б) надёжно подтверждает владение номером телефона, (в) позволяет клиенту API (в будущем — мобильному/веб-приложению) получить токены после подтверждения в боте.

Telegram предоставляет несколько механизмов аутентификации, каждый со своими предпосылками (наличие сайта, наличие Mini App), поэтому выбор влияет на всю архитектуру Auth-сервиса.

### Рассмотренные варианты

1. **Бот + `request_contact` + nonce/deep-link** (выбран)
   - Плюсы: работает полностью без фронтенда — клиент вызывает `POST /auth/session/init`, получает `nonce` и deep-link `https://t.me/<bot>?start=<nonce>`, пользователь подтверждает номер кнопкой `request_contact`, клиент опрашивает статус сессии; бот и так нужен для уведомлений; Telegram сам подтверждает номер (контакт приходит из аккаунта); nonce (TTL 2–5 мин, single-use, Redis) криптографически связывает клиентскую сессию с Telegram-пользователем.
   - Минусы: собственный протокол связывания (nonce + polling), который нужно аккуратно реализовать; критичная проверка `message.contact.user_id == message.from.id` против пересылки чужого контакта; зависимость доступности входа от Telegram Bot API.
2. **Telegram Login Widget**
   - Плюсы: стандартный, документированный механизм с HMAC-проверкой данных.
   - Минусы: требует **веб-страницу** с виджетом — фронтенда нет; не отдаёт номер телефона (только Telegram ID и имя), а вход по номеру — требование ТЗ. **Отклонён.**
3. **Mini App `initData`**
   - Плюсы: криптографически подписанные данные пользователя, современный UX внутри Telegram.
   - Минусы: требует разработки Mini App (то есть фронтенда, пусть и встроенного в Telegram) — вне scope v1; номер телефона всё равно нужно запрашивать отдельно. **Отклонён.**

### Решение

Реализуется flow «бот + `request_contact` + nonce/deep-link» строго по разделу 5.1 ТЗ: init-сессия с nonce → deep-link в бота → кнопка отправки контакта → webhook с проверкой `X-Telegram-Bot-Api-Secret-Token` и владения контактом → связывание с nonce → выдача пары JWT (см. ADR-012) → клиент получает токены поллингом `GET /auth/session/{nonce}`.

### Последствия

- Auth-сервис самодостаточен и тестируем без какого-либо фронтенда; будущие клиенты (mobile/web) используют тот же init/poll-протокол.
- Обязательные меры безопасности становятся частью DoD: nonce single-use + TTL, secret_token вебхука, проверка `contact.user_id == from.id`, нормализация телефона в E.164, rate limiting на `init` и webhook, аудит в `auth_audit_log`.
- Появляется операционная зависимость от Telegram Bot API — при его недоступности вход невозможен (принимаем: уведомления в v1 всё равно только через Telegram).
- **Условия пересмотра:** появление веб-фронтенда (можно добавить Login Widget как дополнительный способ входа), появление Mini App (добавить проверку `initData`), либо требование альтернативных каналов входа (SMS OTP через местных провайдеров). Новые способы входа добавляются рядом, не ломая существующий flow.

## ADR-004: Catalog и Location — отдельные микросервисы

- **Статус:** accepted
- **Дата:** 2026-07-06
- **Автор:** AI-инженер, по согласованию с ТЗ (мастер-промт, разделы 5.3, 5.6)

### Контекст

Catalog (дерево категорий + динамические атрибуты) и Location (регионы/города Узбекистана) — справочные сервисы: данные меняются редко, читаются часто, агрессивно кешируются. ТЗ явно допускает объединение их в один «reference»-сервис на старте с требованием зафиксировать выбор в этом журнале.

Аргументы «за объединение» очевидны: меньше процессов, меньше БД, меньше деплой-юнитов. Нужно решить, перевешивают ли они чистоту границ.

### Рассмотренные варианты

1. **Два отдельных сервиса: Catalog и Location** (выбран)
   - Плюсы: чистый паттерн database-per-service без компромиссов с первого дня; домены действительно разные — у Catalog нетривиальная логика (наследование атрибутов по дереву, админский CRUD, инвалидация кеша), у Location почти статика; независимая эволюция и деплой; Go-сервисы дёшевы по ресурсам (десятки МБ RAM на инстанс), стоимость лишнего контейнера пренебрежима; структура сервисов 1:1 соответствует спецификации ТЗ (раздел 5) и топологии.
   - Минусы: чуть больше boilerplate (два скелета, две БД, две записи в Compose); два справочных клиента у потребителей вместо одного.
2. **Единый reference-сервис (Catalog + Location)** — допустимая альтернатива по ТЗ
   - Плюсы: меньше инфраструктурных единиц на старте; один кеш-слой и один клиент.
   - Минусы: смешение доменов в одной БД и кодовой базе; будущее разделение (если Catalog обрастёт логикой) — болезненная миграция с разрезанием БД; ломается симметрия «сервис из раздела 5 ТЗ = деплой-юнит».

### Решение

Catalog и Location реализуются как **два отдельных микросервиса**, каждый со своей БД (db-per-service), своим кешем (Redis, cache-aside с инвалидацией по событию) и своим публичным API. Общий каркас (скелет сервиса, middleware, кеш-хелперы) выносится в `pkg/shared`, что снижает цену дублирования.

### Последствия

- Границы сервисов совпадают со спецификацией ТЗ — навигация по проекту и по SPMS-документам однозначна.
- Немного больше стартовой работы в CHAPTER 2 (два скелета вместо одного); компенсируется шаблонностью справочных сервисов.
- Потребители (Listing, Search, gateway) обращаются к двум справочным API; для валидации атрибутов Listing использует Catalog, для валидации гео — Location.
- **Условия пересмотра:** если эксплуатация покажет, что накладные расходы на два процесса ощутимы для хоста (маловероятно для Go), или оба сервиса останутся тривиальными CRUD без собственной логики — допустимо слияние в reference-сервис отдельным ADR. Обратное (разделение) уже не потребуется — оно сделано сейчас.

## ADR-005: Доступ к БД — pgx + чистые (hand-written) репозитории, без sqlc

- **Статус:** accepted
- **Дата:** 2026-07-06
- **Автор:** AI-инженер, по согласованию с ТЗ (мастер-промт, разделы 3, 10)

### Контекст

ТЗ фиксирует PostgreSQL + драйвер **pgx** и допускает два стиля работы с запросами: генерация типизированного кода через **sqlc** либо чистые репозитории, написанные вручную. Нужно выбрать единый стиль для всех сервисов.

Определяющий фактор — характер запросов. Значительная часть критичных запросов Bozor **динамическая**: листинг объявлений (`GET /api/v1/ads`, `GET /api/v1/me/ads`) с произвольным набором фильтров (статус, категория, регион, диапазон цены), сортировок и cursor-пагинацией; админские выборки модерации; выборки промо. sqlc силён на статических запросах, но динамическое построение WHERE/ORDER BY в нём неудобно (плодятся near-duplicate запросы или запросы с частоколом `CASE WHEN $n IS NULL`).

### Рассмотренные варианты

1. **pgx + чистые репозитории** (выбран)
   - Плюсы: полный контроль над динамическим построением запросов (фильтры листинга, фасетные выборки); нет шага codegen в сборке; проще транзакционные сценарии (outbox в той же транзакции, что и доменная запись); меньше инструментов в цепочке; сканирование удобно через `pgx.CollectRows`/`RowToStructByName`.
   - Минусы: типобезопасность запросов не проверяется на этапе компиляции — нужна дисциплина: только параметризованные запросы, обязательные интеграционные тесты репозиториев на testcontainers-go; больше ручного кода на простых CRUD.
2. **sqlc**
   - Плюсы: типизированный код из SQL, ошибки запросов ловятся на этапе генерации; единообразие; меньше рутины на статических запросах.
   - Минусы: динамические фильтры листинга — главный болевой пункт; дополнительный шаг codegen и его конфигурация в каждом сервисе; сгенерированный код хуже сочетается с ручным управлением транзакциями для outbox-паттерна.
3. **ORM (GORM/ent)** — вне рекомендаций ТЗ
   - Минусы: скрытые запросы и неявное поведение противоречат требованиям производительности и явного контроля; ТЗ ORM не предлагает. **Отклонён без детальной проработки.**

### Решение

Во всех сервисах используется **pgx (pgxpool) + чистые репозитории**. Обязательные правила: только параметризованные запросы; репозитории принимают `context.Context` и абстракцию транзакции (для атомарной записи домена + outbox); каждый репозиторий покрывается интеграционными тестами на testcontainers-go; общие хелперы (построитель условий, пагинация, сканирование) — в `pkg/shared`.

### Последствия

- Гибкость на динамических запросах достигается без обходных приёмов; транзакционный outbox реализуется прямолинейно.
- Цена — дисциплина ревью SQL и обязательные интеграционные тесты (это и так входит в DoD по RULES.md).
- Отсутствие codegen упрощает Makefile/CI.
- **Условия пересмотра:** решение не «всё или ничего» — **sqlc можно вводить точечно** в сервисах/пакетах с преимущественно статическими запросами (например, Location, Reviews), если объём ручного CRUD-кода станет обременительным. Такое частичное внедрение оформляется отдельным ADR с указанием границ применения.

## ADR-006: HTTP-роутер — chi

- **Статус:** accepted
- **Дата:** 2026-07-06
- **Автор:** AI-инженер, по согласованию с ТЗ (мастер-промт, разделы 3, 10)

### Контекст

Все сервисы Bozor выставляют REST/JSON API (внешние — через gateway), и ТЗ требует **единый роутер во всех сервисах** с общей цепочкой middleware: recover, logging, otel, auth, rate-limit, единые ошибки RFC 7807. ТЗ называет chi выбором по умолчанию и echo — допустимой альтернативой.

### Рассмотренные варианты

1. **chi** (выбран, дефолт ТЗ)
   - Плюсы: полная совместимость со стандартным `net/http` (`http.Handler`) — любой сторонний/собственный middleware подключается без адаптеров; идиоматичный, минимальный API; зрелая экосистема middleware; стандартные хендлеры упрощают интеграцию otelhttp; ноль привязки к фреймворку в сигнатурах хендлеров.
   - Минусы: нет встроенных удобств «фреймворка» (биндинг/валидация запросов) — реализуются хелперами в `pkg/shared`.
2. **echo**
   - Плюсы: встроенный биндинг и валидация, богатый набор встроенных middleware, немного меньше boilerplate на хендлер.
   - Минусы: собственный тип контекста (`echo.Context`) вместо стандартного `http.Handler` — привязка к фреймворку во всех хендлерах и middleware; смешение двух роутеров в монорепо запрещено требованием единообразия. **Отклонён.**

### Решение

Во всех сервисах, включая API Gateway, используется **chi**. Стандартная цепочка middleware (recover → request-id → logging → otel → auth → rate-limit) собирается один раз в `pkg/shared` и переиспользуется каждым сервисом. Хелперы биндинга/валидации запросов и рендеринга ошибок RFC 7807 — также в `pkg/shared`.

### Последствия

- Единообразный вид всех HTTP-слоёв: новый сервис поднимается копированием скелета; ревью и онбординг проще.
- Хендлеры остаются стандартными `http.Handler`/`http.HandlerFunc` — минимальная связанность с библиотекой, лёгкое тестирование через `httptest`.
- Недостающие удобства (биндинг, валидация) — небольшая разовая инвестиция в `pkg/shared`.
- **Условия пересмотра:** практически не предвидится; замена роутера возможна лишь при прекращении поддержки chi или критических проблемах производительности (маловероятно — chi это тонкая обёртка над stdlib). Любая замена — только сразу во всех сервисах, единообразие нарушать нельзя.

## ADR-007: Миграции — goose, forward-only

- **Статус:** accepted
- **Дата:** 2026-07-06
- **Автор:** AI-инженер, по согласованию с ТЗ (мастер-промт, разделы 3, 9, 10)

### Контекст

Каждый сервис владеет собственной БД (db-per-service), схемы эволюционируют на протяжении всех глав роадмапа. Нужен единый инструмент миграций с автозапуском при старте в Compose, поддержкой сидов (категории, регионы, каталог услуг, ценообразование) и политикой **forward-only** (откаты в проде запрещены; исправление — новой миграцией вперёд). ТЗ называет goose дефолтом, golang-migrate — альтернативой.

### Рассмотренные варианты

1. **goose** (выбран, дефолт ТЗ)
   - Плюсы: SQL-миграции с up/down в одном файле и понятной нумерацией; поддержка Go-миграций для сложных сидов/трансформаций данных; удобно встраивается в сервис как библиотека (автозапуск миграций при старте контейнера) и работает как CLI; простая модель версионирования.
   - Минусы: менее строгая проверка «грязного» состояния, чем у golang-migrate (компенсируется forward-only-политикой и ревью).
2. **golang-migrate**
   - Плюсы: широкая поддержка источников/БД; строгий dirty-state механизм.
   - Минусы: up/down в отдельных файлах (больше файлового шума); Go-миграции для программных сидов менее удобны; преимущества (много СУБД, много источников) для нашего стека избыточны. **Отклонён.**

### Решение

Во всех сервисах используется **goose** с политикой **forward-only**: down-секции пишутся только для локальной разработки и никогда не выполняются в прод-окружении; деструктивные изменения (DROP, изменение типа) — только с явным планом и через промежуточные совместимые шаги (expand → migrate → contract). Миграции каждого сервиса живут рядом с сервисом, применяются автоматически при старте, проходят обязательное ревью. Сиды оформляются как отдельные миграции (SQL или Go).

### Последствия

- Единый воспроизводимый механизм эволюции всех N баз; `docker compose up` даёт актуальные схемы и сиды без ручных шагов.
- Forward-only дисциплинирует проектирование изменений: обратная совместимость схемы на время выката становится нормой (важно при нескольких репликах сервиса).
- При автозапуске миграций несколькими репликами используется advisory-lock (goose поддерживает), чтобы избежать гонок.
- **Условия пересмотра:** потребность в фичах, которых нет в goose (например, специфичные источники миграций в CI/CD-платформе при переходе на K8s) — тогда сравнение с golang-migrate повторяется отдельным ADR. Политика forward-only пересмотру не подлежит.

## ADR-008: Идентификаторы — UUIDv7 через github.com/google/uuid

- **Статус:** accepted
- **Дата:** 2026-07-06
- **Автор:** AI-инженер, по согласованию с ТЗ (мастер-промт, раздел 6)

### Контекст

Всем сущностям системы (users, ads, media, conversations, payments и т.д.) нужны первичные ключи, которые: генерируются на стороне приложения (важно для outbox — ID известен до коммита и попадает в событие), глобально уникальны между сервисами и БД (db-per-service, будущий шардинг), не раскрывают бизнес-информацию (перечислимость последовательных ID — источник IDOR-разведки) и не разрушают локальность B-tree-индексов при вставке.

### Рассмотренные варианты

1. **UUIDv7 (github.com/google/uuid)** (выбран)
   - Плюсы: старшие биты — Unix-время в миллисекундах → **сортируемость по времени** и монотонная вставка в индекс (нет случайного разброса страниц, как у v4); стандарт RFC 9562; нативный тип `uuid` в PostgreSQL; библиотека google/uuid — де-факто стандарт в Go, поддерживает `uuid.NewV7()`; генерация на клиенте без похода в БД.
   - Минусы: 16 байт против 8 у bigint; временная компонента слегка раскрывает момент создания записи (приемлемо: `created_at` и так публичен в API).
2. **UUIDv4**
   - Плюсы: та же уникальность и та же библиотека.
   - Минусы: случайное распределение по индексу — write-amplification и фрагментация B-tree на больших таблицах (`ads`, `messages`); нет сортируемости по времени. **Отклонён.**
3. **bigserial / последовательности БД**
   - Плюсы: компактность, скорость.
   - Минусы: ID известен только после INSERT (неудобно для outbox); перечислимость (IDOR-разведка, раскрытие объёмов бизнеса); коллизии при слиянии/шардинге данных. **Отклонён.**
4. **ULID/KSUID (сторонние библиотеки)**
   - Плюсы: тоже time-ordered.
   - Минусы: нестандартные типы/кодировки, лишняя зависимость; UUIDv7 даёт то же самое в рамках стандартного `uuid`-типа. **Отклонены.**

### Решение

Все первичные ключи — **UUIDv7**, генерация через `github.com/google/uuid` (`uuid.Must(uuid.NewV7())` c обработкой ошибки в инфраструктурном коде), хранение — нативный тип `uuid` в PostgreSQL. Генерация выполняется в сервисном слое до транзакции, чтобы ID был доступен для outbox-события и ответа API.

### Последствия

- Единая идентификация во всех сервисах и событиях; cursor-пагинация по `id` получает естественный хронологический порядок.
- Индексы по PK ведут себя как при монотонных ключах — предсказуемая производительность вставки в горячие таблицы.
- Версия библиотеки фиксируется в `pkg/shared` (обёртка `NewID()`), чтобы все сервисы генерировали ID единообразно.
- **Условия пересмотра:** практически не предвидится. Точечный пересмотр возможен для сверхгорячих таблиц (например, `messages`) в пользу составных ключей при партиционировании — отдельным ADR.

## ADR-009: Деньги — int64 в тийинах, валюта хранится явно

- **Статус:** accepted
- **Дата:** 2026-07-06
- **Автор:** AI-инженер, по согласованию с ТЗ (мастер-промт, разделы 6, Приложение B)

### Контекст

Деньги присутствуют в нескольких сервисах: цена объявления (Listing), прайс платных услуг (`pricing`), кошелёк и леджер транзакций двойной записи (Payments), суммы в интеграциях Payme/Click. Ошибки округления в деньгах недопустимы, особенно в леджере, где суммы проводок обязаны сходиться до единицы. Валюта площадки — UZS; тийин — минимальная единица (1 сум = 100 тийинов). Payme в своём API также оперирует суммами в тийинах.

### Рассмотренные варианты

1. **int64 в минимальных единицах (тийинах) + явное поле `currency`** (выбран)
   - Плюсы: точная целочисленная арифметика без ошибок представления; прямое соответствие формату Payme; `BIGINT` в PostgreSQL — компактно и быстро; диапазона int64 хватает с запасом (~9.2×10^18 тийинов); явная валюта исключает неоднозначность трактовки суммы и закладывает мультивалютность «на вырост».
   - Минусы: конвертация тийины↔сумы на границах (API/отображение, Click оперирует суммами в сумах) — нужны хелперы; риск ошибок «сумма в чём?» — купируется типом-обёрткой и конвенцией именования.
2. **NUMERIC/DECIMAL в БД + decimal-библиотека в Go**
   - Плюсы: «человеческие» суммы в БД; произвольная точность.
   - Минусы: в Go нет стандартного decimal — сторонняя зависимость во всех сервисах; медленнее и тяжелее BIGINT; риск случайной float-конвертации на границах сериализации. **Отклонён.**
3. **float64**
   - Минусы: двоичное представление не хранит десятичные доли точно; для денег недопустим. **Отклонён безусловно.**

### Решение

Все денежные суммы хранятся как **int64 в тийинах** (минимальных единицах UZS), рядом всегда — явное поле **`currency`** (в v1 значение `"UZS"`, CHECK-ограничение). В Go вводится тип `Money { Amount int64; Currency string }` в `pkg/shared` с конвертацией в/из сумов только на границах (рендеринг, интеграция Click). Поля в БД и JSON именуются с суффиксом, снимающим неоднозначность (например, `amount_tiyin`). Леджер кошелька оперирует только тийинами.

### Последствия

- Арифметика денег точна во всех сервисах; сверка леджера — целочисленная.
- Интеграция Payme прямая (тийины), для Click — конвертация в хелпере адаптера.
- Пропорциональный расчёт возвратов (неиспользованные дни услуги) выполняется в тийинах с явно зафиксированным правилом округления (в пользу пользователя — уточняется в ARCHITECTURE.md при реализации CHAPTER 8).
- **Условия пересмотра:** появление мультивалютности (потребует таблицы валют с числом минимальных единиц) или деноминация UZS. Само представление int64+currency при этом сохраняется — меняется только справочник валют.

## ADR-010: Логирование — стандартный log/slog (JSON), без zerolog

- **Статус:** accepted
- **Дата:** 2026-07-06
- **Автор:** AI-инженер, по согласованию с ТЗ (мастер-промт, разделы 8, 10)

### Контекст

Требования к логированию: структурированный JSON во всех сервисах, обязательные поля trace-id/correlation-id (request-id), уровни, единый формат для сбора в Loki и корреляции с трейсами Tempo. RULES.md допускает slog или zerolog; нужен один вариант для всего монорепо.

### Рассмотренные варианты

1. **log/slog (stdlib)** (выбран)
   - Плюсы: **стандартная библиотека** — ноль внешних зависимостей у самого базового инструмента, который импортирует каждый пакет; `JSONHandler` из коробки; расширяемость через собственный `Handler` (автодобавление trace_id/request_id из контекста); slog стал точкой интеграции экосистемы (мосты для otel, middleware); API стабилен как часть Go.
   - Минусы: несколько медленнее zerolog на синтетических бенчмарках (аллокации на атрибуты); чуть более многословный API.
2. **zerolog**
   - Плюсы: один из самых быстрых JSON-логгеров (zero-allocation), fluent API.
   - Минусы: внешняя зависимость в каждом сервисе ради базовой функции; свой несовместимый API — привязка всего кода к библиотеке; экосистема Go консолидируется вокруг slog-интерфейса. **Отклонён.**

Производительность логгера не является узким местом Bozor: объём логов на запрос умеренный, а разница в наносекундах на запись несопоставима с сетевыми и БД-задержками.

### Решение

Во всех сервисах используется **log/slog** с `JSONHandler` (вывод в stdout — 12-factor, сбор через Loki). В `pkg/shared/logging` создаётся преднастроенный логгер: обязательные поля `service`, `env`, `version`; кастомный Handler добавляет `trace_id`, `span_id`, `request_id` из `context.Context`; уровень задаётся через env. Прямое использование `fmt.Print`/`log` в сервисном коде запрещено линтером.

### Последствия

- Меньше зависимостей: базовый слой логирования не тянет сторонних библиотек; обновляется вместе с Go.
- Единый формат логов гарантирует работоспособность корреляции логи↔трейсы в Grafana из коробки.
- Весь код логирует через `*slog.Logger` из контекста/DI — замена Handler (например, на семплирующий) не требует правок в бизнес-коде.
- **Условия пересмотра:** только доказанная профилировкой деградация от логгера на горячем пути (крайне маловероятно); в этом случае сначала рассматривается семплирование/снижение уровня, и лишь затем — альтернативный Handler-бэкенд под тем же slog-API (без смены API — это преимущество slog).

## ADR-011: Монорепо с go.work, отдельный go.mod на сервис + общий модуль pkg/shared

- **Статус:** accepted
- **Дата:** 2026-07-06
- **Автор:** AI-инженер, по согласованию с ТЗ (мастер-промт, раздел 10)

### Контекст

ТЗ фиксирует монорепо со структурой `/services/<name>`, `/pkg/shared`, `/api`, `/deploy`, `/scripts`, `/docs`. Открытым остаётся вопрос модульной организации Go-кода: один общий `go.mod` на весь репозиторий или отдельные модули. От этого зависят изоляция зависимостей, размер Docker-образов, скорость CI и удобство локальной разработки (13 сервисов + общая библиотека).

### Рассмотренные варианты

1. **go.work + отдельный go.mod на каждый сервис + отдельный модуль `pkg/shared`** (выбран)
   - Плюсы: **изоляция зависимостей** — тяжёлые библиотеки одного сервиса (Telegram Bot API у Auth/Notification, Typesense-клиент у Search, платёжные SDK у Payments) не попадают в `go.sum` и граф сборки остальных; независимые graceful-обновления зависимостей по сервисам; Docker-сборка каждого сервиса видит только свой модуль + shared (меньше кеш-инвалидаций); `go.work` даёт бесшовную локальную разработку — правки в `pkg/shared` немедленно видны всем сервисам без `replace`-директив; общий код версионируется вместе с сервисами в одном репозитории (атомарные PR через границы модулей).
   - Минусы: больше файлов `go.mod`/`go.sum`; CI и Makefile должны уметь итерироваться по модулям (`go work` / список модулей); в Docker-сборке нужно копировать `pkg/shared` рядом с сервисом.
2. **Единый go.mod на весь монорепозиторий**
   - Плюсы: максимально просто — одна команда `go build ./...`, один граф зависимостей.
   - Минусы: общий `go.sum` — зависимость любого сервиса становится зависимостью всех (раздутые образы при неаккуратной сборке, лишние срабатывания govulncheck, конфликт версий одной библиотеки между сервисами неразрешим); обновление зависимости затрагивает весь репозиторий сразу. **Отклонён.**
3. **Полирепо (репозиторий на сервис)**
   - Плюсы: предельная изоляция.
   - Минусы: противоречит ТЗ (монорепо); версионирование `pkg/shared` через теги и прокси — постоянные накладные расходы; атомарные изменения контракта + потребителей невозможны в одном PR. **Отклонён.**

### Решение

Принята схема: **корневой `go.work`**, перечисляющий все модули; **отдельный `go.mod`** в каждом `services/<name>`; **отдельный модуль `pkg/shared`** (config, logging, errors, otel, auth-middleware, http/grpc-хелперы, outbox). Внутри репозитория сервисы ссылаются на shared по имени модуля, разрешение — через workspace (без `replace`). CI линтует/тестирует каждый модуль; Docker-сборка сервиса копирует `pkg/shared` и свой каталог. `go.work` в образы не попадает (используется `GOFLAGS=-mod=mod` либо `go work vendor`-подход — фиксируется в RULES.md при реализации Stage 0.1).

### Последствия

- Зависимости сервисов не «протекают» друг в друга; аудит зависимостей (govulncheck) точен по каждому сервису.
- Изменение `pkg/shared` — это осознанное событие: CI прогоняет тесты всех сервисов-потребителей; ломающие правки shared видны в том же PR.
- Makefile получает цели уровня модуля (`lint-<svc>`, `test-<svc>`) и агрегирующие цели по workspace.
- Порог входа чуть выше (нужно понимать go.work), компенсируется документацией в README/RULES.md.
- **Условия пересмотра:** если накладные расходы на мульти-модульность окажутся выше пользы при малом числе реально различающихся зависимостей — допустим переход на единый go.mod (простая механическая миграция). Выделение `pkg/shared` в отдельный репозиторий — только при появлении внешних потребителей.

## ADR-012: JWT — github.com/golang-jwt/jwt/v5; access ~15 мин, refresh ~30 дней с ротацией, reuse-detection и family-отзывом

- **Статус:** accepted
- **Дата:** 2026-07-06
- **Автор:** AI-инженер, по согласованию с ТЗ (мастер-промт, разделы 3, 5.1, 8)

### Контекст

После Telegram-авторизации (ADR-003) Auth-сервис выпускает токены, которые проверяются на API Gateway для каждого запроса (stateless-проверка, без похода в Auth). Требования ТЗ: короткоживущий access-JWT (claims `sub`, `roles`, `exp`, `iat`, `jti`), долгоживущий refresh с хранением в виде хеша, ротацией при каждом использовании, привязкой к `device_id` и **reuse-detection** (повторное использование уже отозванного refresh — признак кражи → отзыв всего семейства токенов устройства).

Нужно выбрать библиотеку JWT и зафиксировать параметры жизненного цикла токенов.

### Рассмотренные варианты

**Библиотека:**

1. **github.com/golang-jwt/jwt/v5** (выбрана)
   - Плюсы: де-факто стандарт в Go-экосистеме (поддерживаемый форк dgrijalva/jwt-go); v5 — актуальный API с строгой валидацией (`WithValidMethods`, обязательная проверка `exp`); широкая совместимость примеров/middleware; достаточна для нашего сценария (подпись и верификация компактных JWT).
   - Минусы: только JWS-примитивы — сессионную логику (ротация, семейства) пишем сами (мы и так пишем её по требованиям ТЗ).
2. **lestrrat-go/jwx**
   - Плюсы: полный набор JOSE (JWE, JWK, key sets).
   - Минусы: избыточен — шифрование токенов и JWKS-ротация ключей в v1 не требуются; тяжелее API. **Отклонён (кандидат при переходе на JWKS).**
3. **Opaque-токены (случайные строки + проверка в Redis/БД)**
   - Плюсы: мгновенный отзыв access.
   - Минусы: каждый запрос требует похода в хранилище — теряется stateless-проверка на gateway, добавляется задержка и точка отказа; противоречит выбору «JWT access» в ТЗ. **Отклонён для access; для refresh фактически используется (см. решение).**

**Параметры (варьировались TTL):** более короткий access (5 мин) увеличивает нагрузку на refresh-эндпоинт; более длинный (60 мин) увеличивает окно жизни украденного access и задержку применения бана. 15 минут — баланс, предложенный ТЗ. Refresh 30 дней соответствует ожиданию «вход раз в месяц» для classifieds-аудитории.

### Решение

- Библиотека — **github.com/golang-jwt/jwt/v5**; подпись HS256 с секретом из env в v1 (один издатель — Auth; gateway получает ключ верификации через env). Переход на RS256/EdDSA + JWKS — задокументированная эволюция при появлении внешних верификаторов.
- **Access-токен:** JWT, TTL **~15 минут**, claims: `sub` (user_id), `roles`, `exp`, `iat`, `jti`. Проверяется на gateway; `user_id`/roles пробрасываются во внутренние вызовы заголовками/метаданными.
- **Refresh-токен:** непрозрачная криптослучайная строка (не JWT), TTL **~30 дней**; в БД (`refresh_tokens`) хранится только хеш (SHA-256), вместе с `user_id`, `device_id`, `family_id`, `expires_at`, `revoked_at`.
- **Ротация:** каждый `POST /auth/refresh` гасит использованный refresh и выдаёт новую пару (access + refresh) в том же `family_id`.
- **Reuse-detection:** предъявление refresh-токена, который уже был использован/отозван, трактуется как компрометация → **отзывается всё семейство** (`family_id`) токенов данного устройства, событие пишется в `auth_audit_log`.
- **Logout:** отзыв семейства текущего устройства; ролевые изменения/бан применяются не позднее истечения access-TTL (≤15 мин), при необходимости мгновенного бана gateway дополнительно сверяет короткий denylist в Redis по `jti` (включается при реализации банов в CHAPTER 6).

### Последствия

- Gateway проверяет access локально — нет синхронной зависимости от Auth на каждом запросе; Auth масштабируется независимо.
- Окно действия отозванных прав ограничено 15 минутами — приемлемо для v1 и явно задокументировано; для банов предусмотрен denylist-механизм.
- Схема refresh (хеш + семейства + ротация) устойчива к утечке БД (хранится только хеш) и к краже refresh-токена (reuse-detection ловит параллельное использование).
- Реализация семейства/ротации покрывается обязательными тестами (включая сценарий reuse) — входит в DoD Stage 1.4.
- **Условия пересмотра:** появление внешних потребителей верификации токенов или нескольких издателей → переход на асимметричную подпись (RS256/EdDSA) с JWKS (возможна смена библиотеки на jwx) — отдельным ADR; изменение TTL — по данным эксплуатации (нагрузка на refresh, инциденты безопасности).

## ADR-013: Пул соединений — один PostgreSQL-инстанс + PgBouncer в transaction-режиме с prepared statements

- **Статус:** accepted
- **Дата:** 2026-07-07
- **Автор:** AI-инженер, по согласованию с ТЗ (мастер-промт, разделы 3, 10, Приложение B)

### Контекст

Схема db-per-service (ADR-004, инвариант ТЗ) требует изолированной БД на каждый из ~12 сервисов. На одном Compose-хосте разворачивать 12 отдельных инстансов PostgreSQL расточительно, поэтому используется **один инстанс с отдельной базой на сервис** (init-скрипт создаёт `bozor_auth`, `bozor_catalog`, …). Каждый сервис имеет несколько реплик и держит собственный pgxpool; при десятках тысяч DAU суммарное число прямых соединений к PostgreSQL легко превышает комфортные `max_connections`. Нужен внешний пулер.

Ключевой нюанс: pgx v5 по умолчанию работает в режиме `cache_statement` — кеширует **серверные** prepared statements по имени. В PgBouncer с `pool_mode = transaction` серверное соединение возвращается в пул после каждой транзакции, поэтому именованный statement, подготовленный в одной транзакции, в следующей уже на другом бэкенде — классическая ошибка `prepared statement "stmtcache_..." does not exist`. Исторически это вынуждало переключать pgx в `simple_protocol`/`exec` (теряя протокол-биндинг и часть производительности) либо использовать session-режим пула (убивает мультиплексирование).

### Рассмотренные варианты

1. **Один инстанс + PgBouncer `transaction` + `max_prepared_statements > 0`, pgx на дефолтном `cache_statement`** (выбран)
   - Плюсы: PgBouncer >= 1.21 умеет отслеживать протокол-левел prepared statements в transaction-режиме (`max_prepared_statements`), поэтому дефолтное поведение pgx работает **без изменений в коде сервисов**; сохраняется максимальное мультиплексирование клиентов (`MAX_CLIENT_CONN=1000` поверх небольшого `DEFAULT_POOL_SIZE`); никакой потери типобезопасности/производительности extended-протокола. Наш образ — `edoburu/pgbouncer:v1.24.1` (проверено: env `MAX_PREPARED_STATEMENTS` рендерится в `pgbouncer.ini`).
   - Минусы: небольшой расход памяти PgBouncer на кеш дескрипторов statement'ов; зависимость от версии PgBouncer >= 1.21.
2. **pgx в `simple_protocol`/`exec`-режиме, PgBouncer `transaction` без prepared statements**
   - Плюсы: работает с любым (в т.ч. старым) PgBouncer.
   - Минусы: `simple_protocol` теряет протокол-биндинг параметров (pgx интерполирует их клиентски) и серверный кеш планов; `exec` шлёт describe на каждый запрос. Ненужный компромисс при доступном PgBouncer 1.24. **Отклонён.**
3. **PgBouncer в `session`-режиме**
   - Плюсы: prepared statements работают «из коробки».
   - Минусы: соединение закреплено за клиентом на всю сессию — мультиплексирование теряется, число серверных соединений ≈ числу клиентских, что обесценивает пулер под нашей нагрузкой. **Отклонён.**
4. **Отдельный инстанс PostgreSQL на сервис**
   - Плюсы: максимальная изоляция ресурсов/отказов.
   - Минусы: 12 инстансов на одном хосте — избыточный оверхед по RAM/CPU/эксплуатации; db-per-service уже даёт логическую изоляцию. **Отклонён** (пересматривается при выносе горячих БД на отдельные хосты).

### Решение

Один PostgreSQL-инстанс с базой на сервис; перед ним **PgBouncer в `pool_mode = transaction`** с `MAX_PREPARED_STATEMENTS=200`, `MAX_CLIENT_CONN=1000`, `DEFAULT_POOL_SIZE=20`, `AUTH_TYPE=scram-sha-256`. Сервисы подключаются через PgBouncer (порт 6432) и оставляют pgx на дефолтном `cache_statement`. Все прикладные соединения идут только через пулер; прямой доступ к 5432 — лишь для миграций/обслуживания.

### Последствия

- Код репозиториев (ADR-005) не требует спец-настройки exec-режима pgx — на один класс ошибок меньше; при этом сохраняется мультиплексирование под высокой конкуррентностью.
- Миграции goose (ADR-007) выполняются в отдельной транзакции/через прямое соединение к 5432, чтобы DDL и advisory-lock не конфликтовали с transaction-пулером.
- Значения пула (`DEFAULT_POOL_SIZE`, `max_connections=200` на самом PostgreSQL) — стартовые; пересматриваются на Stage 10.3 по данным нагрузочного теста.
- **Условия пересмотра:** вынос отдельной БД на выделенный инстанс/хост при её росте; отказ от prepared statements в пользу `exec`-режима, если понадобится совместимость со старым PgBouncer или внешним пулером без их поддержки.
