# Bozor — Журнал архитектурных решений (ADR)

Этот документ — единственный журнал архитектурных решений проекта **Bozor** (backend маркетплейса объявлений на Go, микросервисы). Каждое значимое техническое решение фиксируется здесь в формате ADR (Architecture Decision Record) и, если оно критично для системы, отражается в `ARCHITECTURE.md`.

## Формат записи

Каждый ADR оформляется по следующей схеме:

- **ADR-NNN: Заголовок** — краткая формулировка решения.
- **Статус:** `proposed` | `accepted` | `deprecated` | `superseded by ADR-NNN`.
- **Дата:** дата принятия решения (YYYY-MM-DD).
- **Автор:** кто принял решение и на каком основании.
- **Контекст:** какая проблема решается, какие ограничения и требования влияют на выбор.
- **Рассмотренные варианты:** перечень альтернатив с плюсами и минусами.
- **Решение:** что выбрано и почему.
- **Последствия:** что это означает для проекта — позитивные и негативные эффекты, а также **условия пересмотра** (при каких обстоятельствах решение нужно вернуть на обсуждение).

Нумерация сквозная (`ADR-001`, `ADR-002`, …). Принятые ADR не редактируются задним числом: если решение меняется, создаётся новый ADR со ссылкой на устаревший (`superseded by`).

## ADR-001: Брокер событий — NATS JetStream, а не Kafka

- **Статус:** accepted
- **Дата:** 2026-07-06
- **Автор:** AI-инженер, по согласованию с ТЗ (мастер-промт, раздел 3)

### Контекст

Архитектура Bozor событийно-ориентированная: все межсервисные интеграции (индексация поиска, уведомления, модерация, матчинг сохранённых поисков, обработка платежей) строятся на доменных событиях `bozor.<domain>.<event>`, публикуемых через transactional outbox. Нужен брокер с персистентностью (at-least-once), поддержкой durable-консьюмеров, ack/redelivery и DLQ.

Ключевое ограничение — развёртывание на **одном хосте под Docker Compose**. Ресурсный след инфраструктурных компонентов должен быть минимальным, чтобы на одном хосте помещались все прикладные сервисы, PostgreSQL, Redis, Typesense, MinIO и стек наблюдаемости. Целевая нагрузка (десятки тысяч DAU) не создаёт экстремального throughput по событиям.

### Рассмотренные варианты

1. **NATS JetStream**
   - Плюсы: один лёгкий бинарник, минимальное потребление CPU/RAM; персистентные stream'ы, durable/pull-консьюмеры, ack, redelivery, max-deliver + DLQ-паттерн; иерархические subjects идеально ложатся на схему `bozor.<domain>.<event>`; простая эксплуатация в Compose; отличный Go-клиент.
   - Минусы: экосистема (коннекторы, инструменты) беднее, чем у Kafka; меньше опыта индустрии на экстремальных объёмах; нет такого богатого тулинга для реплея/аналитики потоков.
2. **Apache Kafka**
   - Плюсы: индустриальный стандарт для высоких throughput; огромная экосистема (Kafka Connect, Schema Registry, ksqlDB); проверенные гарантии упорядоченности по партициям; долгое хранение и реплей.
   - Минусы: тяжёлый ресурсный след (JVM, брокеры + контроллер), заметно усложняет один Compose-хост; операционная сложность (партиции, ребалансировки, тюнинг) избыточна для текущей нагрузки.
3. **Redis Streams** (не как основной кандидат)
   - Плюсы: Redis уже есть в стеке; минимальная дополнительная инфраструктура.
   - Минусы: слабее гарантии долговременной персистентности; consumer groups менее удобны для схемы многих независимых доменных потоков; смешивание ролей кеша и брокера — операционный риск.

### Решение

Выбран **NATS JetStream** как единый брокер доменных событий. Публикация — только через transactional outbox (relay-воркер читает `outbox` и публикует в JetStream), потребители идемпотентны, «ядовитые» сообщения после исчерпания max-deliver уходят в DLQ-stream.

### Последствия

- Инфраструктура остаётся лёгкой: весь стек уверенно помещается на один Compose-хост.
- Код прикладных сервисов не должен зависеть от специфики брокера: публикация/подписка инкапсулируются в `pkg/shared` (тонкий слой поверх клиента), чтобы замена брокера не расползалась по кодовой базе.
- Semantика доставки — at-least-once; дедупликация обязана обеспечиваться на стороне консьюмеров (идемпотентность по `event_id`).
- **Условия пересмотра:** устойчивый рост throughput событий за пределы комфортной работы JetStream на выделенном хосте (сотни тысяч событий/сек), потребность в длительном хранении и реплее потоков для аналитики, потребность в экосистеме Kafka Connect. В этом случае **Kafka — задокументированная замена**; путь миграции: перевод relay-воркера и консьюмеров на Kafka-клиент при сохранении envelope и именования событий.

## ADR-002: Поисковый движок — Typesense, а не OpenSearch / PostgreSQL FTS

- **Статус:** accepted
- **Дата:** 2026-07-06
- **Автор:** AI-инженер, по согласованию с ТЗ (мастер-промт, раздел 3)

### Контекст

Поиск — центральный read-путь площадки: полнотекстовый поиск на узбекском и русском с typo-tolerance, фасетные фильтры по динамическим атрибутам категорий, сортировки (релевантность, цена, дата, `bumped_at`, топ-блок), гео-поиск по координатам и пагинация. Поисковый индекс — это read-модель (CQRS-lite), которая строится из событий Listing-сервиса.

Ограничения те же, что и в ADR-001: один Compose-хост, минимальный ресурсный след, при этом p95 поиска целево < 200–300 мс.

### Рассмотренные варианты

1. **Typesense**
   - Плюсы: очень лёгкий (C++, один бинарник, in-memory индекс) — комфортно живёт на общем хосте; фасеты, typo-tolerance и гео-поиск **из коробки** без тюнинга; простой API и предсказуемая низкая латентность; простая эксплуатация.
   - Минусы: индекс в RAM — объём данных ограничен памятью; нет сложных агрегаций/аналитики (как у Elasticsearch aggregations); меньше гибкости анализаторов для морфологии; кластерные возможности скромнее.
2. **OpenSearch / Elasticsearch**
   - Плюсы: богатейшие возможности — сложные агрегации, кастомные анализаторы (морфология русского), percolate-запросы (полезно для сохранённых поисков), зрелое кластерирование.
   - Минусы: JVM, большой ресурсный след (гигабайты RAM только на движок), операционная сложность (шарды, реплики, тюнинг heap) — тяжело для одного Compose-хоста.
3. **PostgreSQL FTS (tsvector/pg_trgm)**
   - Плюсы: нулевая дополнительная инфраструктура; транзакционная консистентность с данными.
   - Минусы: слабая typo-tolerance; фасетирование по динамическим атрибутам — тяжёлые GROUP BY на нагруженной БД; смешивает поисковую нагрузку с OLTP-нагрузкой основной БД, что противоречит цели разгрузки PostgreSQL; нет удобного гео-ранжирования «из коробки».

### Решение

Выбран **Typesense** как поисковый движок для коллекции `ads`. Индексация — асинхронная, по событиям `bozor.ad.*` (outbox → consumer → Typesense); обязателен инструмент полной переиндексации (reindex) из БД Listing-сервиса.

### Последствия

- Быстрый старт: фасеты, typo и гео работают без длительного тюнинга; поисковая нагрузка снята с PostgreSQL.
- Так как у Typesense нет percolate, матчинг сохранённых поисков реализуется собственным matcher-компонентом (см. спецификацию Favorites/Saved-Search); это осознанная плата за лёгкость движка.
- Объём индекса нужно контролировать (только активные объявления + необходимые поля), т.к. индекс держится в памяти.
- Слой поиска изолируется за интерфейсом в Search-сервисе, чтобы замена движка не затронула публичный API `GET /api/v1/ads/search`.
- **Условия пересмотра:** потребность в сложных агрегациях/аналитике по объявлениям, продвинутой морфологии uz/ru, либо рост индекса за пределы доступной RAM. В этом случае **OpenSearch — задокументированная замена**; путь миграции: реализация второго индексатора и переключение read-пути после полной переиндексации.

## ADR-003: Авторизация — Telegram-бот с request_contact + nonce/deep-link

- **Статус:** accepted
- **Дата:** 2026-07-06
- **Автор:** AI-инженер, по согласованию с ТЗ (мастер-промт, раздел 5.1)

### Контекст

В v1 у Bozor **нет фронтенда** — есть только API. При этом требуется регистрация и вход по номеру телефона, а канал взаимодействия с пользователем — Telegram (он же используется для уведомлений). Нужен flow, который: (а) работает без веб-страницы, (б) надёжно подтверждает владение номером телефона, (в) позволяет клиенту API (в будущем — мобильному/веб-приложению) получить токены после подтверждения в боте.

Telegram предоставляет несколько механизмов аутентификации, каждый со своими предпосылками (наличие сайта, наличие Mini App), поэтому выбор влияет на всю архитектуру Auth-сервиса.

### Рассмотренные варианты

1. **Бот + `request_contact` + nonce/deep-link** (выбран)
   - Плюсы: работает полностью без фронтенда — клиент вызывает `POST /auth/session/init`, получает `nonce` и deep-link `https://t.me/<bot>?start=<nonce>`, пользователь подтверждает номер кнопкой `request_contact`, клиент опрашивает статус сессии; бот и так нужен для уведомлений; Telegram сам подтверждает номер (контакт приходит из аккаунта); nonce (TTL 2–5 мин, single-use, Redis) криптографически связывает клиентскую сессию с Telegram-пользователем.
   - Минусы: собственный протокол связывания (nonce + polling), который нужно аккуратно реализовать; критичная проверка `message.contact.user_id == message.from.id` против пересылки чужого контакта; зависимость доступности входа от Telegram Bot API.
2. **Telegram Login Widget**
   - Плюсы: стандартный, документированный механизм с HMAC-проверкой данных.
   - Минусы: требует **веб-страницу** с виджетом — фронтенда нет; не отдаёт номер телефона (только Telegram ID и имя), а вход по номеру — требование ТЗ. **Отклонён.**
3. **Mini App `initData`**
   - Плюсы: криптографически подписанные данные пользователя, современный UX внутри Telegram.
   - Минусы: требует разработки Mini App (то есть фронтенда, пусть и встроенного в Telegram) — вне scope v1; номер телефона всё равно нужно запрашивать отдельно. **Отклонён.**

### Решение

Реализуется flow «бот + `request_contact` + nonce/deep-link» строго по разделу 5.1 ТЗ: init-сессия с nonce → deep-link в бота → кнопка отправки контакта → webhook с проверкой `X-Telegram-Bot-Api-Secret-Token` и владения контактом → связывание с nonce → выдача пары JWT (см. ADR-012) → клиент получает токены поллингом `GET /auth/session/{nonce}`.

### Последствия

- Auth-сервис самодостаточен и тестируем без какого-либо фронтенда; будущие клиенты (mobile/web) используют тот же init/poll-протокол.
- Обязательные меры безопасности становятся частью DoD: nonce single-use + TTL, secret_token вебхука, проверка `contact.user_id == from.id`, нормализация телефона в E.164, rate limiting на `init` и webhook, аудит в `auth_audit_log`.
- Появляется операционная зависимость от Telegram Bot API — при его недоступности вход невозможен (принимаем: уведомления в v1 всё равно только через Telegram).
- **Условия пересмотра:** появление веб-фронтенда (можно добавить Login Widget как дополнительный способ входа), появление Mini App (добавить проверку `initData`), либо требование альтернативных каналов входа (SMS OTP через местных провайдеров). Новые способы входа добавляются рядом, не ломая существующий flow.

## ADR-004: Catalog и Location — отдельные микросервисы

- **Статус:** accepted
- **Дата:** 2026-07-06
- **Автор:** AI-инженер, по согласованию с ТЗ (мастер-промт, разделы 5.3, 5.6)

### Контекст

Catalog (дерево категорий + динамические атрибуты) и Location (регионы/города Узбекистана) — справочные сервисы: данные меняются редко, читаются часто, агрессивно кешируются. ТЗ явно допускает объединение их в один «reference»-сервис на старте с требованием зафиксировать выбор в этом журнале.

Аргументы «за объединение» очевидны: меньше процессов, меньше БД, меньше деплой-юнитов. Нужно решить, перевешивают ли они чистоту границ.

### Рассмотренные варианты

1. **Два отдельных сервиса: Catalog и Location** (выбран)
   - Плюсы: чистый паттерн database-per-service без компромиссов с первого дня; домены действительно разные — у Catalog нетривиальная логика (наследование атрибутов по дереву, админский CRUD, инвалидация кеша), у Location почти статика; независимая эволюция и деплой; Go-сервисы дёшевы по ресурсам (десятки МБ RAM на инстанс), стоимость лишнего контейнера пренебрежима; структура сервисов 1:1 соответствует спецификации ТЗ (раздел 5) и топологии.
   - Минусы: чуть больше boilerplate (два скелета, две БД, две записи в Compose); два справочных клиента у потребителей вместо одного.
2. **Единый reference-сервис (Catalog + Location)** — допустимая альтернатива по ТЗ
   - Плюсы: меньше инфраструктурных единиц на старте; один кеш-слой и один клиент.
   - Минусы: смешение доменов в одной БД и кодовой базе; будущее разделение (если Catalog обрастёт логикой) — болезненная миграция с разрезанием БД; ломается симметрия «сервис из раздела 5 ТЗ = деплой-юнит».

### Решение

Catalog и Location реализуются как **два отдельных микросервиса**, каждый со своей БД (db-per-service), своим кешем (Redis, cache-aside с инвалидацией по событию) и своим публичным API. Общий каркас (скелет сервиса, middleware, кеш-хелперы) выносится в `pkg/shared`, что снижает цену дублирования.

### Последствия

- Границы сервисов совпадают со спецификацией ТЗ — навигация по проекту и по SPMS-документам однозначна.
- Немного больше стартовой работы в CHAPTER 2 (два скелета вместо одного); компенсируется шаблонностью справочных сервисов.
- Потребители (Listing, Search, gateway) обращаются к двум справочным API; для валидации атрибутов Listing использует Catalog, для валидации гео — Location.
- **Условия пересмотра:** если эксплуатация покажет, что накладные расходы на два процесса ощутимы для хоста (маловероятно для Go), или оба сервиса останутся тривиальными CRUD без собственной логики — допустимо слияние в reference-сервис отдельным ADR. Обратное (разделение) уже не потребуется — оно сделано сейчас.

## ADR-005: Доступ к БД — pgx + чистые (hand-written) репозитории, без sqlc

- **Статус:** accepted
- **Дата:** 2026-07-06
- **Автор:** AI-инженер, по согласованию с ТЗ (мастер-промт, разделы 3, 10)

### Контекст

ТЗ фиксирует PostgreSQL + драйвер **pgx** и допускает два стиля работы с запросами: генерация типизированного кода через **sqlc** либо чистые репозитории, написанные вручную. Нужно выбрать единый стиль для всех сервисов.

Определяющий фактор — характер запросов. Значительная часть критичных запросов Bozor **динамическая**: листинг объявлений (`GET /api/v1/ads`, `GET /api/v1/me/ads`) с произвольным набором фильтров (статус, категория, регион, диапазон цены), сортировок и cursor-пагинацией; админские выборки модерации; выборки промо. sqlc силён на статических запросах, но динамическое построение WHERE/ORDER BY в нём неудобно (плодятся near-duplicate запросы или запросы с частоколом `CASE WHEN $n IS NULL`).

### Рассмотренные варианты

1. **pgx + чистые репозитории** (выбран)
   - Плюсы: полный контроль над динамическим построением запросов (фильтры листинга, фасетные выборки); нет шага codegen в сборке; проще транзакционные сценарии (outbox в той же транзакции, что и доменная запись); меньше инструментов в цепочке; сканирование удобно через `pgx.CollectRows`/`RowToStructByName`.
   - Минусы: типобезопасность запросов не проверяется на этапе компиляции — нужна дисциплина: только параметризованные запросы, обязательные интеграционные тесты репозиториев на testcontainers-go; больше ручного кода на простых CRUD.
2. **sqlc**
   - Плюсы: типизированный код из SQL, ошибки запросов ловятся на этапе генерации; единообразие; меньше рутины на статических запросах.
   - Минусы: динамические фильтры листинга — главный болевой пункт; дополнительный шаг codegen и его конфигурация в каждом сервисе; сгенерированный код хуже сочетается с ручным управлением транзакциями для outbox-паттерна.
3. **ORM (GORM/ent)** — вне рекомендаций ТЗ
   - Минусы: скрытые запросы и неявное поведение противоречат требованиям производительности и явного контроля; ТЗ ORM не предлагает. **Отклонён без детальной проработки.**

### Решение

Во всех сервисах используется **pgx (pgxpool) + чистые репозитории**. Обязательные правила: только параметризованные запросы; репозитории принимают `context.Context` и абстракцию транзакции (для атомарной записи домена + outbox); каждый репозиторий покрывается интеграционными тестами на testcontainers-go; общие хелперы (построитель условий, пагинация, сканирование) — в `pkg/shared`.

### Последствия

- Гибкость на динамических запросах достигается без обходных приёмов; транзакционный outbox реализуется прямолинейно.
- Цена — дисциплина ревью SQL и обязательные интеграционные тесты (это и так входит в DoD по RULES.md).
- Отсутствие codegen упрощает Makefile/CI.
- **Условия пересмотра:** решение не «всё или ничего» — **sqlc можно вводить точечно** в сервисах/пакетах с преимущественно статическими запросами (например, Location, Reviews), если объём ручного CRUD-кода станет обременительным. Такое частичное внедрение оформляется отдельным ADR с указанием границ применения.

## ADR-006: HTTP-роутер — chi

- **Статус:** accepted
- **Дата:** 2026-07-06
- **Автор:** AI-инженер, по согласованию с ТЗ (мастер-промт, разделы 3, 10)

### Контекст

Все сервисы Bozor выставляют REST/JSON API (внешние — через gateway), и ТЗ требует **единый роутер во всех сервисах** с общей цепочкой middleware: recover, logging, otel, auth, rate-limit, единые ошибки RFC 7807. ТЗ называет chi выбором по умолчанию и echo — допустимой альтернативой.

### Рассмотренные варианты

1. **chi** (выбран, дефолт ТЗ)
   - Плюсы: полная совместимость со стандартным `net/http` (`http.Handler`) — любой сторонний/собственный middleware подключается без адаптеров; идиоматичный, минимальный API; зрелая экосистема middleware; стандартные хендлеры упрощают интеграцию otelhttp; ноль привязки к фреймворку в сигнатурах хендлеров.
   - Минусы: нет встроенных удобств «фреймворка» (биндинг/валидация запросов) — реализуются хелперами в `pkg/shared`.
2. **echo**
   - Плюсы: встроенный биндинг и валидация, богатый набор встроенных middleware, немного меньше boilerplate на хендлер.
   - Минусы: собственный тип контекста (`echo.Context`) вместо стандартного `http.Handler` — привязка к фреймворку во всех хендлерах и middleware; смешение двух роутеров в монорепо запрещено требованием единообразия. **Отклонён.**

### Решение

Во всех сервисах, включая API Gateway, используется **chi**. Стандартная цепочка middleware (recover → request-id → logging → otel → auth → rate-limit) собирается один раз в `pkg/shared` и переиспользуется каждым сервисом. Хелперы биндинга/валидации запросов и рендеринга ошибок RFC 7807 — также в `pkg/shared`.

### Последствия

- Единообразный вид всех HTTP-слоёв: новый сервис поднимается копированием скелета; ревью и онбординг проще.
- Хендлеры остаются стандартными `http.Handler`/`http.HandlerFunc` — минимальная связанность с библиотекой, лёгкое тестирование через `httptest`.
- Недостающие удобства (биндинг, валидация) — небольшая разовая инвестиция в `pkg/shared`.
- **Условия пересмотра:** практически не предвидится; замена роутера возможна лишь при прекращении поддержки chi или критических проблемах производительности (маловероятно — chi это тонкая обёртка над stdlib). Любая замена — только сразу во всех сервисах, единообразие нарушать нельзя.

## ADR-007: Миграции — goose, forward-only

- **Статус:** accepted
- **Дата:** 2026-07-06
- **Автор:** AI-инженер, по согласованию с ТЗ (мастер-промт, разделы 3, 9, 10)

### Контекст

Каждый сервис владеет собственной БД (db-per-service), схемы эволюционируют на протяжении всех глав роадмапа. Нужен единый инструмент миграций с автозапуском при старте в Compose, поддержкой сидов (категории, регионы, каталог услуг, ценообразование) и политикой **forward-only** (откаты в проде запрещены; исправление — новой миграцией вперёд). ТЗ называет goose дефолтом, golang-migrate — альтернативой.

### Рассмотренные варианты

1. **goose** (выбран, дефолт ТЗ)
   - Плюсы: SQL-миграции с up/down в одном файле и понятной нумерацией; поддержка Go-миграций для сложных сидов/трансформаций данных; удобно встраивается в сервис как библиотека (автозапуск миграций при старте контейнера) и работает как CLI; простая модель версионирования.
   - Минусы: менее строгая проверка «грязного» состояния, чем у golang-migrate (компенсируется forward-only-политикой и ревью).
2. **golang-migrate**
   - Плюсы: широкая поддержка источников/БД; строгий dirty-state механизм.
   - Минусы: up/down в отдельных файлах (больше файлового шума); Go-миграции для программных сидов менее удобны; преимущества (много СУБД, много источников) для нашего стека избыточны. **Отклонён.**

### Решение

Во всех сервисах используется **goose** с политикой **forward-only**: down-секции пишутся только для локальной разработки и никогда не выполняются в прод-окружении; деструктивные изменения (DROP, изменение типа) — только с явным планом и через промежуточные совместимые шаги (expand → migrate → contract). Миграции каждого сервиса живут рядом с сервисом, применяются автоматически при старте, проходят обязательное ревью. Сиды оформляются как отдельные миграции (SQL или Go).

### Последствия

- Единый воспроизводимый механизм эволюции всех N баз; `docker compose up` даёт актуальные схемы и сиды без ручных шагов.
- Forward-only дисциплинирует проектирование изменений: обратная совместимость схемы на время выката становится нормой (важно при нескольких репликах сервиса).
- При автозапуске миграций несколькими репликами используется advisory-lock (goose поддерживает), чтобы избежать гонок.
- **Условия пересмотра:** потребность в фичах, которых нет в goose (например, специфичные источники миграций в CI/CD-платформе при переходе на K8s) — тогда сравнение с golang-migrate повторяется отдельным ADR. Политика forward-only пересмотру не подлежит.

## ADR-008: Идентификаторы — UUIDv7 через github.com/google/uuid

- **Статус:** accepted
- **Дата:** 2026-07-06
- **Автор:** AI-инженер, по согласованию с ТЗ (мастер-промт, раздел 6)

### Контекст

Всем сущностям системы (users, ads, media, conversations, payments и т.д.) нужны первичные ключи, которые: генерируются на стороне приложения (важно для outbox — ID известен до коммита и попадает в событие), глобально уникальны между сервисами и БД (db-per-service, будущий шардинг), не раскрывают бизнес-информацию (перечислимость последовательных ID — источник IDOR-разведки) и не разрушают локальность B-tree-индексов при вставке.

### Рассмотренные варианты

1. **UUIDv7 (github.com/google/uuid)** (выбран)
   - Плюсы: старшие биты — Unix-время в миллисекундах → **сортируемость по времени** и монотонная вставка в индекс (нет случайного разброса страниц, как у v4); стандарт RFC 9562; нативный тип `uuid` в PostgreSQL; библиотека google/uuid — де-факто стандарт в Go, поддерживает `uuid.NewV7()`; генерация на клиенте без похода в БД.
   - Минусы: 16 байт против 8 у bigint; временная компонента слегка раскрывает момент создания записи (приемлемо: `created_at` и так публичен в API).
2. **UUIDv4**
   - Плюсы: та же уникальность и та же библиотека.
   - Минусы: случайное распределение по индексу — write-amplification и фрагментация B-tree на больших таблицах (`ads`, `messages`); нет сортируемости по времени. **Отклонён.**
3. **bigserial / последовательности БД**
   - Плюсы: компактность, скорость.
   - Минусы: ID известен только после INSERT (неудобно для outbox); перечислимость (IDOR-разведка, раскрытие объёмов бизнеса); коллизии при слиянии/шардинге данных. **Отклонён.**
4. **ULID/KSUID (сторонние библиотеки)**
   - Плюсы: тоже time-ordered.
   - Минусы: нестандартные типы/кодировки, лишняя зависимость; UUIDv7 даёт то же самое в рамках стандартного `uuid`-типа. **Отклонены.**

### Решение

Все первичные ключи — **UUIDv7**, генерация через `github.com/google/uuid` (`uuid.Must(uuid.NewV7())` c обработкой ошибки в инфраструктурном коде), хранение — нативный тип `uuid` в PostgreSQL. Генерация выполняется в сервисном слое до транзакции, чтобы ID был доступен для outbox-события и ответа API.

### Последствия

- Единая идентификация во всех сервисах и событиях; cursor-пагинация по `id` получает естественный хронологический порядок.
- Индексы по PK ведут себя как при монотонных ключах — предсказуемая производительность вставки в горячие таблицы.
- Версия библиотеки фиксируется в `pkg/shared` (обёртка `NewID()`), чтобы все сервисы генерировали ID единообразно.
- **Условия пересмотра:** практически не предвидится. Точечный пересмотр возможен для сверхгорячих таблиц (например, `messages`) в пользу составных ключей при партиционировании — отдельным ADR.

## ADR-009: Деньги — int64 в тийинах, валюта хранится явно

- **Статус:** accepted
- **Дата:** 2026-07-06
- **Автор:** AI-инженер, по согласованию с ТЗ (мастер-промт, разделы 6, Приложение B)

### Контекст

Деньги присутствуют в нескольких сервисах: цена объявления (Listing), прайс платных услуг (`pricing`), кошелёк и леджер транзакций двойной записи (Payments), суммы в интеграциях Payme/Click. Ошибки округления в деньгах недопустимы, особенно в леджере, где суммы проводок обязаны сходиться до единицы. Валюта площадки — UZS; тийин — минимальная единица (1 сум = 100 тийинов). Payme в своём API также оперирует суммами в тийинах.

### Рассмотренные варианты

1. **int64 в минимальных единицах (тийинах) + явное поле `currency`** (выбран)
   - Плюсы: точная целочисленная арифметика без ошибок представления; прямое соответствие формату Payme; `BIGINT` в PostgreSQL — компактно и быстро; диапазона int64 хватает с запасом (~9.2×10^18 тийинов); явная валюта исключает неоднозначность трактовки суммы и закладывает мультивалютность «на вырост».
   - Минусы: конвертация тийины↔сумы на границах (API/отображение, Click оперирует суммами в сумах) — нужны хелперы; риск ошибок «сумма в чём?» — купируется типом-обёрткой и конвенцией именования.
2. **NUMERIC/DECIMAL в БД + decimal-библиотека в Go**
   - Плюсы: «человеческие» суммы в БД; произвольная точность.
   - Минусы: в Go нет стандартного decimal — сторонняя зависимость во всех сервисах; медленнее и тяжелее BIGINT; риск случайной float-конвертации на границах сериализации. **Отклонён.**
3. **float64**
   - Минусы: двоичное представление не хранит десятичные доли точно; для денег недопустим. **Отклонён безусловно.**

### Решение

Все денежные суммы хранятся как **int64 в тийинах** (минимальных единицах UZS), рядом всегда — явное поле **`currency`** (в v1 значение `"UZS"`, CHECK-ограничение). В Go вводится тип `Money { Amount int64; Currency string }` в `pkg/shared` с конвертацией в/из сумов только на границах (рендеринг, интеграция Click). Поля в БД и JSON именуются с суффиксом, снимающим неоднозначность (например, `amount_tiyin`). Леджер кошелька оперирует только тийинами.

### Последствия

- Арифметика денег точна во всех сервисах; сверка леджера — целочисленная.
- Интеграция Payme прямая (тийины), для Click — конвертация в хелпере адаптера.
- Пропорциональный расчёт возвратов (неиспользованные дни услуги) выполняется в тийинах с явно зафиксированным правилом округления (в пользу пользователя — уточняется в ARCHITECTURE.md при реализации CHAPTER 8).
- **Условия пересмотра:** появление мультивалютности (потребует таблицы валют с числом минимальных единиц) или деноминация UZS. Само представление int64+currency при этом сохраняется — меняется только справочник валют.

## ADR-010: Логирование — стандартный log/slog (JSON), без zerolog

- **Статус:** accepted
- **Дата:** 2026-07-06
- **Автор:** AI-инженер, по согласованию с ТЗ (мастер-промт, разделы 8, 10)

### Контекст

Требования к логированию: структурированный JSON во всех сервисах, обязательные поля trace-id/correlation-id (request-id), уровни, единый формат для сбора в Loki и корреляции с трейсами Tempo. RULES.md допускает slog или zerolog; нужен один вариант для всего монорепо.

### Рассмотренные варианты

1. **log/slog (stdlib)** (выбран)
   - Плюсы: **стандартная библиотека** — ноль внешних зависимостей у самого базового инструмента, который импортирует каждый пакет; `JSONHandler` из коробки; расширяемость через собственный `Handler` (автодобавление trace_id/request_id из контекста); slog стал точкой интеграции экосистемы (мосты для otel, middleware); API стабилен как часть Go.
   - Минусы: несколько медленнее zerolog на синтетических бенчмарках (аллокации на атрибуты); чуть более многословный API.
2. **zerolog**
   - Плюсы: один из самых быстрых JSON-логгеров (zero-allocation), fluent API.
   - Минусы: внешняя зависимость в каждом сервисе ради базовой функции; свой несовместимый API — привязка всего кода к библиотеке; экосистема Go консолидируется вокруг slog-интерфейса. **Отклонён.**

Производительность логгера не является узким местом Bozor: объём логов на запрос умеренный, а разница в наносекундах на запись несопоставима с сетевыми и БД-задержками.

### Решение

Во всех сервисах используется **log/slog** с `JSONHandler` (вывод в stdout — 12-factor, сбор через Loki). В `pkg/shared/logging` создаётся преднастроенный логгер: обязательные поля `service`, `env`, `version`; кастомный Handler добавляет `trace_id`, `span_id`, `request_id` из `context.Context`; уровень задаётся через env. Прямое использование `fmt.Print`/`log` в сервисном коде запрещено линтером.

### Последствия

- Меньше зависимостей: базовый слой логирования не тянет сторонних библиотек; обновляется вместе с Go.
- Единый формат логов гарантирует работоспособность корреляции логи↔трейсы в Grafana из коробки.
- Весь код логирует через `*slog.Logger` из контекста/DI — замена Handler (например, на семплирующий) не требует правок в бизнес-коде.
- **Условия пересмотра:** только доказанная профилировкой деградация от логгера на горячем пути (крайне маловероятно); в этом случае сначала рассматривается семплирование/снижение уровня, и лишь затем — альтернативный Handler-бэкенд под тем же slog-API (без смены API — это преимущество slog).

## ADR-011: Монорепо с go.work, отдельный go.mod на сервис + общий модуль pkg/shared

- **Статус:** accepted
- **Дата:** 2026-07-06
- **Автор:** AI-инженер, по согласованию с ТЗ (мастер-промт, раздел 10)

### Контекст

ТЗ фиксирует монорепо со структурой `/services/<name>`, `/pkg/shared`, `/api`, `/deploy`, `/scripts`, `/docs`. Открытым остаётся вопрос модульной организации Go-кода: один общий `go.mod` на весь репозиторий или отдельные модули. От этого зависят изоляция зависимостей, размер Docker-образов, скорость CI и удобство локальной разработки (13 сервисов + общая библиотека).

### Рассмотренные варианты

1. **go.work + отдельный go.mod на каждый сервис + отдельный модуль `pkg/shared`** (выбран)
   - Плюсы: **изоляция зависимостей** — тяжёлые библиотеки одного сервиса (Telegram Bot API у Auth/Notification, Typesense-клиент у Search, платёжные SDK у Payments) не попадают в `go.sum` и граф сборки остальных; независимые graceful-обновления зависимостей по сервисам; Docker-сборка каждого сервиса видит только свой модуль + shared (меньше кеш-инвалидаций); `go.work` даёт бесшовную локальную разработку — правки в `pkg/shared` немедленно видны всем сервисам без `replace`-директив; общий код версионируется вместе с сервисами в одном репозитории (атомарные PR через границы модулей).
   - Минусы: больше файлов `go.mod`/`go.sum`; CI и Makefile должны уметь итерироваться по модулям (`go work` / список модулей); в Docker-сборке нужно копировать `pkg/shared` рядом с сервисом.
2. **Единый go.mod на весь монорепозиторий**
   - Плюсы: максимально просто — одна команда `go build ./...`, один граф зависимостей.
   - Минусы: общий `go.sum` — зависимость любого сервиса становится зависимостью всех (раздутые образы при неаккуратной сборке, лишние срабатывания govulncheck, конфликт версий одной библиотеки между сервисами неразрешим); обновление зависимости затрагивает весь репозиторий сразу. **Отклонён.**
3. **Полирепо (репозиторий на сервис)**
   - Плюсы: предельная изоляция.
   - Минусы: противоречит ТЗ (монорепо); версионирование `pkg/shared` через теги и прокси — постоянные накладные расходы; атомарные изменения контракта + потребителей невозможны в одном PR. **Отклонён.**

### Решение

Принята схема: **корневой `go.work`**, перечисляющий все модули; **отдельный `go.mod`** в каждом `services/<name>`; **отдельный модуль `pkg/shared`** (config, logging, errors, otel, auth-middleware, http/grpc-хелперы, outbox). Внутри репозитория сервисы ссылаются на shared по имени модуля, разрешение — через workspace (без `replace`). CI линтует/тестирует каждый модуль; Docker-сборка сервиса копирует `pkg/shared` и свой каталог. `go.work` в образы не попадает (используется `GOFLAGS=-mod=mod` либо `go work vendor`-подход — фиксируется в RULES.md при реализации Stage 0.1).

### Последствия

- Зависимости сервисов не «протекают» друг в друга; аудит зависимостей (govulncheck) точен по каждому сервису.
- Изменение `pkg/shared` — это осознанное событие: CI прогоняет тесты всех сервисов-потребителей; ломающие правки shared видны в том же PR.
- Makefile получает цели уровня модуля (`lint-<svc>`, `test-<svc>`) и агрегирующие цели по workspace.
- Порог входа чуть выше (нужно понимать go.work), компенсируется документацией в README/RULES.md.
- **Условия пересмотра:** если накладные расходы на мульти-модульность окажутся выше пользы при малом числе реально различающихся зависимостей — допустим переход на единый go.mod (простая механическая миграция). Выделение `pkg/shared` в отдельный репозиторий — только при появлении внешних потребителей.

## ADR-012: JWT — github.com/golang-jwt/jwt/v5; access ~15 мин, refresh ~30 дней с ротацией, reuse-detection и family-отзывом

- **Статус:** accepted
- **Дата:** 2026-07-06
- **Автор:** AI-инженер, по согласованию с ТЗ (мастер-промт, разделы 3, 5.1, 8)

### Контекст

После Telegram-авторизации (ADR-003) Auth-сервис выпускает токены, которые проверяются на API Gateway для каждого запроса (stateless-проверка, без похода в Auth). Требования ТЗ: короткоживущий access-JWT (claims `sub`, `roles`, `exp`, `iat`, `jti`), долгоживущий refresh с хранением в виде хеша, ротацией при каждом использовании, привязкой к `device_id` и **reuse-detection** (повторное использование уже отозванного refresh — признак кражи → отзыв всего семейства токенов устройства).

Нужно выбрать библиотеку JWT и зафиксировать параметры жизненного цикла токенов.

### Рассмотренные варианты

**Библиотека:**

1. **github.com/golang-jwt/jwt/v5** (выбрана)
   - Плюсы: де-факто стандарт в Go-экосистеме (поддерживаемый форк dgrijalva/jwt-go); v5 — актуальный API с строгой валидацией (`WithValidMethods`, обязательная проверка `exp`); широкая совместимость примеров/middleware; достаточна для нашего сценария (подпись и верификация компактных JWT).
   - Минусы: только JWS-примитивы — сессионную логику (ротация, семейства) пишем сами (мы и так пишем её по требованиям ТЗ).
2. **lestrrat-go/jwx**
   - Плюсы: полный набор JOSE (JWE, JWK, key sets).
   - Минусы: избыточен — шифрование токенов и JWKS-ротация ключей в v1 не требуются; тяжелее API. **Отклонён (кандидат при переходе на JWKS).**
3. **Opaque-токены (случайные строки + проверка в Redis/БД)**
   - Плюсы: мгновенный отзыв access.
   - Минусы: каждый запрос требует похода в хранилище — теряется stateless-проверка на gateway, добавляется задержка и точка отказа; противоречит выбору «JWT access» в ТЗ. **Отклонён для access; для refresh фактически используется (см. решение).**

**Параметры (варьировались TTL):** более короткий access (5 мин) увеличивает нагрузку на refresh-эндпоинт; более длинный (60 мин) увеличивает окно жизни украденного access и задержку применения бана. 15 минут — баланс, предложенный ТЗ. Refresh 30 дней соответствует ожиданию «вход раз в месяц» для classifieds-аудитории.

### Решение

- Библиотека — **github.com/golang-jwt/jwt/v5**; подпись HS256 с секретом из env в v1 (один издатель — Auth; gateway получает ключ верификации через env). Переход на RS256/EdDSA + JWKS — задокументированная эволюция при появлении внешних верификаторов.
- **Access-токен:** JWT, TTL **~15 минут**, claims: `sub` (user_id), `roles`, `exp`, `iat`, `jti`. Проверяется на gateway; `user_id`/roles пробрасываются во внутренние вызовы заголовками/метаданными.
- **Refresh-токен:** непрозрачная криптослучайная строка (не JWT), TTL **~30 дней**; в БД (`refresh_tokens`) хранится только хеш (SHA-256), вместе с `user_id`, `device_id`, `family_id`, `expires_at`, `revoked_at`.
- **Ротация:** каждый `POST /auth/refresh` гасит использованный refresh и выдаёт новую пару (access + refresh) в том же `family_id`.
- **Reuse-detection:** предъявление refresh-токена, который уже был использован/отозван, трактуется как компрометация → **отзывается всё семейство** (`family_id`) токенов данного устройства, событие пишется в `auth_audit_log`.
- **Logout:** отзыв семейства текущего устройства; ролевые изменения/бан применяются не позднее истечения access-TTL (≤15 мин), при необходимости мгновенного бана gateway дополнительно сверяет короткий denylist в Redis по `jti` (включается при реализации банов в CHAPTER 6).

### Последствия

- Gateway проверяет access локально — нет синхронной зависимости от Auth на каждом запросе; Auth масштабируется независимо.
- Окно действия отозванных прав ограничено 15 минутами — приемлемо для v1 и явно задокументировано; для банов предусмотрен denylist-механизм.
- Схема refresh (хеш + семейства + ротация) устойчива к утечке БД (хранится только хеш) и к краже refresh-токена (reuse-detection ловит параллельное использование).
- Реализация семейства/ротации покрывается обязательными тестами (включая сценарий reuse) — входит в DoD Stage 1.4.
- **Условия пересмотра:** появление внешних потребителей верификации токенов или нескольких издателей → переход на асимметричную подпись (RS256/EdDSA) с JWKS (возможна смена библиотеки на jwx) — отдельным ADR; изменение TTL — по данным эксплуатации (нагрузка на refresh, инциденты безопасности).

## ADR-013: Пул соединений — один PostgreSQL-инстанс + PgBouncer в transaction-режиме с prepared statements

- **Статус:** accepted
- **Дата:** 2026-07-07
- **Автор:** AI-инженер, по согласованию с ТЗ (мастер-промт, разделы 3, 10, Приложение B)

### Контекст

Схема db-per-service (ADR-004, инвариант ТЗ) требует изолированной БД на каждый из ~12 сервисов. На одном Compose-хосте разворачивать 12 отдельных инстансов PostgreSQL расточительно, поэтому используется **один инстанс с отдельной базой на сервис** (init-скрипт создаёт `bozor_auth`, `bozor_catalog`, …). Каждый сервис имеет несколько реплик и держит собственный pgxpool; при десятках тысяч DAU суммарное число прямых соединений к PostgreSQL легко превышает комфортные `max_connections`. Нужен внешний пулер.

Ключевой нюанс: pgx v5 по умолчанию работает в режиме `cache_statement` — кеширует **серверные** prepared statements по имени. В PgBouncer с `pool_mode = transaction` серверное соединение возвращается в пул после каждой транзакции, поэтому именованный statement, подготовленный в одной транзакции, в следующей уже на другом бэкенде — классическая ошибка `prepared statement "stmtcache_..." does not exist`. Исторически это вынуждало переключать pgx в `simple_protocol`/`exec` (теряя протокол-биндинг и часть производительности) либо использовать session-режим пула (убивает мультиплексирование).

### Рассмотренные варианты

1. **Один инстанс + PgBouncer `transaction` + `max_prepared_statements > 0`, pgx на дефолтном `cache_statement`** (выбран)
   - Плюсы: PgBouncer >= 1.21 умеет отслеживать протокол-левел prepared statements в transaction-режиме (`max_prepared_statements`), поэтому дефолтное поведение pgx работает **без изменений в коде сервисов**; сохраняется максимальное мультиплексирование клиентов (`MAX_CLIENT_CONN=1000` поверх небольшого `DEFAULT_POOL_SIZE`); никакой потери типобезопасности/производительности extended-протокола. Наш образ — `edoburu/pgbouncer:v1.24.1` (проверено: env `MAX_PREPARED_STATEMENTS` рендерится в `pgbouncer.ini`).
   - Минусы: небольшой расход памяти PgBouncer на кеш дескрипторов statement'ов; зависимость от версии PgBouncer >= 1.21.
2. **pgx в `simple_protocol`/`exec`-режиме, PgBouncer `transaction` без prepared statements**
   - Плюсы: работает с любым (в т.ч. старым) PgBouncer.
   - Минусы: `simple_protocol` теряет протокол-биндинг параметров (pgx интерполирует их клиентски) и серверный кеш планов; `exec` шлёт describe на каждый запрос. Ненужный компромисс при доступном PgBouncer 1.24. **Отклонён.**
3. **PgBouncer в `session`-режиме**
   - Плюсы: prepared statements работают «из коробки».
   - Минусы: соединение закреплено за клиентом на всю сессию — мультиплексирование теряется, число серверных соединений ≈ числу клиентских, что обесценивает пулер под нашей нагрузкой. **Отклонён.**
4. **Отдельный инстанс PostgreSQL на сервис**
   - Плюсы: максимальная изоляция ресурсов/отказов.
   - Минусы: 12 инстансов на одном хосте — избыточный оверхед по RAM/CPU/эксплуатации; db-per-service уже даёт логическую изоляцию. **Отклонён** (пересматривается при выносе горячих БД на отдельные хосты).

### Решение

Один PostgreSQL-инстанс с базой на сервис; перед ним **PgBouncer в `pool_mode = transaction`** с `MAX_PREPARED_STATEMENTS=200`, `MAX_CLIENT_CONN=1000`, `DEFAULT_POOL_SIZE=20`, `AUTH_TYPE=scram-sha-256`. Сервисы подключаются через PgBouncer (порт 6432) и оставляют pgx на дефолтном `cache_statement`. Все прикладные соединения идут только через пулер; прямой доступ к 5432 — лишь для миграций/обслуживания.

### Последствия

- Код репозиториев (ADR-005) не требует спец-настройки exec-режима pgx — на один класс ошибок меньше; при этом сохраняется мультиплексирование под высокой конкуррентностью.
- Миграции goose (ADR-007) выполняются в отдельной транзакции/через прямое соединение к 5432, чтобы DDL и advisory-lock не конфликтовали с transaction-пулером.
- Значения пула (`DEFAULT_POOL_SIZE`, `max_connections=200` на самом PostgreSQL) — стартовые; пересматриваются на Stage 10.3 по данным нагрузочного теста.
- **Условия пересмотра:** вынос отдельной БД на выделенный инстанс/хост при её росте; отказ от prepared statements в пользу `exec`-режима, если понадобится совместимость со старым PgBouncer или внешним пулером без их поддержки.

## ADR-014: API Gateway — тонкий reverse-proxy с optional-auth, проброс идентичности и fail-open rate-limit

- **Статус:** accepted
- **Дата:** 2026-07-07
- **Автор:** AI-инженер, по согласованию с ТЗ (мастер-промт, раздел 5; ARCHITECTURE.md §2.6)

### Контекст

API Gateway (Stage 0.4) — единая точка входа за edge-nginx: маршрутизация `/api/v1/*` на внутренние сервисы, проверка JWT и проброс идентичности, rate-limit (Redis token-bucket), CORS, request-id, единый формат ошибок RFC 7807. Требование ТЗ — **gateway без бизнес-логики**. На момент реализации ни один прикладной сервис ещё не поднят, поэтому нужно принять решения, которые не завязаны на конкретные сервисы и не придётся ломать позже.

Три развилки требуют фиксации: (1) где проходит граница «требуется вход» — на gateway или в сервисах; (2) поведение rate-limiter при недоступности Redis; (3) как gateway передаёт проверенную идентичность внутрь, не открывая вектор подделки.

### Рассмотренные варианты

**1. Модель аутентификации на gateway**
- **Optional-auth (выбрано):** токен проверяется, только если передан; отсутствие токена — не ошибка (аноним пропускается), недействительный токен → 401. Требование «нужен вход» для конкретного эндпоинта обеспечивают сами сервисы (у них уже есть `authx.FromForwardedHeaders` + проверка роли). Плюсы: gateway не кодирует бизнес-правила доступа (браузинг объявлений публичен, публикация — нет — это знание домена, а не шлюза); таблица маршрутов остаётся тонкой и не протухает при добавлении эндпоинтов; ровно то, что просит ТЗ («gateway тонкий»). Минус: «дырявый» эндпоинт, забывший проверку входа, не подстрахован на gateway — компенсируется тем, что проверка владения/роли и так обязательна в сервисах (инвариант IDOR).
- **Строгая таблица protected-роутов на gateway (отклонено):** gateway хранит, какие пути требуют логина. Плюс: централизованный «забор». Минус: дублирует и размазывает доменные правила доступа между gateway и сервисами; таблица требует правки при каждом новом эндпоинте; ближе к бизнес-логике, что запрещено ТЗ. Возврат к этому возможен отдельным ADR, если появится потребность в edge-политиках.

**2. Rate-limit при недоступности Redis**
- **Fail-open (выбрано):** ошибка лимитера логируется (WARN) и запрос пропускается. Плюс: защита от злоупотреблений не должна превращать блип Redis в полный отказ шлюза (иначе rate-limiter — единая точка отказа всего API). Минус: в окно недоступности Redis лимиты не действуют — приемлемо, т.к. цель rate-limit — сгладить злоупотребление, а не обеспечить корректность.
- **Fail-closed (отклонено):** при ошибке Redis → 503. Делает Redis SPOF для всего трафика. Не подходит для edge-компонента.

**3. Проброс идентичности**
- **Strip + inject заголовков `X-User-Id`/`X-User-Roles` (выбрано):** gateway на входе **вычищает** эти заголовки у клиента (антиподделка), а после успешной проверки JWT **проставляет** их из клеймов; внутренние сервисы доверяют им (`authx.FromForwardedHeaders`) в пределах приватной сети. Ключ rate-limit — по `user_id` (если аутентифицирован) либо по IP. Соответствует ТЗ: «сервисы доверяют уже проверенной идентичности».
- **Пере-подпись внутреннего JWT (отклонено для v1):** избыточно на одном хосте в доверенной сети; добавляет криптооверхед на каждый вызов. Кандидат при переходе на mTLS/недоверенную сеть.

**Rate-limit — реализация:** token-bucket поверх Redis, **атомарно через Lua** (проверка+списание в одном round-trip); время берётся сервером Redis (`redis.call('TIME')`), чтобы реплики gateway были согласованы без клиентского clock-skew; TTL ключа = время полного пополнения (неактивные ключи сами исчезают). Клиент получает заголовки `X-RateLimit-Limit/Remaining/Reset` и `Retry-After`, при исчерпании — 429 RFC 7807.

### Решение

Gateway на **chi** с общей middleware-цепочкой из `pkg/shared` (recover → request-id → lang → access-log → otel → CORS → strip-identity; далее под `/api/v1`: optional-auth → rate-limit → reverse-proxy). Проксирование — `httputil.ReverseProxy` на сервис из таблицы префиксов; недоступность апстрима → 503 RFC 7807 (а не голый 502). Redis-клиент — `github.com/redis/go-redis/v9`; роутер — `github.com/go-chi/chi/v5`. Образ — multi-stage → distroless static nonroot; healthcheck без shell через флаг `/gateway -health`. Бизнес-логики в gateway нет.

Redis-клиент пока живёт в модуле gateway; общий `pkg/shared/redisx` будет выделен, когда Redis понадобится второму сервису (YAGNI).

### Последствия

- Добавление новых сервисов сводится к строке в таблице маршрутов + записи `UPSTREAM_<SERVICE>` (по умолчанию `http://<service>:8080`); auth/limit/CORS/ошибки работают из коробки.
- Ответственность за «требуется вход» и авторизацию — на сервисах (инвариант IDOR, RULES §11); gateway централизует только проверку подписи токена и проброс идентичности.
- Rate-limit не является SPOF; ценой является отсутствие лимитов в редкие окна недоступности Redis (залогировано).
- **Условия пересмотра:** переход на недоверенную внутреннюю сеть (→ mTLS/внутренний JWT вместо доверия заголовкам); потребность в edge-политиках доступа (→ строгая таблица protected-роутов); вынос token-bucket/`redisx` в `pkg/shared` при втором потребителе Redis; подключение OTLP-экспортёра на Stage 0.5.

## ADR-015: Наблюдаемость — OTel traces→Tempo, OTel metrics→Prometheus (pull), логи→Loki через Alloy; без центрального коллектора

- **Статус:** accepted
- **Дата:** 2026-07-07
- **Автор:** AI-инженер, по согласованию с ТЗ (мастер-промт, раздел 8; ARCHITECTURE.md §7.5)

### Контекст

Stage 0.5 разворачивает три столпа наблюдаемости на одном Compose-хосте: трейсы, метрики (RED на сервис, USE на ресурсы), логи (структурированный JSON с trace/request id) — с визуализацией в Grafana и базовыми правилами алертинга. Нужно выбрать конвейер сбора, минимально нагружающий хост и единообразный для всех будущих сервисов, чтобы новый сервис получал наблюдаемость «из коробки» через `pkg/shared`.

### Рассмотренные варианты и решения

**Трейсы — OTel SDK → OTLP gRPC → Tempo (напрямую).**
- Tempo — часть Grafana-стека, дешёвое трейс-хранилище с нативным OTLP-приёмником; `pkg/shared/otelx.Setup` уже экспортирует по `OTEL_EXPORTER_OTLP_ENDPOINT`. Отклонён Jaeger (отдельная экосистема; Grafana-стек предпочтительнее для единого UI).

**Метрики — OTel MeterProvider + Prometheus-экспортёр; Prometheus scrape (pull) по `/metrics`.**
- Добавлен `otelx.SetupMetrics`: ставит глобальный MeterProvider с экспортёром `exporters/prometheus`; инструментация `otelhttp` (уже в цепочке middleware) **автоматически** пишет RED-метрики HTTP-сервера (`http_server_request_duration_seconds{http_route,http_response_status_code,...}`) — нового middleware не потребовалось. `/metrics` отдаёт их вместе с `go_*`/`process_*` (для USE).
- Отклонён ручной `client_golang`-middleware: OTel-путь консистентнее с трейсами и даёт `http_route`/метод/статус без дублирующего кода. Модель pull (Prometheus scrape) выбрана вместо push/remote-write ради простоты на одном хосте.

**Логи — JSON в stdout → Grafana Alloy (docker discovery) → Loki.**
- Сервисы уже логируют JSON в stdout (`pkg/shared/logging`); Alloy читает логи контейнеров через Docker API, извлекает `level`/`service`/`trace_id` в метки и шлёт в Loki. В Grafana derived field связывает `trace_id` из лога с трассой в Tempo.
- **Выбран Alloy, а не Promtail:** Promtail объявлен EOL (замена — Alloy). Отклонён вариант «Loki Docker logging driver» (недекларативная установка плагина в Compose).

**Топология — без центрального otel-collector.**
- Каждый сервис экспортирует трейсы напрямую в Tempo; метрики собираются pull'ом; логи — Alloy'ем. На одном хосте это меньше движущихся частей. Отклонён центральный OTel Collector (оправдан при росте: маршрутизация/семплинг/трансформация — вводится отдельным ADR).

**Алертинг — правила в Prometheus; Alertmanager позже.**
- Заданы правила (TargetDown, GatewayHighErrorRate, GatewayHighLatencyP95) — они вычисляются и видны в Prometheus `/alerts`. Доставка (Alertmanager + канал, напр. Telegram) подключается отдельным шагом, когда появится приёмник.

### Последствия

- Новый сервис получает наблюдаемость, вызвав `otelx.Setup` + `otelx.SetupMetrics` и смонтировав `/metrics`; логи собираются автоматически (Alloy видит любой контейнер). Дашборды/алерты уже опираются на semconv-имена метрик — переиспользуемы между сервисами.
- Grafana provision'ит источники (Prometheus/Loki/Tempo) и дашборды (Gateway RED, Infra Health/USE) из файлов — воспроизводимо, без ручной настройки.
- Прямой экспорт в Tempo создаёт зависимость сервиса от доступности Tempo для трасс (не критично: OTLP-экспортёр батчит и не блокирует запросы; при пустом endpoint — no-op).
- **Условия пересмотра:** рост нагрузки/числа сервисов → ввод центрального OTel Collector (семплинг, буферизация, единая точка экспорта) и/или переход метрик на remote-write; подключение Alertmanager и каналов доставки; вынос хранилищ трасс/логов на отдельные тома/хосты; добавление cadvisor/node-exporter для полноценного USE по контейнерам/хосту.

## ADR-016: Атрибуты категорий — наследование по материализованному path, вычисление на лету

- **Статус:** accepted
- **Дата:** 2026-07-08
- **Автор:** AI-инженер, по согласованию с ТЗ (ROADMAP Stage 2.3; ARCHITECTURE.md — Catalog)

### Контекст

Stage 2.3 вводит атрибуты объявлений (типизированные характеристики: enum/int/decimal/bool/string, единицы, обязательность, фильтруемость) и привязку их к категориям. Ключевое требование — **наследование вниз по дереву**: атрибуты, заданные на разделе (например, «Транспорт» → марка, год), должны применяться ко всем подкатегориям («Легковые», «Мото», …), где к ним добавляются собственные (пробег, коробка). Дерево категорий уже хранит материализованный `path` (slug'и через `/`) и `level` (ADR по Catalog, Stage 2.2). Нужно решить, как хранить и вычислять «эффективный» набор атрибутов категории.

### Рассмотренные варианты

**A. Junction-таблица `category_attributes` + вычисление эффективного набора на лету по материализованному path (выбрано).**
- Привязка хранится только на той категории, где атрибут объявлен (без копий). Эффективный набор для целевой категории `T` = все привязки категорий-предков-и-себя, найденные одним JOIN по условию `T.path = anc.path OR T.path LIKE anc.path || '/%'`, с `ORDER BY anc.level` (предки сверху) и дедупом к самому верхнему объявлению; флаг `inherited = source_id != target_id`.
- Плюсы: единственный источник истины (изменение на разделе мгновенно видно всем потомкам, без миграции данных); переиспользует уже существующий `path`; запрос — обычный self-join без рекурсии; вставка/удаление привязки — одна строка.
- Минусы: `LIKE anc.path || '/%'` — префиксный поиск (индексируемо, но при огромном дереве стоит text_pattern_ops-индекс); эффективный набор считается на каждый запрос — снимается кешированием (Stage 2.4).

**B. Рекурсивный CTE по `parent_id`.**
- Обход вверх по `parent_id` рекурсивным CTE. Отклонён: `path` уже даёт предков без рекурсии — self-join проще и предсказуемее по плану; рекурсия не даёт преимуществ, т.к. глубина дерева мала и путь материализован.

**C. Денормализация — копировать унаследованные атрибуты в каждую категорию.**
- Отклонён: изменение атрибута раздела требует массового обновления потомков (риск рассинхрона), дублирование данных, сложная инвалидация. Противоречит «единственному источнику истины».

**D. Разрешение наследования в коде приложения (загрузить все привязки, собрать в памяти).**
- Отклонён как основной путь: тащит лишние данные и переносит логику пути в приложение; БД решает это одним запросом. Дедуп/флаг `inherited` оставлены в коде поверх упорядоченной выборки — минимальная логика.

### Детали решения

- Схема (`00002_attributes`): `attributes` (slug UNIQUE, `type` CHECK по 5 типам, `unit` опц., `is_required`, `is_filterable`), `attribute_options` (варианты enum, FK `ON DELETE CASCADE`, UNIQUE(attribute_id, slug)), `category_attributes` (junction, PK(category_id, attribute_id), обе FK `ON DELETE CASCADE`, `sort_order`).
- Инвариант enum: у типа `enum` обязаны быть варианты, у остальных типов — запрещены (валидация в app-слое). Обновление атрибута заменяет набор вариантов целиком в транзакции.
- Порядок эффективного набора: сначала атрибуты предков (по `level`), затем собственные — чтобы UI шёл «общие → специфичные»; дубликат (атрибут привязан на нескольких уровнях) схлопывается к верхнему объявлению.
- Сид (`00003_seed_catalog`, идемпотентный, ADR-007): 12 разделов OLX.uz + подкатегории и реальные атрибуты Недвижимости/Транспорта/Электроники — демонстрируют наследование (общее на разделе, специфичное на подкатегории).

### Последствия

- Управление атрибутами простое: объявил на разделе — получили все потомки; специфику вешаем на подкатегорию. Нет миграций данных при изменении набора.
- Эффективный набор — «горячий» запрос публичного API (`GET /categories/{id}/attributes`); в Stage 2.4 он кешируется в Redis с инвалидацией при изменении атрибутов/привязок (аналогично кешу дерева).
- Значения атрибутов у объявлений (CHAPTER 3, listing-ads) будут валидироваться против эффективного набора категории (тип, обязательность, принадлежность enum-варианту).
- **Условия пересмотра:** очень большое/глубокое дерево → индекс `text_pattern_ops` на `path` или переход на `ltree`; потребность в переопределении унаследованного атрибута на потомке (сейчас дедуп берёт верхнее объявление) → явное правило приоритета.

## ADR-017: Media — хранение оригиналов в объектном хранилище, тип по содержимому, лимит на объявление

- **Статус:** accepted
- **Дата:** 2026-07-09
- **Автор:** AI-инженер, по согласованию с ТЗ (ROADMAP Stage 3.1; ARCHITECTURE.md — Media, Приложение B — инварианты)

### Контекст

Stage 3.1 вводит Media-сервис: приём загружаемых пользователем изображений объявлений (`POST /media`, multipart), сохранение оригинала и метаданных, публикация события для последующей обработки (превью/EXIF — воркер 3.2). Нужно решить: где и как хранить бинарь оригинала; как определять тип файла, не доверяя клиенту; как связывать медиа с объявлением при том, что Media — отдельный сервис со своей БД (инвариант «никаких кросс-БД доступов»); как ограничивать число изображений на объявление.

### Рассмотренные варианты

**A. Оригинал — в MinIO/S3 по детерминированному object_key, метаданные — в БД `media`, тип определяется сервером по содержимому, привязка к объявлению по `ad_id` без FK (выбрано).**
- Бинарь кладётся в бакет `bozor-media` под ключ `originals/<uuidv7>.<ext>`; в PostgreSQL хранится только строка метаданных (`owner_user_id`, `ad_id` NULL, `bucket`, `object_key` UNIQUE, `mime_type`, `size_bytes`, `status`, `width/height` NULL до обработки). Тип берётся из `http.DetectContentType` по первым 512 байтам (sniffing), а не из заголовка `Content-Type` части или расширения. Лимит на объявление — `COUNT(*) WHERE ad_id` перед вставкой.
- Плюсы: БД не раздувается блобами; object_key детерминирован и уникален (UUIDv7 → сортируемость по времени); тип нельзя подделать переименованием/подменой заголовка; отсутствие FK на `ads` соблюдает db-per-service (связь финализируется в `ad_images`, Stage 3.3); запись метаданных + событие в outbox — одна транзакция (инвариант outbox).
- Минусы: два хранилища (объектное + реляционное) требуют согласованности — решается компенсацией (при сбое вставки в БД залитый объект удаляется) и статусом `orphan` + TTL-очисткой (воркер 3.2); `ad_id` без FK допускает «висячую» привязку к несуществующему черновику — приемлемо, целостность обеспечит листинг при финализации.

**B. Хранить бинарь в БД (bytea).** Отклонён: раздувает БД и WAL, дорогие бэкапы, плохо раздаётся по HTTP/CDN; объектное хранилище для этого и предназначено.

**C. Доверять типу из multipart-заголовка/расширения.** Отклонён как небезопасный: клиент может назвать PDF/исполняемый файл `image/png` — sniffing по содержимому обязателен (проверено вживую: PDF с заявленным `image/png` → 422 unsupported_type).

**D. FK `ad_id` на таблицу объявлений.** Невозможен: `ads` живёт в БД другого сервиса (listing). Кросс-БД FK нарушил бы db-per-service.

### Детали решения

- Схема (`00001_init_media`): `media` (id uuid PK, `object_key` UNIQUE, `size_bytes` CHECK > 0, `status` CHECK in uploaded/ready/orphan, `width/height` NULL), частичный индекс `idx_media_ad WHERE ad_id IS NOT NULL` и `idx_media_owner`; `outbox` (DDL синхронизирован с `pkg/shared/outbox`).
- Допустимые типы: JPEG/PNG/WebP (`AllowedTypes` → расширение). Лимиты: 10 MiB на файл (`MaxBytesReader` на теле запроса + проверка размера в домене), 10 изображений на объявление.
- Владелец берётся из проброшенной gateway идентичности (`X-User-Id` после проверки access-JWT, ADR-014); анонимная загрузка → 401. Чтение метаданных (`GET /media/{id}`) публично.
- Публичный URL строится из `MEDIA_PUBLIC_BASE_URL` + object_key; бакет отдаётся download-политикой MinIO (настроена в `minio-init`, Stage 0.3). CDN подменит base URL без изменения кода.
- Событие `bozor.media.uploaded` (без PII: media_id, owner_user_id, ad_id, bucket, object_key, mime, size) публикуется через outbox → relay → NATS для воркера обработки (3.2).

### Последствия

- Оригиналы масштабируются независимо от БД; замена MinIO на облачный S3/CDN — смена endpoint и base URL.
- Согласованность «объект ↔ строка» гарантируется компенсацией при сбое и будущей очисткой сирот; воркер 3.2 проставит `width/height`, превью и переведёт `status` в `ready`.
- Листинг (3.3) свяжет подтверждённые media с объявлением через `ad_images`, тогда же решится судьба непривязанных (`orphan`).
- **Условия пересмотра:** потребность в приватных объектах (presigned URL вместо публичного бакета) → выдача временных ссылок в сервисе; антивирус/модерация загрузок → отдельный шаг перед `ready`; поддержка видео/крупных файлов → многочастная загрузка и другие лимиты.

## ADR-018: Обработка медиа — фоновый воркер в процессе сервиса, превью через imaging, снятие EXIF, очистка сирот

- **Статус:** accepted
- **Дата:** 2026-07-09
- **Автор:** AI-инженер, по согласованию с ТЗ (ROADMAP Stage 3.2; ARCHITECTURE.md — Media)

### Контекст

Stage 3.2 добавляет обработку загруженных изображений: превью 120/480/1080 px, удаление EXIF (в т.ч. GPS — приватность), presigned/CDN-ссылки и очистку «сирот» (медиа без объявления) по TTL; задачи — через брокер. Нужно решить: где выполнять обработку, какой библиотекой обрабатывать изображения без cgo (Windows-разработка), как гарантировать «ровно один раз» и как убирать неиспользуемые загрузки.

### Рассмотренные варианты и решения

**1. Где выполнять — фоновые горутины в процессе Media-сервиса (выбрано против отдельного worker-деплоя).**
- Консьюмер NATS (`natsx.Consume`, durable `media-processor`) и тикер очистки запускаются рядом с HTTP-сервером и outbox-relay — как уже сделано для relay (Stage 3.1). Один деплой, общий пул БД и клиент MinIO.
- Плюсы: меньше движущихся частей на Compose; переиспользует существующую обвязку; масштабирование консьюмера — добавлением реплик сервиса (durable-консьюмер делит работу). Минус: обработка и API делят ресурсы процесса — при росте нагрузки выносится в отдельный `cmd/worker` (событийная модель это позволяет без изменения контракта).

**2. Обработка изображений — github.com/disintegration/imaging (чистый Go, без cgo).**
- Декодирование через `image.Decode` (+ `golang.org/x/image/webp` для WebP), разворот по EXIF-ориентации (`AutoOrientation`), вписывание `imaging.Fit` (Lanczos, БЕЗ увеличения), кодирование JPEG q82 (PNG-источник → PNG, сохраняя альфу). Отклонены cgo-обёртки (libvips/ImageMagick): на Windows-разработке нет cgo (см. STATUS notes), а нагрузка (десятки тысяч DAU) для чистого Go приемлема.

**3. Удаление EXIF — через повторное кодирование, а не парсинг/зачистку тегов.**
- Декодирование в пиксели и повторное кодирование стандартным `image/jpeg`/`image/png` не переносит никаких метаданных — EXIF/GPS отбрасываются автоматически. Оригинал перезаписывается очищенной версией по тому же object_key (публичный бакет → на нём не должно быть GPS). Проверено вживую: внедрённый маркер `Exif`/GPS в загруженном JPEG отсутствует в обработанном оригинале.

**4. Идемпотентность — inbox `processed_events` + переход статуса только из uploaded.**
- Перед обработкой проверяется inbox (`outbox.AlreadyProcessed`); перевод `uploaded→ready`, событие `bozor.media.processed` и отметка inbox — в одной транзакции. Повторная доставка/ретрай не плодят превью и события. «Ядовитые»/исчерпавшие 3 попытки сообщения уходят в DLQ (natsx). Media-сервис — первый консьюмер событий в проекте.

**5. Превью — детерминированные ключи, дедуп по итоговому размеру, без апскейла.**
- Ключи `previews/<id>/<size>.<ext>`; для каждого целевого размера `Fit` вписывает без увеличения, дубликаты по фактическим размерам (маленький оригинал) схлопываются. Публичные URL превью строятся от CDN-базы (`MEDIA_PUBLIC_BASE_URL`).

**6. presigned/CDN — превью публично (CDN), оригинал владельцу по presigned-ссылке.**
- `GET /media/{id}` публичен; если идентичность запроса (проброшенная gateway) совпадает с владельцем — в ответе появляется временная presigned-ссылка на оригинал (`MEDIA_PRESIGN_TTL`). Известный нюанс: presigned-URL подписывается на внутренний endpoint MinIO (`minio:9000`); в проде `MINIO_ENDPOINT`/подпись настраиваются на публичный хост или переписываются на CDN.

**7. Очистка сирот — периодический тикер по TTL.**
- Медиа без `ad_id`, старше `MEDIA_ORPHAN_TTL` (по умолчанию 24h), пакетами удаляются каждые `MEDIA_CLEAN_INTERVAL`: сначала объекты (оригинал + превью, `RemoveObject` идемпотентен), затем запись + событие `bozor.media.deleted` в одной транзакции. Привязанные к объявлению медиа не трогаются.

### Последствия

- Загруженное фото автоматически получает превью и лишается GPS без участия клиента; лента/карточка объявления используют превью (CDN), владелец — presigned-оригинал.
- Непривязанные загрузки не накапливаются вечно. Listing-сервис (3.3) свяжет медиа с объявлением (`ad_id`), выведя их из-под очистки.
- **Условия пересмотра:** рост нагрузки → вынести обработку в отдельный `cmd/worker`; потребность в WebP/AVIF-превью или строгой ориентации на весь парк устройств → сменить кодек/пайплайн; приватные оригиналы → отдавать строго presigned (закрыть публичный доступ к `originals/`).

## ADR-019: Синхронный контракт Listing↔Catalog — gRPC (proto+buf), генерат в pkg/shared/pb

- **Статус:** accepted
- **Дата:** 2026-07-09
- **Автор:** AI-инженер, по согласованию с ТЗ (ARCHITECTURE §3.3/§4.5; ROADMAP Stage 3.3)

### Контекст

Stage 3.3 вводит Listing-сервис, который при создании объявления обязан валидировать значения атрибутов против эффективного набора категории, живущего в Catalog (другой сервис/БД — кросс-БД доступ запрещён). Нужен синхронный запрос-ответ. ARCHITECTURE §3.2 фиксирует для таких сценариев gRPC (protobuf, buf); pkg/shared уже содержит grpcx (otel+recovery) с 0.2. Это первый gRPC-контракт в проекте — решение задаёт схему для будущих (chat, moderation и т.д.).

### Рассмотренные варианты

**A. gRPC-контракт `catalog.v1.CatalogService.GetEffectiveAttributes`, генерат Go в общий модуль pkg/shared/pb (выбрано).**
- `.proto` в `proto/catalog/v1/`, `buf generate` (плагины protoc-gen-go/-go-grpc, `make proto`) → `pkg/shared/pb/catalog/v1` (package catalogv1). Общий модуль импортируется и сервером (catalog), и клиентом (listing) — единый источник контракта, без дублей. Catalog поднимает gRPC-сервер на отдельном порту (:9090) поверх существующего `app.AttributeService.Effective`; listing — gRPC-клиент, реализующий доменный интерфейс `app.CatalogValidator` (codes.NotFound→ErrCategoryNotFound). Транспорт через grpcx (otel-трейс сквозь вызов, recovery).
- Плюсы: строгий типизированный контракт и обратная совместимость по правилам buf; трассировка запроса Listing→Catalog в Tempo; клиент за интерфейсом — тесты listing не зависят от сети (фейковый CatalogValidator); переиспользуемо для будущих сервисов.
- Минусы: генерируемый код и tooling (buf, плагины); +порт у catalog. Приемлемо: buf уже в стеке, инфраструктура одноразовая.

**B. HTTP-вызов Catalog (`GET /categories/{id}/attributes`).** Отклонён как основной: ТЗ требует gRPC для межсервисных синхронных вызовов; слабее контракт (JSON без схемы), нет codegen/breaking-checks. HTTP-эндпоинт остаётся для публичного клиента, но сервис-сервис — gRPC.

**C. Дублировать данные Catalog в Listing (кэш/репликация атрибутов событиями).** Отклонён для 3.3: усложнение и риск рассинхрона ради валидации; синхронный вызов проще и точен. Кэширование ответов Catalog — возможная будущая оптимизация.

### Детали решения

- Контракт: `GetEffectiveAttributes(category_id) → repeated AttributeSpec{slug,type,required,options}`; NOT_FOUND при отсутствии категории. Значение `type` — строка (enum/int/decimal/bool/string), `options` — slug'и вариантов enum.
- Генерат в pkg/shared/pb (общий модуль уже тянет grpc/protobuf через grpcx). buf.yaml/buf.gen.yaml в корне; `make proto` ставит плагины и генерирует; пакет proto = `catalog.v1` (директория совпадает — buf lint STANDARD чист).
- Валидация значений — в domain listing (`ValidateAttributes`): неизвестный атрибут запрещён, обязательные обязательны, значение соответствует типу (enum⊆options, int/decimal парсятся, bool∈{true,false}, string непусто).
- Отказоустойчивость: соединение grpc.NewClient ленивое; недоступность Catalog всплывает при создании (ошибка → 5xx). Валидация — обязательный барьер: без Catalog объявление не создать (консистентность важнее доступности записи).

### Последствия

- Listing — источник истины по объявлениям — гарантирует, что сохранённые значения атрибутов согласованы с актуальной схемой категории Catalog (проверено вживую: несуществующая категория → gRPC NOT_FOUND → 422; неверный enum/обязательный/неизвестный атрибут → 422).
- Появилась схема межсервисного gRPC: proto→pkg/shared/pb→сервер+клиент+grpcx. Следующие контракты повторяют её.
- **Условия пересмотра:** рост числа вызовов на создание → кэш эффективных наборов в Listing (инвалидация по bozor.category.*/событиям атрибутов); потребность в мягкой деградации → разрешить создание в draft без немедленной валидации с отложенной проверкой на submit (pending).

## ADR-020: Жизненный цикл объявления — доменная статусная машина, in-process консьюмер решений модерации (inbox) и воркер истечения

- **Статус:** accepted
- **Дата:** 2026-07-09
- **Автор:** AI-инженер, по согласованию с ТЗ (ARCHITECTURE §4.5; ROADMAP Stage 3.4)

### Контекст

Stage 3.4 наделяет объявление полным жизненным циклом (ARCHITECTURE §4.5): draft→pending→active/rejected; active→sold/expired/archived; blocked; повторная модерация после правки; renew; истечение по `expires_at`. Решения модерации приходят из будущего Moderation-сервиса событиями `bozor.ad.approved|rejected` (CHAPTER 6 ещё не реализован), а истечение — фоновый процесс по времени. Нужны: (1) единая точка правил переходов, (2) надёжное потребление решений «ровно один раз», (3) периодический перевод активных в expired. pkg/shared уже даёт outbox (transactional outbox + inbox `processed_events`), natsx (durable-консьюмер с DLQ) и образец в Media (ADR-018).

### Рассмотренные варианты

**A. Статусная машина в domain + in-process консьюмер решений (inbox-идемпотентность) + тикер-воркер истечения (выбрано).**
- Правила переходов — таблицей `transitions` в `domain` (единственный источник; `CanTransitionTo`), переходы применяются условным `UPDATE ... WHERE status = $from` (защита от гонок/повторов). Действия владельца (submit/sold/renew/archive) — методы `app.Service` поверх статусной машины с проверкой права (`ErrForbidden`) и публикацией `bozor.ad.updated|sold` через outbox одной транзакцией.
- Консьюмер `bozor.ad.approved|rejected` (durable `listing-moderation`, до 3 попыток → DLQ) идемпотентен через inbox `processed_events` + переход только из `pending`: повторная доставка и события «не в модерации» не выполняют работу дважды. Одобрение проставляет `published_at` и `expires_at = now + LISTING_AD_TTL`, публикует `bozor.ad.updated(active)` (источник истины для Search).
- Воркер истечения — тикер (`LISTING_EXPIRE_INTERVAL`) выбирает активные с `expires_at <= now` пакетом (`LISTING_EXPIRE_BATCH`, индекс `idx_ads_expires`) и переводит active→expired условным UPDATE (перепроверка `expires_at <= now()` — только что продлённое renew не истекает), публикуя `bozor.ad.expired`.
- Плюсы: один деплой (HTTP + relay + консьюмер + тикер рядом, как в Media); правила переходов в одном месте и тестируются таблицей; exactly-once по решениям без распределённых блокировок; события источника истины идут в шину транзакционно. Минусы: воркеры живут в каждой реплике сервиса (координация не нужна — UPDATE `WHERE status=$from` и SKIP-LOCKED-стиль делают проходы идемпотентными и безопасными при конкуренции).

**B. Отдельный worker-деплой для истечения/консьюмера.** Отклонён: лишний деплой ради двух лёгких процессов; согласуется с ADR-018 (воркеры в процессе сервиса).

**C. Активация без inbox (полагаться только на guard `status=pending`).** Отклонён: guard делает переход идемпотентным, но inbox нужен, чтобы не переигрывать бизнес-эффекты (событие в outbox) на дублях доставки и корректно отмечать «уже видели»; повторяет проверенную в Media схему.

**D. Планировщик БД (pg_cron) для истечения.** Отклонён: внешняя зависимость и разнесение логики; тикер в сервисе проще, наблюдаем (лог/метрики) и переносим.

### Детали решения

- Событийная модель Listing: публикует `bozor.ad.created` (создание, draft), `bozor.ad.updated` (submit→pending, активация, reject, renew, archive, block), `bozor.ad.sold`, `bozor.ad.expired`; потребляет `bozor.ad.approved|rejected`. Полезная нагрузка едина (`app.AdEvent`, без PII) и несёт `status` + `published_at`/`expires_at` для индексации в Search.
- Статусная машина дополнена: `active→pending` (повторная модерация после правки активного), `expired→active` (реактивация продлением). `renew` активного — сдвиг `expires_at` без смены статуса; истёкшего — реактивация.
- Разграничение: **владелец** двигает submit/sold/renew/archive (право по `user_id`); **модерация** — approve/reject (через события) и blocked; истечение — воркер. Правка ключевых полей активного (→ повторная модерация) и REST PATCH/DELETE/лента — за Stage 3.6.
- Идемпотентность/гонки: все переходы — условный `UPDATE ... WHERE status=$from` (+ `expires_at<=now()` для истечения); `published_at`/`expires_at` не сбрасываются в NULL (COALESCE), лишь проставляются. Миграция `00002_listing_inbox` (`processed_events`, DDL синхр. с pkg/shared/outbox).

### Последствия

- Объявление проходит полный жизненный цикл; решения модерации применяются «ровно один раз», истечение автоматическое; каждое изменение статуса — событие в шине (Search/Notification подхватят). Проверено вживую (см. STATUS 3.4): approve→active(+published/expires)+`bozor.ad.updated`; дубль решения не плодит эффектов (inbox); reject→rejected и повторная модерация rejected→pending; renew active (сдвиг срока) и expired→active; sold/archive; forbidden 403; недопустимый переход 409; воркер active→expired+`bozor.ad.expired`.
- **Условия пересмотра:** появление Moderation (CHAPTER 6) уточнит контракт `bozor.ad.approved|rejected` (причина отклонения и т.п.); масштабирование — вынести истечение в выделенный воркер/лидер-выборы, если тикеры в репликах создадут заметную нагрузку; Stage 3.6 добавит REST-поверхность и правку-с-повторной-модерацией поверх этой статусной машины.

## ADR-021: Счётчик просмотров — буфер в Redis (хэш + атомарный Lua-drain), пакетный флеш в PostgreSQL, чтение с учётом буфера

- **Статус:** accepted
- **Дата:** 2026-07-09
- **Автор:** AI-инженер, по согласованию с ТЗ (ARCHITECTURE §4.5; ROADMAP Stage 3.5)

### Контекст

Просмотр карточки объявления должен инкрементировать счётчик, но прямой `UPDATE ads SET views_count = views_count + 1` на каждый просмотр создаёт write-hotspot: популярное объявление порождает поток одиночных апдейтов одной строки (блокировки, раздувание, нагрузка на БД при десятках тысяч DAU). ARCHITECTURE §4.5 предписывает буферизацию в Redis (`INCR` по ad_id) с периодическим флешем агрегатов в PostgreSQL. Listing уже имеет фоновые воркеры (ADR-020) и relay; Redis используется другими сервисами (кеш каталога, сессии, rate-limit).

### Рассмотренные варианты

**A. Хэш `listing:views` в Redis + атомарный Lua-drain + пакетный UPDATE + чтение с учётом буфера (выбрано).**
- Просмотр (`GET /ads/{id}`): `HINCRBY listing:views <ad_id> 1` — весь буфер в одном ключе-хэше. Воркер флеша (`LISTING_VIEWS_FLUSH_INTERVAL`, дефолт 30s) атомарно снимает и обнуляет буфер Lua-скриптом (`HGETALL` + `DEL` в одном EVAL — однопоточность Redis гарантирует отсутствие потери инкрементов: пришедшие после снятия попадают в новый ключ) и прибавляет агрегаты к `ads.views_count` одним batch-UPDATE (`UPDATE … FROM (VALUES …)`), устраняя write-hotspot. Чтение объявления возвращает `views_count = персистентное + буферизованное` (`HGET`) — near-real-time без ожидания флеша.
- Устойчивость: инкремент/чтение буфера — best-effort (сбой Redis логируется, но не ломает чтение объявления — просмотры не критичны); при сбое записи в БД снятые счётчики возвращаются в буфер (`HINCRBY`, повтор на следующем тике).
- Плюсы: один ключ Redis (компактно, атомарный снимок всего буфера), пакетный апдейт вместо потока одиночных, near-real-time чтение, отказоустойчивость к обеим сторонам (Redis/БД). Минусы: редкий двойной/потерянный подсчёт на стыке флеша (снятие атомарно, но отказ БД после снятия + отказ возврата → потеря; отказ возврата крайне маловероятен) — для просмотров приемлемо.

**B. Ключ на объявление (`INCR listing:views:<id>`) + `GETDEL` при флеше.** Отклонён: флеш требует `SCAN` по множеству ключей (дороже атомарного снимка одного хэша), а `GETDEL` теряет инкременты, пришедшие между чтением и удалением; хэш+Lua решает обе проблемы.

**C. Прямой `UPDATE` на каждый просмотр (без буфера).** Отклонён: ровно тот write-hotspot, который этап устраняет.

**D. Инкремент через отдельный эндпоинт `POST /ads/{id}/view`.** Отклонён для 3.5: backend-only, просмотр карточки = `GET /ads/{id}` (лента `GET /ads` в 3.6 просмотры не считает). Выделенный эндпоинт/дедуп по зрителю — возможная будущая доводка.

### Детали решения

- Пакет `internal/views` (Counter поверх `redis.Cmdable`): `Incr`/`Buffered`/`Drain` (Lua)/`Restore`. Воркер `internal/worker` (`ViewFlusher`) — тикер `Drain → AddViews → (при ошибке) Restore`.
- `repo.AddViews(map[ad_id]delta)` — один `UPDATE ads … FROM (VALUES …)`; несуществующие id молча отбрасываются; пустая карта — no-op.
- `app.Service.Get` учитывает просмотр (best-effort `Incr` + добавление `Buffered` к ответу); `ViewCounter` может быть nil (счётчик отключён). Ответ API дополнен `views_count`.
- Конфиг: `REDIS_ADDR`/`REDIS_PASSWORD`, `LISTING_VIEWS_FLUSH_INTERVAL`; readyz включает ping Redis.

### Последствия

- Просмотры копятся в Redis и сливаются пачкой — карточка популярного объявления не порождает поток одиночных UPDATE (проверено вживую: 3 GET → `views_count` 1/2/3 из буфера при `views_count=0` в БД; через флеш БД=3, буфер очищен; следующий GET=4 = персистентное 3 + буфер 1; лог «просмотры слиты в БД ads=1»).
- **Условия пересмотра:** дедуп просмотров по зрителю/окну (антинакрутка) — при необходимости; при росте числа объявлений в буфере — шардирование ключа/`HSCAN`-пакетирование флеша; экспорт `views_count` в Search (сортировка по популярности) — событием при флеше, если понадобится.

## ADR-022: API объявлений — частичный PATCH с повторной модерацией ключевых полей, hard-delete с событием, лента из БД до Search

- **Статус:** accepted
- **Дата:** 2026-07-09
- **Автор:** AI-инженер, по согласованию с ТЗ (ARCHITECTURE §4.5; ROADMAP Stage 3.6)

### Контекст

Stage 3.6 завершает CHAPTER 3: полный REST-контур объявлений (`POST/GET/PATCH/DELETE /ads`, лента `GET /ads`, `GET /me/ads`) и полный набор событий `bozor.ad.*`. Часть уже была: создание (3.3), жизненный цикл submit/sold/renew/archive + консьюмер модерации (3.4), счётчик просмотров (3.5). Остались правка, удаление и списки. Ключевые вопросы: как править (частично/целиком), когда правка active шлёт на повторную модерацию, чем удалять (soft/hard), и как отдавать ленту до появления Search (CHAPTER 4).

### Рассмотренные варианты

**A. Частичный PATCH (nil=не менять) + повторная модерация по ключевым полям; hard-delete + bozor.ad.deleted; лента из БД (status=active) до Search (выбрано).**
- PATCH: поля-указатели (nil — не трогаем), атрибуты/изображения — replace-all при наличии; после применения — ре-валидация ValidateCore/ValidateImages + значений атрибутов против (возможно новой) категории Catalog (gRPC). Правка **ключевых** полей (заголовок, описание, цена, категория, атрибуты, изображения) активного объявления → `active → pending` (повторная модерация, ARCHITECTURE §4.5); неключевые (телефон, регион/город, гео) статус не меняют. Терминальные/blocked не редактируются (ErrNotEditable→409).
- DELETE: физическое удаление строки (каскад на `ad_attribute_values`/`ad_images` по FK) + `bozor.ad.deleted` в одной tx — Search снимет индекс, Media освободит привязку по TTL. Только владелец.
- Лента `GET /ads`: выборка активных из БД (фильтры category/region, сортировка bumped/published/цена, пагинация limit/offset) без дочерних сущностей — карточка ленты не требует полного набора; детальный `GET /ads/{id}` отдаёт всё. Полнотекст/фасеты/typo/гео — в Search (CHAPTER 4) поверх read-модели Typesense.
- Плюсы: минимальные полезные нагрузки PATCH; консистентность значений атрибутов после правки; событийная согласованность (updated/deleted); лента доступна сразу, без блокировки на Search. Минусы: DB-лента ограничена (без релевантности/фасетов) — временно, до 4.x; hard-delete необратим (для истории — событие в шине).

**B. Полная замена (PUT) вместо PATCH.** Отклонён: клиент обязан слать всё состояние; частичная правая цены/статуса неудобна и чревата гонками перезаписи.

**C. Soft-delete (статус `deleted`/`archived`).** Отклонён для удаления пользователем: archived уже есть для «убрать из выдачи, сохранить»; DELETE подразумевает удаление, событие даёт потребителям снять производные данные. (Возвраты/удаление модерацией с сохранением истории — отдельная логика CHAPTER 6/8.)

**D. Лента только через Search с самого начала.** Отклонён: Search — CHAPTER 4; блокировать базовую ленту на нём нельзя, а простая DB-выборка активных закрывает потребность DoD CHAPTER 3 «API все объявления».

### Детали решения

- Изменяемость: `Status.Editable()` = draft|pending|rejected|active. `applyUpdate` возвращает признак изменения ключевого поля → пересчёт статуса. `repo.UpdateWithEvent` — UPDATE полей + replace-all детей + outbox одной tx.
- Пагинация: limit по умолчанию 20, максимум 100; offset ≥ 0 (клампинг в app). Сортировка ленты по умолчанию `COALESCE(bumped_at, published_at) DESC` (свежесть/поднятия), плюс `price_asc|price_desc`. `GET /me/ads` — по `created_at DESC`, опциональный фильтр `status` (невалидный → 422).
- Полный набор событий Listing: `created` (3.3), `updated` (submit/approve/reject/renew/archive/edit), `sold`, `expired` (3.4), `deleted` (3.6) — все через transactional outbox. `GET /ads` и `GET /ads/{id}` публичны; PATCH/DELETE/`/me/ads` требуют владельца.

### Последствия

- CHAPTER 3 (Listings & Media) завершён: создание с фото и атрибутами, полный жизненный цикл, счётчик просмотров, правка с повторной модерацией, удаление, лента и «мои объявления», полный `bozor.ad.*`. Проверено вживую (см. STATUS 3.6): PATCH title draft→draft; PATCH price active→pending (повторная модерация) + событие; PATCH sold→409 not_editable; чужой→403, аноним→401; DELETE→204→GET 404 + `bozor.ad.deleted`; лента активных с сортировкой/фильтром региона; `/me/ads` по статусу, аноним→401, невалидный статус→422; все события published в NATS.
- **Условия пересмотра:** с приходом Search (CHAPTER 4) лента `GET /ads` может проксироваться/дополняться выдачей Typesense (релевантность, фасеты, гео, топ-блок); при необходимости истории удалений — перейти на soft-delete/архив событий; курсорная пагинация вместо offset при больших офсетах.

## ADR-023: Search-сервис — read-модель Typesense (CQRS-lite), тонкий HTTP-клиент без внешней зависимости, коллекция ads

- **Статус:** accepted
- **Дата:** 2026-07-09
- **Автор:** AI-инженер, по согласованию с ТЗ (ARCHITECTURE §4.7; ROADMAP Stage 4.1)

### Контекст

CHAPTER 4 вводит поиск: полнотекст (uz/ru, typo), фасеты, сортировки, гео — на Typesense (ADR из STATUS: Typesense, не OpenSearch). Это read-модель (CQRS-lite): источник истины — Listing (Postgres), Search держит денормализованную проекцию объявлений в коллекции Typesense, обновляемую по событиям `bozor.ad.*` (индексатор — Stage 4.2). Stage 4.1 — каркас сервиса и схема коллекции `ads`. Нужен способ общаться с Typesense из Go.

### Рассмотренные варианты

**A. Тонкий собственный HTTP-клиент Typesense (net/http) + схема коллекции ads как Go-структуры (выбрано).**
- `internal/search`: Client (Health, EnsureCollection, далее Upsert/Delete/Search) поверх net/http; схема `AdsSchema()` — типизированные структуры Field/CollectionSchema, сериализуемые в JSON REST-контракта Typesense. Сервис при старте идемпотентно создаёт коллекцию (GET → 404 → POST; 409 при гонке реплик — ок).
- Плюсы: ноль внешних клиентских зависимостей (Typesense REST прост и стабилен), полный контроль над запросами/ошибками/таймаутами, лёгкая тестируемость через httptest, единый стиль с остальным кодом. Соответствует духу проекта (минимум codegen/зависимостей — ср. ADR-005 про sqlc).
- Минусы: реализуем нужные вызовы сами (health/collection/index/search — небольшой объём). Приемлемо: поверхность API, которую использует сервис, невелика и покрыта тестами.

**B. Официальный `typesense-go`.** Отклонён: тянет крупную зависимость с собственной версионностью под версии сервера; для нашего узкого набора операций выгода мала, а связность с внешним релизным циклом растёт. Возврат к нему — если понадобится большой объём продвинутых возможностей API.

**C. PostgreSQL FTS вместо Typesense.** Уже отклонён на уровне проекта (STATUS decisions): Typesense даёт typo-tolerance/фасеты/гео из коробки; здесь лишь подтверждаем.

### Детали решения

- Коллекция `ads` (read-модель, ARCHITECTURE §4.7): `title`/`description` (string, полнотекст), `category_id`/`category_path`/`currency`/`status` (string-фасеты), `region_id` (int32-фасет)/`city_id` (int64), `price` (int64, сортируемое), `attrs` (object, фасет; `enable_nested_fields=true` — фасеты по динамическим `attrs.<slug>`), `created_at`/`bumped_at`/`promotion_rank` (сортируемые), `is_top` (bool-фасет), `location` (geopoint). `default_sorting_field=created_at`.
- Заголовок/описание — **одно** поле на объявление (не uz/ru-пара): текст объявления пользовательский, произвольного языка; двуязычие покрывается токенизацией/typo Typesense, а не раздвоением полей (в отличие от справочников Catalog). Locale можно донастроить позже пофибо.
- Идемпотентность старта: EnsureCollection не пересоздаёт существующую коллекцию (схема стабильна); изменения схемы — через пересоздание + полная переиндексация (инструмент reindex, 4.2). Readyz включает Health Typesense.
- Каркас: config (TYPESENSE_HOST/PORT/API_KEY, fail-fast на ключе), transport (health/ready/metrics; API поиска — 4.3), main (otel/metrics/ensure-collection/serve), Dockerfile distroless, compose (dep typesense healthy, dev-порт 8086, scrape-job).

### Последствия

- Появилась read-модель поиска: коллекция `ads` в Typesense создаётся при старте, сервис наблюдаем и готов принимать документы (индексатор — 4.2) и запросы (API — 4.3). Проверено вживую: search healthy, лог «коллекция ads готова created=true», рестарт → created=false (идемпотентно), readyz ready (health Typesense), схема в Typesense содержит все поля (enable_nested_fields=true, geopoint, default_sorting_field=created_at).
- **Условия пересмотра:** маршрут `GET /api/v1/ads/search` в gateway пересекается с префиксом `/api/v1/ads` (Listing) — развести в 4.3 (более специфичный маршрут на search); при необходимости продвинутых возможностей Typesense — вернуться к официальному клиенту (вариант B); при мультиязычных полях — ввести locale/раздельные поля.

## ADR-024: Индексатор Search — событие как триггер + чтение актуального состояния из Listing, реконсиляция Typesense, reindex через внутренний экспорт

- **Статус:** accepted
- **Дата:** 2026-07-09
- **Автор:** AI-инженер, по согласованию с ТЗ (ARCHITECTURE §4.7; ROADMAP Stage 4.2)

### Контекст

Read-модель Search (ADR-023) должна наполняться из объявлений Listing по событиям `bozor.ad.*`. События Listing (`app.AdEvent`) несут минимум (id, статус, цена, категория, регион, заголовок) — без описания, атрибутов, города, гео, — которых требует полноценный индекс (полнотекст описания, фасеты `attrs.*`, гео). Нужен способ получить полную проекцию и не потерять консистентность при переупорядочивании доставки (at-least-once JetStream).

### Рассмотренные варианты

**A. Событие — триггер; индексатор читает АКТУАЛЬНОЕ состояние объявления из Listing и сводит документ (upsert active / delete иначе); reindex — через внутренний постраничный экспорт (выбрано).**
- На любое `bozor.ad.*` индексатор берёт из события только `ad_id`, запрашивает полную проекцию у Listing (`GET /internal/ads/{id}`), и: не найдено/не active → удалить документ; active → upsert. Документ Typesense (id=ad_id) — идемпотентная запись.
- Устойчивость к переупорядочиванию: каждое событие сводит индекс к ТЕКУЩЕЙ истине (не применяет дельту из устаревшего события). Идемпотентность — по id документа (upsert/delete). Отсутствие БД у Search: не нужен inbox — Typesense-операции сами идемпотентны.
- Reindex: одноразовый режим бинаря (`-reindex`) постранично (keyset по id) читает `GET /internal/ads/export` (активные с полными данными) и upsert'ит — восстановление индекса с нуля.
- Внутренние read-эндпоинты Listing — HTTP (не под `/api/v1`, gateway их наружу не проксирует; только внутренняя сеть), просмотр НЕ учитывается (в отличие от публичного `GET /ads/{id}`). Плюсы: одна кодовая ветка построения документа для live и reindex; события остаются минимальными (не раздуваем контракт ради Search); строгая консистентность. Минусы: HTTP-запрос к Listing на каждое индекс-событие (объём мал — только смены статуса, не просмотры) и связность Search→Listing на чтении.

**B. Обогатить события Listing полной проекцией (event-carried state transfer).** Отклонён: раздувает контракт `bozor.ad.*` для всех потребителей ради нужд Search; при переупорядочивании устаревшее событие может перезаписать индекс более старым состоянием (нет чтения текущей истины); для reindex всё равно нужен источник полного списка.

**C. Индексатор с собственной БД/inbox для дедупа и хранения проекции.** Отклонён: лишнее состояние; идемпотентность даёт сам Typesense (по id документа), чтение истины из Listing снимает вопрос порядка.

**D. gRPC для Search→Listing (по ADR-019).** Рассмотрен: ADR-019 предписывает gRPC для синхронных межсервисных вызовов. Здесь выбран HTTP для ВНУТРЕННЕГО read/export (bulk-переиндексация + простое чтение проекции) как прагматичный минимум без proto-обвязки; при росте требований (стриминг, строгий контракт) — мигрировать на gRPC. Зафиксировано как отступление с обоснованием.

### Детали решения

- Search: `listingclient` (GetAd→404=удалено, ListActive keyset), `indexer` (Handle: sync(ad_id); Reindex: постранично), `search.Client` += UpsertDocument (action=upsert)/DeleteDocument (404=ok). Консьюмер `search-indexer` (durable, maxDeliver=5→DLQ) на `bozor.ad.created|updated|deleted|approved|bumped|sold|expired`. Документ: title/description/category_id/region_id/city_id/price/currency/status/attrs(map)/created_at(unix)/bumped_at/is_top/location([lat,lng]).
- Listing: `repo.ListActiveFull` (active, keyset по id, с детьми), `app.ExportByID`/`ExportActive`, внутренние `GET /internal/ads/{id}` и `GET /internal/ads/export` (DTO exportAd, без учёта просмотра).

### Последствия

- Индекс поддерживается в согласии с источником истины по событиям, устойчиво к переупорядочиванию; reindex восстанавливает индекс с нуля. Проверено вживую: создание+одобрение объявления → документ в Typesense со всеми полями (attrs.brand/year, гео, created_at unix), полнотекст «prado» → found=1 с подсветкой (typo/токенизация); продажа → документ снят из индекса (4→3, GET 404); reindex после очистки индекса (3→0) восстановил 3 документа (лог «переиндексация выполнена documents=3»); active в БД = документов в Typesense.
- **Условия пересмотра:** при высоком темпе событий — кэшировать/батчить чтение Listing или перейти на обогащённые события с версионированием; внутренний read Listing→gRPC (ADR-019), если понадобится строгий контракт/стриминг; версионное поле в документе при риске реального переупорядочивания на уровне брокера.

## ADR-025: API поиска — app-слой строит запрос Typesense за интерфейсом, санитизация фильтров, разведение маршрута /api/v1/ads/search в gateway

- **Статус:** accepted
- **Дата:** 2026-07-09
- **Автор:** AI-инженер, по согласованию с ТЗ (ARCHITECTURE §4.7; ROADMAP Stage 4.3)

### Контекст

Поверх наполняемой read-модели (ADR-023/024) нужен публичный контракт поиска: `GET /api/v1/ads/search` (полнотекст uz/ru с typo-tolerance, фильтры категория/регион/город/цена/атрибуты, сортировки релевантность/цена/дата/bumped/дистанция, пагинация, гео) и `GET /api/v1/ads/search/facets`. Три вопроса требовали решения: (1) где строить запрос Typesense, (2) как безопасно собирать `filter_by` из пользовательского ввода, (3) как развести публичный путь `/api/v1/ads/search` от префикса Listing `/api/v1/ads` в gateway (зафиксировано в ADR-024 как долг Stage 4.3).

### Рассмотренные варианты

**(1) Слой построения запроса.**
- **A. Прикладной `app`-слой за интерфейсом `Docs` (выбрано).** `transport` парсит query→доменный `Query`; `app` переводит `Query` в `search.SearchParams` (filter_by/sort_by/facet_by) и проецирует результат в доменные `AdHit`/`Facet`; `search.Client.Search` — только HTTP к Typesense. Публичный контракт не зависит от движка: замена Typesense (OpenSearch, PG FTS) не трогает transport и тесты (buildParams/buildFilter/buildSort покрыты юнитами без сети). Согласуется с ADR-022 (слой поиска за интерфейсом).
- **B. Строить запрос Typesense прямо в transport.** Отклонён: привязывает HTTP-контракт к движку, дублирует сборку в хендлерах, труднее тестировать сборку фильтров изолированно.

**(2) Безопасность `filter_by`.** Typesense filter_by — это выражение с операторами (`&&`, `:=`, `>=`, `:()`), собираемое из пользовательских значений. Решение: строковые значения оборачиваются в бэктики (точное совпадение, безопасно к пробелам/спецсимволам), а сами бэктики вырезаются из значения (нельзя выйти из литерала); имена динамических полей `attrs.<slug>` санитизируются до `[a-z0-9_-]` (слаг не может внести оператор/поле). Числа форматируются через strconv. Мусорные query-параметры парсятся лениво (→0/nil), поиск устойчив к шуму. Ошибка движка отдаётся как `503 search_unavailable` без утечки деталей Typesense наружу.

**(3) Разведение маршрута в gateway.**
- **A. Более специфичный префикс `/api/v1/ads/search` → search перед `/api/v1/ads` → listing-ads (выбрано).** chi (radix-trie) отдаёт приоритет статическому сегменту `search` над wildcard `/ads/*`, независимо от порядка регистрации; устаревший `/api/v1/search` удалён. Публичный путь совпадает с ARCHITECTURE §4.7 (`GET /api/v1/ads/search`).
- **B. Отдельный хост/поддомен для поиска.** Отклонён: избыточно для монолитного compose; ломает единый `/api/v1/*` контракт.
- **C. Разведение по порядку регистрации.** Отклонён: хрупко; chi и так решает по специфичности, порядок не нужен.

### Детали решения

- `search.Client.Search` (GET /collections/ads/documents/search): собирает `q/query_by/query_by_weights/filter_by/sort_by/facet_by/max_facet_values/page/per_page`, декодирует `found/hits/facet_counts`.
- `app`: `Query`→`buildParams` (q='*' при пустом тексте, query_by=`title,description` веса `2,1`), `buildFilter` (бэктики+санитизация), `buildSort` (relevance без текста→дата), `Facets` (facet_by=category_id,region_id,city_id,currency + attrs.<slug>, per_page=1, max_facet_values=50); `clampPage`(≥1)/`clampPerPage`(20/100).
- `transport`: `GET /api/v1/ads/search[/facets]` (публично), `parseQuery` (attr.<slug>=value→Attrs, facets=csv, lat+lng[+radius_km]→Geo), DTO `searchResponse`/`facetsResponse`.
- gateway: `Routes` += `/api/v1/ads/search`→search (перед `/api/v1/ads`), удалён `/api/v1/search`.

### Последствия

- Публичный поиск работает через gateway и напрямую; движок изолирован. Проверено вживую (3 активных объявления в индексе): полнотекст `q=feed`→found=2 (префикс токена), typo `q=feod`→found=2 (`q=fed`<4 симв.→0 — порог Typesense `min_len_1typo=4`, ожидаемо); фильтры region_id/price-диапазон/attr.brand; сортировки price_asc/desc (1/200/500); фасеты по category/region/currency/attrs.brand (toyota:2, other:1); разведение маршрутов: `/api/v1/ads/search`→search (поле `found`), `/api/v1/ads/{id}` и лента→listing (`user_id`/`limit`).
- **Условия пересмотра:** при мультиязычных полях — раздельные `title_uz/title_ru`/locale (сейчас одно поле, двуязычие токенизацией); фасеты атрибутов сейчас запрашиваются клиентом (`facets=`) — при необходимости строить из эффективного набора Catalog; синонимы/ранжирование промо (is_top/promotion_rank) — Stage 4.4 и CHAPTER 8.

## ADR-026: Топ-блок выдачи — отдельное поле `top`, наполняется отдельным запросом is_top на первой странице (заглушка до Promotions)

- **Статус:** accepted
- **Дата:** 2026-07-09
- **Автор:** AI-инженер, по согласованию с ТЗ (ROADMAP Stage 4.4; §8.6 — ротация Топа)

### Контекст

ROADMAP 4.4 требует подготовить в выдаче поиска место для featured/TOP-объявлений до появления сервиса Promotions (CHAPTER 8). Схема коллекции `ads` уже содержит `is_top` (bool-фасет) и `promotion_rank` (int32, sortable, optional) с 4.1, но пока их никто не проставляет (Promotions ещё нет). Нужно ввести контракт топ-блока сейчас, чтобы CHAPTER 8 включил промо без изменения публичного API.

### Рассмотренные варианты

**A. Отдельное поле `top` в ответе `GET /ads/search`, наполняется отдельным запросом (выбрано).** На первой странице (`page<=1`) сервис делает второй запрос к Typesense: фильтры основного запроса + `is_top:=true`, `q='*'` (топ-блок зависит от категории/региона/фильтров, а не от свободного текста — это врезка промо, не полнотекст), сортировка `promotion_rank:desc,created_at:desc`, `per_page=5` (ротация «макс. 5 на категорию», §8.6). Ответ получает поле `top` рядом с `hits`. Пока `is_top=false` у всех документов → блок пуст; CHAPTER 8 начнёт проставлять `is_top/promotion_rank` — API не меняется.
- Плюсы: контракт готов сейчас; основная выдача и топ-блок разделены (разное ранжирование/лимиты); клиент рисует врезку промо отдельно; на страницах >1 второй запрос не выполняется (экономия round-trip, промо-врезка логически вверху).
- Минусы: второй запрос к Typesense на первой странице (дёшево — фильтр по фасету, лимит 5).

**B. Подмешивать TOP в основную выдачу через ранжирование (boost is_top).** Отклонён для заглушки: смешивает промо с органикой (сложнее показать «блок» и посчитать промо-показы), ранжирующий boost — вопрос CHAPTER 8; здесь нужен именно выделенный блок.

**C. Отдельный эндпоинт `GET /ads/top`.** Отклонён: клиенту пришлось бы делать два запроса и синхронизировать фильтры; поле в общем ответе проще и атомарнее по фильтрам.

**D. Ждать CHAPTER 8 без задела в контракте.** Отклонён: тогда включение промо изменит форму ответа поиска (ломающее изменение для клиентов).

### Детали решения

- `app`: `Result.Top []AdHit`; `Service.Search` при `clampPage(q.Page)==1` вызывает `topBlock` → `buildTopParams(q)` (фильтр запроса + `is_top:=true`, `q='*'`, `sort=promotion_rank:desc,created_at:desc`, `per_page=TopBlockSize=5`); `decodeHits` — общий для основной выдачи и блока.
- `transport`: `searchResponse.Top []adHitDTO` (`toHitDTOs` общий); непусто лишь на первой странице при наличии TOP-объявлений.
- Индексатор/схема без изменений: `is_top` пишется (пока `false`), `promotion_rank` optional. Топ-блок ленты Listing — вне охвата (is_top живёт в Search-индексе; лента — §8.6).

### Последствия

- Публичный ответ поиска содержит `top` уже сейчас; включение промо в CHAPTER 8 (проставление `is_top/promotion_rank` на купивших TOP объявлениях + индексатор) наполнит блок без правок API. Проверено вживую: по умолчанию `top:[]`; после ручного upsert документа с `is_top=true` в категории — он появился в `top` при поиске с фильтром этой категории; на `page=2` блок пуст; после удаления демо-документа блок снова пуст, индекс = 3.
- **Условия пересмотра:** CHAPTER 8.6 — случайная ротация (Typesense не сортирует случайно без спец-поля: ввести поле рандом-ключа или пост-перемешивание), лимит на категорию, промо-показы/метрики; при высокой нагрузке — кэш топ-блока по (категория+фильтры) с коротким TTL.

## ADR-027: User/Profile — отдельный сервис, профиль по событию + ленивое создание, «отсутствие=включено» для настроек уведомлений, разведение /me/ads в gateway

- **Статус:** accepted
- **Дата:** 2026-07-09
- **Автор:** AI-инженер, по согласованию с ТЗ (ARCHITECTURE §4.2; ROADMAP Stage 5.1)

### Контекст

Нужен расширенный профиль пользователя: отображаемые данные, тип аккаунта (физлицо/бизнес), настройки уведомлений, кеш агрегированного рейтинга. Идентичность (номер, Telegram) уже живёт в Auth. Вопросы: (1) где хранить профиль, (2) как и когда его создавать, (3) как представить настройки уведомлений, (4) как развести публичный `/me/ads` (Listing) от `/me` (Profile) в gateway, (5) что делать с кешем рейтинга до появления Reviews.

### Рассмотренные варианты и решения

**(1) Отдельный сервис `user-profile` с БД `bozor_profile` (выбрано).** db-per-service: Auth хранит идентичность и сессии, Profile — расширенные данные. Отклонён «профиль внутри Auth»: смешал бы аутентификацию с профильными данными, раздул бы Auth и его БД.

**(2) Создание профиля по `bozor.user.created` + ленивое создание при первом `/me` (выбрано).** Консьюмер `user-profile-users` создаёт профиль по умолчанию (idempotent: `INSERT … ON CONFLICT (user_id) DO NOTHING` + inbox `processed_events` — exactly-once). Дополнительно `GET/PATCH /me` создаёт дефолт лениво (та же идемпотентная вставка), если событие ещё не обработано — устойчивость к лагу шины (пользователь с валидным JWT не получит 404 на свой профиль). Публичный `GET /users/{id}` НЕ создаёт лениво (иначе плодил бы профили на произвольные id) — отдаёт 404. Отклонён «только по событию»: гонка регистрация→/me давала бы 404.

**(3) Настройки уведомлений: хранить только отклонения, «отсутствие строки = включено» (выбрано).** `GET` возвращает эффективный набор = дефолты (все известные типы, включены) ⊕ сохранённые строки; `PUT` заменяет набор пользователя. Добавление нового типа уведомления не требует миграции данных (для всех сразу «включено» по умолчанию). Отклонено «материализовать полный набор строками на каждого пользователя»: лишние записи и миграции при расширении каталога типов.

**(4) Разведение `/me/ads` от `/me` в gateway более специфичным префиксом (выбрано).** Listing обслуживает `/api/v1/me/ads` (мои объявления), Profile — `/api/v1/me` и `/api/v1/me/notification-prefs`. В таблице маршрутов gateway добавлен `/api/v1/me/ads → listing-ads` перед `/api/v1/me → user-profile`; chi (radix-trie) отдаёт приоритет статическому сегменту независимо от порядка (тот же приём, что `/ads/search` в ADR-025). Отклонено «разводить в самих сервисах»: маршрутизация — ответственность gateway.

**(5) Кеш рейтинга: таблица и read-путь сейчас, consumer — в CHAPTER 9 (выбрано).** `user_ratings_cache` создаётся в 5.1, `GET /users/{id}` читает её (0/0 при отсутствии). Потребление `bozor.review.created` отложено: Reviews (CHAPTER 9) ещё не публикует событие, его схема не зафиксирована. ROADMAP-скоуп 5.1 — только `bozor.user.created`. Отклонено «подписаться на неопределённое событие сейчас»: пришлось бы гадать payload.

### Детали решения

- Модель: `profiles` (user_id PK, тип individual|business [business — на вырост, витрины вне v1], контактная видимость, язык), `notification_prefs` (PK user_id+channel+event_type), `user_ratings_cache`, `outbox`+`processed_events`.
- Слои: domain (Validate, дефолты, каталог типов уведомлений, эффективный набор), repo (идемпотентные вставки, UPDATE+outbox, replace prefs), app (Me/UpdateMe/PublicProfile/Prefs; ленивое создание; очистка nullable пустой строкой/неположительным id; валидация аватара как UUID), worker (консьюмер user.created), transport (`/me`, `/users/{id}`, `/me/notification-prefs`).
- События: потребляет `bozor.user.created`; публикует `bozor.user.updated` (user_id/display_name/user_type/city_id, без PII) через outbox.

### Последствия

- Профиль автоматически появляется при регистрации и устойчив к лагу шины; настройки уведомлений расширяемы без миграций; gateway корректно разводит профиль и мои-объявления. Проверено вживую: событие `bozor.user.created` → профиль создан (идемпотентно; повтор event_id не перезаписывает); `/me` GET/PATCH (business-валидация, UTF-8 имя), `/me/notification-prefs` GET/PUT (выключение типа, отказ неизвестного канала), публичный `/users/{id}` с рейтингом 0/0 и 404 для неизвестного; ленивое создание для нового пользователя; `bozor.user.updated` ушло в шину; маршрутизация `/me/ads`→listing и `/me`→profile подтверждена по логам.
- **Условия пересмотра:** CHAPTER 9 — подписка на `bozor.review.created` (наполнение кеша рейтинга); при добавлении каналов (push/email) — в каталог `notificationChannels`; язык профиля дублирует Auth — при рассинхроне выбрать единый источник; business-витрины (storefront) — отдельный эпик вне v1.

## ADR-028: Favorites — избранное без проверки существования объявления, очистка по bozor.ad.deleted, лента без обогащения

- **Статус:** accepted
- **Дата:** 2026-07-09
- **Автор:** AI-инженер, по согласованию с ТЗ (ARCHITECTURE §4.8; ROADMAP Stage 5.2)

### Контекст

Избранное: `POST/DELETE /api/v1/favorites/{adId}`, `GET /api/v1/me/favorites`. Сервис `favorites-savedsearch` (БД `bozor_favorites`) — общий с сохранёнными поисками (5.3). Вопросы: (1) проверять ли существование объявления при добавлении, (2) как убирать записи на удалённые объявления, (3) отдавать ли обогащённые данные объявления в ленте, (4) как развести `/me/favorites` от `/me` (Profile) в gateway.

### Рассмотренные варианты и решения

**(1) Не проверять существование объявления при добавлении (выбрано).** `POST /favorites/{adId}` пишет пару (user_id, ad_id) без синхронного вызова Listing/Search — нет связности и латентности на записи. `Add` идемпотентен: `INSERT … ON CONFLICT (user_id, ad_id) DO UPDATE SET created_at = favorites.created_at RETURNING created_at` (повтор сохраняет исходное время). Отклонено «валидировать через gRPC/HTTP к Listing»: связность и задержка ради пограничного случая (избранное несуществующего id безвредно и вычищается).

**(2) Очистка по `bozor.ad.deleted` (выбрано).** Консьюмер `favorites-ad-deleted` на `bozor.ad.deleted` делает `DELETE FROM favorites WHERE ad_id = $1` — идемпотентно (повторная доставка безопасна), поэтому inbox/`processed_events` не нужен. ARCHITECTURE (таблица событий) уже назначает Favorites потребителем `bozor.ad.deleted`. Проданные/истёкшие (`bozor.ad.sold|expired`) НЕ удаляются из избранного — пользователь может хотеть видеть их (только hard-delete чистит).

**(3) Лента отдаёт `ad_id` без обогащения (выбрано для 5.2).** `GET /me/favorites` возвращает `ad_id + created_at` (свежие сверху, пагинация). Обогащение (title/price/фото) потребовало бы N запросов к Listing или batch-multiget к Search — отложено (db-per-service; можно добавить позже batch-обогащением через Search). Отклонено «обогащать в сервисе сейчас»: преждевременная связность; у Search пока нет multiget по id.

**(4) Разведение `/me/favorites` от `/me` более специфичным префиксом в gateway (выбрано).** `/api/v1/me/favorites → favorites-savedsearch` добавлен перед `/api/v1/me → user-profile`; chi отдаёт приоритет статическому сегменту (тот же приём, что `/me/ads`, `/ads/search`).

### Детали решения

- Модель: `favorites` (PK user_id+ad_id, created_at; idx user+created_at DESC для ленты, idx ad_id для очистки). Без outbox/inbox (нет исходящих событий; очистка идемпотентна).
- Слои: domain (Favorite), repo (Add/Remove/ListByUser/RemoveByAd), app (валидация ad_id как UUID, пагинация-clamp), worker (консьюмер bozor.ad.deleted), transport (`/favorites/{adId}`, `/me/favorites`).

### Последствия

- Избранное работает без связности на записи; удалённые объявления вычищаются событием; лента лёгкая. Проверено вживую: POST (201, идемпотентный повтор — то же created_at), невалидный id→422, аноним→401, лента свежие-сверху, DELETE→204, `bozor.ad.deleted` убрал запись (лог removed=1); маршрутизация `/me/favorites`→favorites подтверждена по логам.
- **Условия пересмотра:** обогащение ленты (batch-multiget в Search по ad_id, когда появится) — заметно улучшит UX; при массовом импорте избранного — bulk-endpoint; сохранённые поиски (5.3) добавят в этот сервис outbox (событие совпадения) и, вероятно, Redis для дедупа/anti-лавины.

## ADR-029: Matcher сохранённых поисков — событие-триггер + чтение из Listing, грубый отбор кандидатов в БД, дедуп и троттлинг, событие для Notification

- **Статус:** accepted
- **Дата:** 2026-07-09
- **Автор:** AI-инженер, по согласованию с ТЗ (ARCHITECTURE §4.8; ROADMAP Stage 5.3)

### Контекст

Сохранённые поиски с алертами: `POST /saved-searches`, `GET /me/saved-searches`, `DELETE /saved-searches/{id}`. На каждое новое (одобренное) объявление matcher должен найти сохранённые поиски, чьи фильтры совпадают, и уведомить — с батчингом, дедупом и защитой от лавины (ARCHITECTURE §4.8). Вопросы: (1) откуда брать данные объявления для оценки, (2) как не перебирать всю базу поисков на каждое объявление, (3) где точно оценивать фильтры, (4) как обеспечить «не более одного уведомления на пару» и защиту от лавины, (5) как хранить условия поиска.

### Рассмотренные варианты и решения

**(1) Событие `bozor.ad.approved` — триггер; данные объявления читаются из Listing (выбрано).** Событие несёт только `ad_id`; matcher запрашивает `GET /internal/ads/{id}` (источник истины) и получает категорию/регион/город/цену/валюту/атрибуты/заголовок. Тот же приём, что индексатор Search (ADR-024). Отклонено «обогащать событие полной проекцией»: раздувает контракт `bozor.ad.approved` ради одного потребителя.

**(2) Грубый отбор кандидатов индексом БД по category/region (выбрано).** `saved_searches` дублирует `category_id`/`region_id` из `query_json` отдельными столбцами (NULL = «любой») с частичным индексом `WHERE notify_enabled`. Matcher берёт только кандидатов, где `(category IS NULL OR = adCat) AND (region IS NULL OR = adRegion)` — не всю базу. ARCHITECTURE предлагает «в памяти/кеше, индексированные по грубым ключам»; реализовано БД-индексом (та же селективность, без состояния в процессе). In-memory кеш — возможная оптимизация поверх.

**(3) Точная оценка фильтров в приложении (выбрано).** `SearchQuery.Matches(AdView)` в Go: структурные фильтры (цена-диапазон, город, валюта без регистра, все атрибуты) — точно; текст — приблизительно (подстрока без регистра по заголовку/описанию; полнотекст/typo живут в Search). Категория — точное совпадение (как фильтр Search 4.3), не иерархическое.

**(4) Дедуп таблицей + троттлинг `last_notified_at`, одной транзакцией (выбрано).** На каждое совпадение: `INSERT saved_search_matches (saved_search_id, ad_id) ON CONFLICT DO NOTHING` (пара уведомляется ≤ 1 раза, устойчиво к повторной доставке `bozor.ad.approved`); если пара новая — условный `UPDATE last_notified_at = now() WHERE прошло окно троттлинга`: прошло → `outbox.Enqueue` события; не прошло → подавлено (защита от лавины, дедуп-строка не даёт повторить). Redis-альтернатива отклонена: таблица проще, durable и уже в tx с outbox.

**(5) `query_json` (jsonb) зеркалит фильтры Search 4.3 (выбрано).** Единая ментальная модель «сохранить текущий поиск»; грубые ключи денормализованы в столбцы для индекса.

### Детали решения

- Модель: `saved_searches` (query_json + грубые category_id/region_id + notify_enabled + last_notified_at), `saved_search_matches` (дедуп), `outbox`. events += `bozor.saved_search.matched`.
- Слои: domain (SearchQuery.Matches, ValidateName), listingclient (GetAd → AdView), repo (Create/List/Delete/Candidates/RecordMatchWithEvent), app.SavedSearchService (лимит 50, UUIDv7), matcher (консьюмер `bozor.ad.approved`), transport (`/saved-searches`, `/me/saved-searches`).
- Публикует `bozor.saved_search.matched` {saved_search_id, user_id, ad_id, name} → Notification (6.1) рендерит и рассылает.

### Последствия

- Алерты срабатывают на совпадениях, дёшево по нагрузке (грубый отбор), идемпотентно на пару, с защитой от лавины. Проверено вживую: matching + non-matching поиски; `bozor.ad.approved` → 2 кандидата по категории, точная оценка → 1 совпадение (второй отсеян ценой), `bozor.saved_search.matched` опубликовано с корректным payload; повтор события → дедуп (без дубля). Маршрутизация `/me/saved-searches`→favorites подтверждена.
- **Условия пересмотра:** in-memory кеш поисков при росте объёма; иерархическое совпадение категории (по category_path); полнотекстовая оценка текста (вызов Search) вместо подстроки; батч-digest вместо подавления при лавине; вынести matcher в отдельный воркер-деплой при высокой нагрузке.

## ADR-030: Notification — сервис доставки событийных уведомлений (Telegram), проекция получателей, чтение настроек из User/Profile, идемпотентность и rate limiting

- **Статус:** accepted
- **Дата:** 2026-07-09
- **Автор:** AI-инженер, по согласованию с ТЗ (ARCHITECTURE §4.10; ROADMAP Stage 6.1)

### Контекст

Событийные уведомления пользователям: на доменные события (`bozor.ad.approved|rejected|expired`, `bozor.saved_search.matched`, `bozor.chat.message_sent`, `bozor.promotion.activated`, `bozor.payment.succeeded|failed`, `bozor.wallet.refunded`, `bozor.review.created`, `bozor.user.banned`) отправить сообщение в Telegram с учётом языка и настроек пользователя. Требования (ARCHITECTURE §4.10): локализованные шаблоны uz/ru, учёт `notification_prefs`, батчинг/rate limiting (не спамить пользователя и не упираться в лимиты Bot API), статусы доставки, ретраи с backoff, DLQ. Ключевые вопросы: (1) откуда брать `telegram_user_id` (chat_id) и язык, (2) откуда брать `notification_prefs`, (3) как не отправить дубль при at-least-once доставке шины, (4) как обрабатывать ошибки Bot API и лимиты, (5) нужен ли outbox.

### Рассмотренные варианты и решения

**(1) Проекция получателей из `bozor.user.created` (выбрано).** `telegram_user_id` присутствует только в событии регистрации (Auth, ADR-012); User/Profile его не хранит. Notification держит локальную таблицу `recipients` (user_id → telegram_user_id + language_code), наполняемую консьюмером `notification-recipients` на `bozor.user.created` (upsert идемпотентен — inbox не нужен). Отклонён синхронный gRPC к Auth за chat_id: лишняя связность в hot-path.

**(2) `notification_prefs` — fetch-current-state из User/Profile (выбрано).** Настройками владеет User/Profile; Notification читает их через внутренний `GET /internal/users/{id}/notification-prefs` (без авторизации, только внутренняя сеть — как `/internal/ads` Listing) с TTL-кешем (по умолчанию 30 с), чтобы не бить в Profile на всплесках. Тот же приём «чтение истины из сервиса-владельца», что у индексатора Search (ADR-024) и matcher (ADR-029). Отклонено дублирование prefs событиями из User/Profile: он их не эмитит, а добавление событий + проекция + семантика «отсутствие=включено» дороже эндпоинта. Маппинг subject→группа настроек (`ad_status`/`chat_message`/`saved_search`/`review`/`promotion`); `bozor.user.banned` минует настройки (бан уведомляется всегда).

**(3) Идемпотентность по `event_id` таблицей `notifications` (выбрано).** Каждое событие адресовано одному получателю (fan-out сохранённых поисков уже разложен на события matcher'ом, ADR-029), поэтому `notifications.event_id UNIQUE` служит и журналом статусов, и ключом идемпотентности: `BeginDelivery` делает INSERT pending / ON CONFLICT инкремент attempts с `RETURNING status` — доставка продолжается только для `pending` (терминальные `sent`/`failed`/`skipped` не переотправляются). Отдельная таблица inbox не нужна.

**(4) Классификация ошибок Bot API + rate limiting; ретраи/DLQ — механизмом natsx (выбрано).** Клиент Telegram различает постоянные ошибки (400/401/403/404 — заблокирован ботом, неверный chat_id → `failed`, Ack) и повторяемые (429 c `retry_after`, 5xx, сеть → Nak). Повтор с backoff и DLQ обеспечивает natsx (Nak → расписание {1s,5s,15s} → DLQ после `MaxDeliver`), не изобретая свой планировщик. Глобальный token-bucket (`golang.org/x/time/rate`) ограничивает частоту вызовов Bot API. Пустой токен → канал отключён (`skipped/channel_disabled`) — удобно для dev без бота.

**(5) Без outbox (выбрано).** Notification — терминальный потребитель, доменных событий не публикует.

### Детали решения

- БД `bozor_notification`: `recipients` (проекция), `notifications` (журнал+статусы, `event_id UNIQUE`). Порт dev 8089.
- Слои: domain (статусы/группы/`EventPayload`/шаблоны uz/ru), telegram.Client (`Send`, классификация), prefs.Client (HTTP+TTL-кеш), repo, notify.Handler (консьюмер доставки на 11 subjects), notify.Recipients (проекция), transport (`GET /api/v1/notifications` — история со статусами).
- User/Profile: добавлен `GET /internal/users/{id}/notification-prefs`.
- Контракт payload для уведомлений: получатель — в поле `user_id`; события lifecycle Listing уже несут `user_id`+`title`; `bozor.ad.approved|rejected` (Moderation, 6.4) и `bozor.chat.message_sent` (Chat, 7) должны нести получателя в `user_id`.

### Последствия

- Уведомления доставляются идемпотентно, локализованно, с учётом настроек, со статусами и ретраями; лимиты Bot API под контролем. Проверено вживую (mock Bot API): проекция получателя из `bozor.user.created`; `saved_search.matched`/`ad.approved`/`user.banned` → `sent` (ru), uz-получатель → узбекский текст; повтор `event_id` → одна отправка; выключенная группа → `skipped/prefs_disabled`, включённая → `sent`; без токена → `skipped/channel_disabled`.
- **Условия пересмотра:** батчинг-дайджест на получателя при росте объёма (сейчас анти-лавина — в источнике: matcher 5.3 троттлит; глобальный rate-limit защищает Bot API); честить `retry_after` точным Nak-delay вместо фиксированного backoff; абстракция каналов под push/email/SMS; отметка «протухших» pending как failed фоновым процессом; вынести доставку в отдельный воркер-деплой при высокой нагрузке.

## ADR-031: Moderation — авто-модерация по событию отправки на проверку, чтение объявления из Listing, стоп-слова/категории/дубли, авто-одобрение или ручная очередь

- **Статус:** accepted
- **Дата:** 2026-07-09
- **Автор:** AI-инженер, по согласованию с ТЗ (ARCHITECTURE §4.11; ROADMAP Stage 6.2)

### Контекст

Каждое новое/отредактированное объявление должно пройти модерацию до активации (ARCHITECTURE §4.11): авто-проверки (стоп-слова uz/ru, запрещённые категории, детекция дублей); прошло — авто-одобрение (`active`), не прошло — ручная очередь модератора. Вопросы: (1) на какое событие запускать модерацию, (2) откуда брать текст объявления, (3) как реализовать авто-проверки, (4) как одобрять и вести очередь, (5) идемпотентность.

### Рассмотренные варианты и решения

**(1) Триггер — `bozor.ad.updated|created` со статусом `pending` (выбрано).** Объявление создаётся как `draft`; модерация осмысленна лишь когда автор отправил его на проверку (переход в `pending`, Stage 3.4). Консьюмер фильтрует событие по `status == pending` (дёшево, без обращения к Listing для драфтов/активных) и лишь тогда обрабатывает. Отклонено «модерировать на `bozor.ad.created`»: черновик ещё редактируется.

**(2) Текст объявления читается из Listing (выбрано).** Событие не несёт `description` (только `title`), а стоп-слова проверяются по title+description. Модератор запрашивает `GET /internal/ads/{id}` (источник истины) и перепроверяет `status == pending` по факту (защита от гонок) — тот же приём «чтение истины из владельца», что у индексатора Search (ADR-024) и matcher (ADR-029).

**(3) Авто-проверки (выбрано).** Стоп-слова — подстрока без регистра по нормализованному тексту (`Normalize`: нижний регистр, ё→е, схлопнутые пробелы); словарь сидируется базовым набором («копия/реплика/подделка/под оригинал» + узбекские) и расширяется таблицей `stopwords` (управляется админом). Запрещённые категории — точное членство в `forbidden_categories` (пусто по умолчанию). Дубли — совпадение `content_hash` (sha256 нормализованного title+description) среди объявлений того же пользователя (исключая своё `ad_id`). Проверки изображений (NSFW) — на будущее.

**(4) Авто-одобрение через outbox; очередь таблицей (выбрано).** Прошло все проверки → `moderation_task` со статусом `approved` + `bozor.ad.approved` (одной tx через outbox) → Listing активирует (worker.Moderator, 3.4), Search индексирует, matcher матчит (5.3), Notification уведомляет (6.1). Не прошло → задача `manual` в ручной очереди (Stage 6.3), объявление остаётся `pending`. Очередь читается только персоналом (`requireStaff` admin/moderator).

**(5) Идемпотентность (выбрано).** Inbox `processed_events` по `event_id`; одна актуальная задача на объявление (`ad_id UNIQUE`, upsert — повторная модерация правки перезаписывает задачу).

### Детали решения

- БД `bozor_moderation`: `moderation_tasks` (ad_id UNIQUE, content_hash, status, auto_result, reasons), `stopwords` (сид+расширяемый), `forbidden_categories`, `processed_events`, `outbox`. Порт dev 8091.
- Слои: domain (Normalize/ContentHash/MatchedStopwords/Evaluate), listingclient (GetAd), repo (проверки + SaveApprovedTask/SaveManualTask одной tx), worker.Moderator (консьюмер `moderation-auto`), transport (`GET /api/v1/moderation/tasks` — staff).
- Публикует `bozor.ad.approved` {ad_id, user_id, title}. Ручные действия (approve/reject/request-edit) и `bozor.ad.rejected` — Stage 6.3.

### Последствия

- Новое объявление проходит авто-модерацию до активации; чистое активируется без человека, подозрительное уходит в ручную очередь. Проверено вживую: чистое `pending` → `approved` → `bozor.ad.approved` → Listing `active`; со стоп-словами → `manual` (reasons=[копия, под оригинал, реплика]), остаётся `pending`; дубль того же пользователя → `manual` [duplicate]; очередь: moderator→200, user→403, аноним→401.
- **Условия пересмотра:** проверки изображений (NSFW); иерархическая проверка категорий; нечёткая детекция дублей (шинглы/эмбеддинги) вместо точного хэша; кеш стоп-слов вместо чтения на каждое событие; вынести авто-модерацию в отдельный воркер-деплой; учёт атрибутов объявления в стоп-словах.

## ADR-032: Ручные действия модератора — approve/reject/request-edit над ручной очередью, условный UPDATE с публикацией события, повторная модерация переиспользует авто-консьюмер

- **Статус:** accepted
- **Дата:** 2026-07-09
- **Автор:** AI-инженер, по согласованию с ТЗ (ARCHITECTURE §4.11; ROADMAP Stage 6.3)

### Контекст

Задачи, не прошедшие авто-модерацию (ADR-031), попадают в ручную очередь (`status = manual`). Модератор должен их разобрать: одобрить, отклонить (с обязательной причиной) или вернуть на доработку с правом правки и последующей повторной модерацией. API — под ролью moderator/admin. Вопросы: (1) как безопасно применить решение, (2) как связать с жизненным циклом Listing, (3) чем reject отличается от request-edit, (4) как обеспечить повторную модерацию после правки.

### Рассмотренные варианты и решения

**(1) Условный UPDATE с guard по статусу + событие одной транзакцией (выбрано).** Решение применяется `UPDATE moderation_tasks SET status/decided_by/comment/decided_at WHERE ad_id AND status='manual'`; если 0 строк (задача уже решена конкурентно) → `applied=false` → 409, без публикации. Иначе — `outbox.Enqueue` результата в той же tx. App сначала читает задачу (404 если нет, 409 если не `manual`), затем применяет (защита от гонки — повторная проверка в UPDATE).

**(2) Связь с Listing через события (выбрано).** approve → `bozor.ad.approved` → Listing активирует (worker.Moderator, 3.4); reject/request-edit → `bozor.ad.rejected` → Listing переводит в `rejected` (редактируемый статус). Notification (6.1) рендерит причину из payload.

**(3) reject vs request-edit — статус задачи + флаг в payload (выбрано).** В статусной машине Listing нет отдельного «needs_edit»; `rejected` уже редактируемо и допускает повторную отправку. Поэтому оба публикуют `bozor.ad.rejected`, но: reject → задача `rejected`, `request_edit=false`; request-edit → задача `edit_requested`, `request_edit=true` (Notification может формулировать мягче). Оба требуют причину/пояснение (`ValidateReason`). Отклонено введение отдельного статуса в Listing (лишняя сложность ради формулировки).

**(4) Повторная модерация переиспользует авто-консьюмер 6.2 (выбрано).** Автор правит `rejected`-объявление и отправляет заново (`submit`: rejected→pending, событие `bozor.ad.updated(pending)`); консьюмер авто-модерации (ADR-031) обрабатывает его заново. Отдельного механизма «re-review» не требуется.

### Детали решения

- Миграция 00002: `moderation_tasks` += `decided_by`, `comment`, `decided_at`; статус пополнен `edit_requested`.
- Слои: domain (StatusEditRequested, ValidateReason), repo (GetTask, DecideWithEvent), app.Service (Approve/Reject/RequestEdit), transport.DecisionHandler (POST `/tasks/{adId}/approve|reject|request-edit`, staff-only).
- Ошибки: 404 task_not_found, 409 task_not_pending, 422 reason_required/reason_too_long.

### Последствия

- Модератор разбирает очередь; решения атомарны, идемпотентны к гонкам, аудируемы (кто/когда/почему). Проверено вживую: reject→Listing rejected (comment валидный UTF-8), approve→Listing active; 422 без причины, 409 повторно, 404 несуществующей; полный цикл request-edit→правка автора→resubmit→повторная авто-модерация→auto-approve→active.
- **Условия пересмотра:** отдельная таблица аудита решений (история, а не последнее); оптимистическая блокировка версией; массовые действия над очередью; SLA/эскалация «зависших» manual-задач; толерантность к не-UUID `decided_by` (сейчас uuid — X-User-Id из JWT всегда UUID).

## ADR-033: Жалобы и баны — takedown как отдельное событие `bozor.ad.blocked` (снятие из любого статуса), бан через `bozor.user.banned` с отзывом сессий в Auth

**Дата:** 2026-07-09 · **Статус:** принято · **Глава:** 6.4

### Контекст

Пользователи подают жалобы на объявления/пользователей/сообщения; модератор разбирает их (dismiss/warn/takedown) и может забанить нарушителя. Возникли два вопроса интеграции: (1) как takedown должен реально снимать объявление, если оно уже опубликовано (active), а не только отклонять подачу; (2) как бан должен немедленно прекращать доступ забаненного и уведомлять его.

Изначально takedown публиковал `bozor.ad.rejected`. Но консьюмер модерации в Listing (ADR-031) переводит объявление только `pending→rejected` (это возврат подачи на доработку, автор правит и отправляет заново). Снятие уже активного объявления по жалобе через `bozor.ad.rejected` не срабатывало: условный `UPDATE ... WHERE status='pending'` не находил строку, объявление оставалось `active` — жалоба «разобрана», а нарушающий контент висел в выдаче.

### Решение

**(1) Takedown = отдельное событие `bozor.ad.blocked`, снятие из любого статуса (выбрано).** Модерация при takedown публикует `bozor.ad.blocked` (payload: ad_id, user_id, title, reason), обогащая владельца/заголовок чтением из Listing. Listing-консьюмер модерации переводит объявление в `blocked` из **текущего** статуса (машина состояний уже разрешает «любой→blocked»), кроме уже `blocked` (идемпотентность). Это семантически отделяет два разных действия: `rejected` — мягкий возврат подачи автору на правку (pending→rejected, редактируемо); `blocked` — жёсткое снятие опубликованного контента по жалобе (любой→blocked). Рассмотренная альтернатива — разрешить `bozor.ad.rejected` из active — отклонена: смешала бы «верни на доработку» и «сними за нарушение» в одном событии и в один нередактируемый статус.

**(2) Бан через `bozor.user.banned`, отзыв сессий в Auth (выбрано).** Модерация пишет бан (temporary со сроком / permanent / shadow) и публикует `bozor.user.banned`. Auth-консьюмер `auth-ban` в одной транзакции переводит пользователя в `status='banned'` и отзывает все активные refresh-токены (`revoked_at=now()`), что немедленно прекращает продление доступа. Идемпотентно: повторная доставка не находит активных токенов. Notification уведомляет забаненного всегда (группа GroupNone, минуя настройки).

### Детали решения

- `pkg/shared/events`: новый subject `SubjectAdBlocked = "bozor.ad.blocked"`.
- Listing `worker/moderation.go`: `plan(eventType, current)` возвращает `(upd, ok, err)`; approved/rejected — только из `pending`; blocked — `current→blocked` (skip, если уже blocked). Консьюмер `listing-moderation` подписан на approved/rejected/blocked.
- Moderation `app/reports.go`: takedown публикует `bozor.ad.blocked` (payload blockedPayload); прочие действия — без события.
- Notification: `bozor.ad.blocked` → группа `ad_status`, шаблон uz/ru «объявление снято с публикации» (+причина).
- Auth `worker/ban.go` (консьюмер `auth-ban`) + `repo.BanUser` (tx: status=banned + revoke refresh-токенов, возвращает число отозванных).
- Миграция moderation 00003: `reports` (target_type ad|user|message, status open|resolved|dismissed), `bans` (type temporary|permanent|shadow, expires_at). API: POST `/api/v1/reports` (аутентифицированный пользователь), staff — GET `/moderation/reports`, POST `/moderation/reports/{id}/resolve`, POST `/moderation/bans`.

### Последствия

- Жалоба на активное объявление действительно снимает его. Проверено вживую: пользователь пожаловался на active-объявление → takedown → Listing `active→blocked`, `bozor.ad.blocked` в outbox доставлено, Listing-inbox отметил, владелец получил уведомление (в dev skipped/channel_disabled — нет токена бота, как у всех уведомлений). Бан: staff забанил пользователя → Auth `status=banned` и оба refresh-токена отозваны, ban-запись temporary со сроком, уведомление создано. Repo-интеграции moderation/auth зелёные, lint 0.
- **Условия пересмотра:** enforcement shadow-бана (скрытие контента без ведома пользователя) — сейчас тип есть, поведение на будущее; авто-разбан по истечении `expires_at` (сейчас срок хранится, снятие — вручную); дедупликация повторных жалоб на один объект; восстановление снятого объявления модератором (blocked→…); учёт `target_type=message` после появления Chat (глава 7).

## ADR-034: Chat — модель диалогов и REST-история, продавец из Listing, публикация `bozor.chat.message_sent` на каждое сообщение (гейтинг по онлайн-статусу отложен до WebSocket)

**Дата:** 2026-07-10 · **Статус:** принято · **Глава:** 7.1

### Контекст

CHAPTER 7 — чат покупатель↔продавец по объявлению. Stage 7.1 по ROADMAP — «модель и REST-история»: таблицы `conversations`/`messages` и эндпоинты `GET /conversations`, `GET /conversations/{id}/messages`. WebSocket-транспорт и backplane между репликами — 7.2, доставка/статусы прочтения — 7.3, модерация/лимиты — 7.4 (ARCHITECTURE §4.9). Вопросы этапа: (1) как создаётся диалог и кто продавец, если WS ещё нет; (2) как обеспечить единственность диалога на пару участников по объявлению; (3) когда публиковать `bozor.chat.message_sent` (Notification уже потребляет его); (4) как ограничить доступ к чужим диалогам.

### Решение

**(1) Новый сервис `chat` (БД `bozor_chat`, dev-порт 8092) с REST-отправкой в 7.1 (выбрано).** Хотя ROADMAP перечисляет для 7.1 только GET-эндпоинты, добавлены POST `/api/v1/conversations` (старт) и POST `/api/v1/conversations/{id}/messages` (отправка): без них история пуста и этап нельзя проверить/продемонстрировать вживую. Логика отправки живёт в `app.SendMessage` — WebSocket (7.2) станет вторым транспортом поверх той же логики, REST останется для истории и клиентов без WS. Продавец определяется как владелец объявления через внутренний read-эндпоинт Listing (`/internal/ads/{id}`, fetch-current-state, как модерация/индексатор) — инициатор всегда покупатель; писать по своему объявлению нельзя (`self_conversation` → 422).

**(2) Единственность по тройке (ad_id, buyer_id, seller_id) через UPSERT (выбрано).** `EnsureConversation` — `INSERT ... ON CONFLICT (ad_id, buyer_id, seller_id) DO UPDATE SET ad_id = EXCLUDED.ad_id RETURNING *`: no-op-обновление позволяет вернуть уже существующую строку (её исходные id/created_at). Повторный старт диалога не плодит дубликаты и не «сбрасывает» историю.

**(3) `bozor.chat.message_sent` публикуется на каждое сообщение через outbox (выбрано для 7.1).** Пока нет отслеживания присутствия (WS — 7.2), гейтинг «оффлайн → уведомить» невозможен, поэтому в 7.1 событие адресату (второму участнику, `user_id` в payload) публикуется всегда — Notification рендерит «Новое сообщение» и шлёт в Telegram. В 7.2 публикация станет условной по онлайн-статусу получателя. Вставка сообщения, сдвиг `last_message_at` и запись в outbox — одной транзакцией (атомарность события источника истины).

**(4) Доступ только участникам (выбрано).** Все эндпоинты — под аутентификацией (`requireAuth`, аноним → 401); отправка и чтение истории проверяют, что пользователь — участник диалога (`Participant`), иначе 403. Роли не требуются: чат доступен любому пользователю.

### Детали решения

- Таблицы: `conversations` (UNIQUE (ad_id, buyer_id, seller_id); индексы по buyer/seller + last_message_at DESC), `messages` (FK на conversation ON DELETE CASCADE, `read_at` — задел под счётчики непрочитанных 7.3), `outbox`. Inbox не нужен — Chat в 7.1 только публикует.
- Слои: domain (Conversation/Message, `Participant`/`Counterpart`, `ValidateBody` — непустое, ≤4000 рун), listingclient (GetAd → продавец), repo (EnsureConversation, GetConversation, ListConversations, ListMessages [последние N в хронологии], InsertMessageWithEvent), app.Service (StartConversation/SendMessage/ListConversations/ListMessages), transport (REST + маппинг ошибок 401/403/404/422). Gateway: `/api/v1/conversations` → chat.
- Ошибки: ad_not_found 404, conversation_not_found 404, forbidden 403 (не участник), self_conversation 422, empty_body/body_too_long 422.

### Последствия

- Двое пользователей переписываются по объявлению через REST; адресат получает уведомление. Проверено вживую (chat healthy, миграция applied=1): покупатель начал диалог по active-объявлению «Camry 2019» (продавец подтянут из Listing) → 201; сообщение (33 рун/79 байт — валидный UTF-8) сохранено, `bozor.chat.message_sent` доставлено в шину, Notification создал уведомление продавцу (в dev skipped/channel_disabled — нет токена бота); продавец видит диалог в списке и историю; повторный старт вернул тот же id (идемпотентность); посторонний → 403; self-chat → 422; несуществующее объявление → 404. Unit + integration (testcontainers) зелёные, lint 0.
- **Условия пересмотра (следующие этапы):** WebSocket `/ws` + backplane (Redis Pub/Sub или NATS) для доставки между репликами и гейтинг публикации по онлайн-статусу (7.2); статусы прочтения и счётчики непрочитанных через `read_at` (7.3); rate-limit отправки, жалоба на сообщение (`target_type=message`, ADR-033) и блокировка в чате (7.4); пагинация истории «до сообщения X» (сейчас последние N).

## ADR-035: Chat WebSocket — эндпоинт `/ws` мимо gateway с самостоятельной JWT-аутентификацией, backplane между репликами через NATS core pub/sub

**Дата:** 2026-07-10 · **Статус:** принято · **Глава:** 7.2

### Контекст

Stage 7.2 — realtime-доставка чата: WebSocket-эндпоинт `/ws`, аутентификация по JWT, backplane для доставки между репликами (отправитель и получатель могут быть подключены к разным инстансам Chat), sticky-балансировка (ARCHITECTURE §4.9, ROADMAP 7.2). Гейтинг «онлайн через WS / оффлайн → Telegram» и статусы прочтения — следующий этап 7.3. Вопросы: (1) как аутентифицировать WS-соединение, если браузерный WebSocket не умеет ставить заголовок Authorization; (2) как доставлять сообщение получателю на другой реплике; (3) как безопасно писать в сокет из нескольких горутин; (4) нужна ли sticky-балансировка.

### Решение

**(1) `/ws` идёт мимо gateway (nginx→chat), чат сам проверяет JWT (выбрано).** Edge-nginx уже проксирует `location /ws` напрямую в chat с upgrade-заголовками. Токен принимается из query-параметра `?token=<jwt>` (браузерный WebSocket API не позволяет задать заголовки — только query) или из `Authorization: Bearer` (программные клиенты); проверяется тем же `authx.Parse` и общим секретом `JWT_SIGNING_KEY`, что и в gateway. До апгрейда невалидный токен → обычный HTTP 401. Проверка Origin отключена (`InsecureSkipVerify`): CSWSH эксплуатирует куки-аутентификацию, а здесь явный bearer/query-токен подделать нельзя; TLS-край — nginx.

**(2) Backplane через NATS core pub/sub на subject `chat.rt.<userID>` (выбрано).** Chat уже подключён к NATS (JetStream для outbox); для realtime используется тот же `nats.Conn`, но core pub/sub (эфемерно, без персиста — история в БД). На подключение пользователя реплика подписывается на `chat.rt.<userID>` (одна подписка на пользователя, refcount по числу соединений); при отправке сообщение публикуется в `chat.rt.<recipientID>`, и NATS доставляет его всем репликам с этой подпиской — реплика получателя пишет кадр в локальные сокеты. Путь единый: если отправитель и получатель на одной реплике, та же подписка (loopback NATS) доставит сообщение. Рассмотрен Redis Pub/Sub (ARCHITECTURE допускает оба) — выбран NATS, чтобы не добавлять второй инфра-зависимости в сервис.

**(3) Один писатель на соединение через буфер (выбрано).** coder/websocket не допускает конкурентную запись. Каждое соединение имеет буферизированный канал `send`; единственная горутина-writer тянет его и пишет в сокет. Кадры (ack, ошибки, входящие из backplane) лишь кладутся в буфер. Переполнение буфера (медленный клиент) закрывает сокет — клиент переподключится и перечитает историю через REST (7.1).

**(4) Sticky-балансировка не требуется для корректности (выбрано).** Backplane делает доставку независимой от того, к какой реплике подключён получатель, поэтому sticky — лишь оптимизация (меньше переподписок при reconnect). nginx оставлен на DNS-резолвинге сервиса; корректность обеспечивает backplane.

### Детали решения

- Пакет `internal/ws`: Backplane (интерфейс + NATSBackplane на core pub/sub), Hub (реестр соединений + подписки backplane с refcount, доставка во все локальные сокеты пользователя), Conn (буфер send + closeSlow), Handler (auth→Accept→read/write-циклы), frame (кадры message/ack/error). Отправка по WS переиспользует `app.SendMessage` (persist + bozor.chat.message_sent), которая теперь возвращает и id адресата для realtime-доставки.
- Протокол: client→server `{conversation_id, body}`; server→client `{type:"message"|"ack"|"error", …}`.
- Контекст соединения отменяется при закрытии сокета ИЛИ остановке сервиса (baseCtx) — graceful shutdown закрывает соединения.
- config/compose: `JWT_SIGNING_KEY` добавлен в Chat (обязателен). Роут `/ws` в роутере Chat (без auth-middleware — обработчик сам аутентифицирует).

### Последствия

- Двое пользователей переписываются в реальном времени, в т.ч. на разных репликах. Проверено вживую (две реплики chat: :8092 и :8093, общий NATS): покупатель, подключённый к реплике A, отправил сообщение по WS → получил ack; продавец, подключённый к реплике B, получил кадр message через backplane (**ключевая проверка cross-replica**); тот же сценарий на одной реплике — тоже доставка. Сообщение сохранено в БД и bozor.chat.message_sent опубликовано (тот же путь app.SendMessage, что и REST). WS без токена/с мусорным токеном → 401. Unit + in-process integration (loopback-backplane, httptest+websocket.Dial: auth-reject, доставка получателю+ack отправителю, кадр ошибки при не-участнике) зелёные, lint 0.
- **Условия пересмотра (7.3+):** гейтинг публикации bozor.chat.message_sent по онлайн-статусу получателя (сейчас публикуется всегда — 7.1 baseline; presence через Redis/NATS request-reply); отметки о прочтении (read_at) и счётчики непрочитанных; sticky-балансировка WS на nginx для снижения churn; ping/pong keepalive и лимиты соединений на пользователя; rate-limit отправки и жалоба на сообщение (7.4).

## ADR-036: Chat доставка и статусы — онлайн-гейтинг Telegram по присутствию (NATS request-reply), доставка в app-слое, отметки о прочтении и счётчики непрочитанных

**Дата:** 2026-07-10 · **Статус:** принято · **Глава:** 7.3

### Контекст

Stage 7.3 — доставка и статусы чата (ARCHITECTURE §4.9, ROADMAP 7.3): онлайн-получателю сообщение идёт по WS, оффлайн — уведомление в Telegram; отметки о прочтении (`read_at`) и счётчики непрочитанных. После 7.2 сообщение публиковало `bozor.chat.message_sent` всегда (baseline 7.1), из-за чего онлайн-собеседник получал и WS-кадр, и дублирующее Telegram-уведомление. Вопросы: (1) как узнать онлайн-статус получателя, который может быть подключён к любой реплике; (2) где разместить логику доставки, чтобы REST- и WS-отправка вели себя одинаково; (3) как реализовать отметки прочтения и счётчики.

### Решение

**(1) Присутствие через NATS request-reply на `chat.presence.<userID>` (выбрано).** Пока пользователь подключён, его реплика (Hub) отвечает на запросы присутствия. Отправитель делает `nc.Request(chat.presence.<recipient>)`; ответ любой реплики → онлайн. NATS сразу возвращает `ErrNoResponders`, если подписчиков нет, поэтому оффлайн-случай не тратит таймаут. Ошибка/таймаут проверки трактуется как «оффлайн» (лучше лишнее уведомление, чем немое). Рассмотрен Redis-ключ присутствия с TTL — отклонён: NATS request-reply самоочищается при падении реплики (нет утечки ключей) и переиспользует уже подключённую шину.

**(2) Доставка и гейтинг — в app-слое, а не в транспорте (выбрано).** В 7.2 realtime-публикацию делал WS-обработчик, из-за чего REST-отправленное сообщение не доходило до онлайн-получателя по WS. В 7.3 `app.SendMessage` сам: проверяет присутствие → если оффлайн, кладёт `bozor.chat.message_sent` в outbox (Telegram), если онлайн — не кладёт; всегда вызывает realtime-доставку получателю (fire-and-forget, no-op если не подключён). Так REST и WS ведут себя одинаково. `repo.InsertMessage(msg, ev *Envelope)` принимает опциональное событие (nil — онлайн, без Telegram). Гонки (получатель отключился между проверкой и доставкой) допустимы — сообщение всегда в истории, гейтинг best-effort.

**(3) Отметки прочтения и счётчики (выбрано).** `MarkRead(conv, reader)` помечает `read_at` у сообщений собеседника (не самого reader'а) и уведомляет автора прочитанных сообщений realtime-кадром `read`. Счётчик непрочитанных — коррелированный подзапрос в списке диалогов: число сообщений собеседника без `read_at`. Отметка доступна и по REST (`POST /conversations/{id}/read`), и по WS (кадр `{type:"read", conversation_id}`).

### Детали решения

- app: `Service` получил зависимости `Presence` (Online) и `Realtime` (DeliverMessage/NotifyRead). `SendMessage` возвращает `(Message, recipientID, error)`; добавлен `MarkRead(conv, reader) → n`.
- ws: `NATSBackplane.RespondPresence` (ответчик chat.presence.<userID>, управляется Hub с refcount вместе с realtime-подпиской); `Presence` (request-reply, `ErrNoResponders`/таймаут → offline); `Delivery` (адаптер Hub → app.Realtime); кадр `read`; `onFrame` диспетчеризует по `type` (read/message). Realtime-публикация убрана из обработчика — теперь в app.
- repo: `InsertMessage(msg, ev *Envelope)` (ev=nil → без outbox); `MarkRead(conv, reader, at) → rows`; `ListConversations` считает `unread` подзапросом.
- transport: `POST /api/v1/conversations/{id}/read` (200 `{marked}`); `conversationDTO.unread_count`.

### Последствия

- Онлайн-получатель видит сообщение мгновенно по WS без дубля в Telegram; оффлайн — получает Telegram. Прочтение и непрочитанные работают. Проверено вживую: buyer шлёт при ОФФЛАЙН-продавце → `bozor.chat.message_sent` в outbox (3→4); при ОНЛАЙН-продавце (WS) → тот же счётчик остался 4 (гейтинг), а продавец получил кадр `message` по WS; `GET /conversations` продавца показал `unread_count=4` → `POST /read` (`marked:4`) → `unread_count=0`, `read_at` проставлен у всех 4; buyer онлайн получил кадр `read` (reader_id продавца, время) после его чтения. Unit + integration (репо: unread-подсчёт, MarkRead только чужих сообщений, InsertMessage с ev=nil не плодит событие; ws: real app+Delivery поверх loopback — доставка+ack, кадр read автору, ошибка не-участника) зелёные, lint 0.
- **Условия пересмотра (7.4+):** rate-limit отправки и жалоба на сообщение (`target_type=message`, ADR-033), блокировка в чате; ping/pong keepalive и лимиты соединений на пользователя; presence-гонки и «печатает…»-индикатор; батч-подтверждение прочтения «до сообщения X».
